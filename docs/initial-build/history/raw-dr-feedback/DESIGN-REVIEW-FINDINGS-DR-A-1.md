# PBE Address Book — Design-Review Findings

Reviewer: DR-A

## Findings

I've now read all eight design documents plus the template. Before the findings register, here is my reconstruction of the system and where the documents resist confident reconstruction.

Restatement: what this system is
Purpose. "Book" is a members-only web address book for the Phi Beta Epsilon (PBE) fraternity at MIT — ~40 undergraduates and ~800 living alumni (the loaded dataset, including deceased brothers back to 1890, is treated as ~2,000 records). It replaces a paper directory last printed in 1990 and inaccurate MIT Alumni Association (MITAA) data with a self-maintained directory. It is explicitly one half of a composite system with Ghost (the Ghost-based PBE News site at pbe400.org): Ghost authenticates every Book user, and Book becomes the single management surface for Ghost's members. Book hard-depends on Ghost; Ghost soft-depends on Book.

Users. Three roles, stored in Book's own users collection independent of Ghost: Brother (sees public fields + peers' shared contact fields; edits own profile; stars), Manager (directory upkeep — edits visible fields, sets deceased, exports, sees restricted flags/dates read-only — but cannot see through an "off" privacy toggle or change consent), Administrator (everything, including through-toggle values, add/delete, role changes, bulk import, backup/restore, Ghost sync). A first-party PBE News "Linter" reads a name-only roster via a service-account token.

Data flow. Firestore is the system of record (profiles, users, majors); images live in versioned GCS objects. A thin Node/Cloud Run backend holds an in-memory cache hydrated on cold start and kept fresh by a Firestore snapshot listener. On load, the React/Vite SPA pulls the entire dataset it's permitted to see (GET /api/profiles) — server-side-projected per role/privacy, brotli-compressed, ETag-conditional — and does all search/filter/sort locally. Writes go PATCH/POST/DELETE with If-Match optimistic concurrency; images upload through the backend but are read directly from the CDN via a signed cookie. Member changes (email, canonical name, two email prefs) push to Ghost in real time; a read-only admin audit reconciles drift but never writes Book.

Where the docs blocked confident reconstruction (these became findings): how an admin actually produces a MITAA export (no UI/endpoint; the consent-aware field rules can't be met by the described CSV path); how "Regenerate all thumbnails" executes (~2,000 per-id calls, no batch endpoint); how backup/restore moves ~100 MB of images through Cloud Run; the backup file format, Ghost-sync report shape, MITAA format, CSV escaping (all still "Session 6 TBD" inside docs marked "delivered"); and whether Add Brother is a page or a mode (the PRD claims "four pages" but routes/manual disagree).

### Headline findings
#### Blockers

MITAA sharing defaults to opt-in (DATABASE-SCHEMA §3.1/§9, D45/D59). allowShareWithMITAA defaults true, so a passive brother's contact data ships to an external party on the next admin export. This recreates the precise opt-out harm the product was built to fix (PRD §1), and D45's ethical defense of open defaults explicitly rests on "the data never leaves the brotherhood" — which is exactly false for this one flag. The blanket "all booleans default true" rule was applied to an external-sharing switch without justifying that case.
Email-change can silently lock a brother out (ENGINEERING-DESIGN §5.1 + §2.1). Sign-in matches on primary email only, and email changes reach Ghost via a best-effort real-time push ("retried a few times, otherwise left for the audit"). On a persistent push failure, the new email fails at Ghost (not a member) and the old email fails at Book (the profile already holds the new one) — total lockout with no self-service recovery, on the very dependency the plan names as its top risk.
Most important should-fixes

No mechanism for "Regenerate all thumbnails" and backup/restore won't fit through Cloud Run at ~100 MB of images (findings 6, 7) — two Admin features with no viable execution path as specified.
Hard delete leaves dangling stars and bigBrotherId references (12) — no inbound-reference cleanup; a Big-Brother chip can link to a 404.
Accessibility targets WCAG 2.1 AA, not current 2.2 AA (8). To its credit it is a concrete, testable standard (axe-core + contrast gate + manual checklist), not a bare assertion — but 2.2's Accessible Authentication / Target Size / Focus-Not-Obscured criteria are squarely relevant to this audience.
No API versioning for the cross-language Linter (9); in-page help is deferred to Phase 6 but its a11y gates apply in Phases 3–5 (10); MITAA export has no UI/endpoint and its consent-aware field rules can't be met by the described CSV path (11).
Cross-cutting structural theme. The Session-5/6 additions (allowCommentReplyEmail, the staff-internal/system visibility classes, Ghost coupling) were not fully back-propagated: allowCommentReplyEmail is in or out of the manager projection depending on which doc you read (4); D45 still says "six/two" where the PRD says "seven/three" (5); the schema still says preferences live in cookies after D30 moved them to localStorage, citing a "PRD §Cookies" section that doesn't exist (3). Separately, several MVP mechanics (backup format, sync-report shape, MITAA format, CSV escaping) remain "Session 6 TBD" inside documents that declare themselves "delivered/authoritative" (16).

## Reviewer's note on coverage

The suite is unusually thorough and internally cross-referenced; most findings are gaps, contradictions, or under-specified mechanics rather than wrong architecture. The two blockers concern **consent** (MITAA default-open external sharing, which recreates the product's founding harm) and **auth integrity** (email-change lockout on a best-effort Ghost push). The recurring structural theme is that **Session-5/6 additions** (`allowCommentReplyEmail`, the staff/system visibility classes, Ghost coupling) were not fully back-propagated into earlier decisions and PRD sections (findings 3–5, 13, 23), and that several **MVP mechanics are deferred inside "delivered" docs** (findings 6, 7, 11, 16).
