# PBE Address Book — Design-Review Findings

Reviewer: DR-A
Review Number: DR-A-3
Review Focus: Privacy
Date of Review: 2026-06-07

## Overview of Findings

This is a **privacy-only** review (security assumed out of scope; no attacker assumed present). It asks a different question than a security review: in *ordinary, correct operation*, where does personal information end up somewhere it should not, who outside the brotherhood receives it, is more collected/kept than needed, and can a person actually see and delete what is held about them (CCPA assumed to apply).

**The design's internal privacy posture is genuinely strong** and deserves credit: server-side field projection (D5/D16/D19) is the right architecture and is treated as the security-critical floor; the auth token is carried in the URL *fragment* specifically to keep it out of logs/history/Referer (D20); the audit stream is disciplined to log field *names, not values* (D61); `localStorage` (not cookies) is chosen for client-only prefs to avoid needless transmission (D30); the bulk payload is marked `private, no-cache` so shared caches never hold a per-role copy (D76); service accounts avoid leakable API keys (D58); and `adminNote` is correctly barred from MITAA files (§10). The Book→Ghost push is well-minimized (email, name, two prefs — nothing else).

**But the privacy story breaks down precisely at the system's edges — the places data leaves the projection.** Three themes dominate:

1. **A whole third-party PII flow is undisclosed and unconsented: Mixpanel.** Every brother's email, full legal name, Constitution ID, and role are sent to a third-party analytics vendor regardless of any privacy toggle, with Do-Not-Track explicitly overridden, and Mixpanel is *not mentioned* in the one user-facing privacy disclosure (USER-MANUAL §8). This bypasses the entire consent model the design is otherwise proud of.
2. **The system has no data-subject machinery and cannot actually delete a person.** There is no privacy notice, no access/deletion/opt-out request path, and a "delete" is a hard delete that nonetheless survives in daily JSON backups, GCS object versioning, audit logs (kept for years), prior MITAA exports, Mixpanel, and is resurrected by Restore.
3. **Defaults broadcast more than they should, including third-party data.** All sharing defaults to ON. That is defensible for intra-brotherhood contact sharing (the documented "nudge"), but the same opt-out default is applied to *external* MITAA sharing and to *third-party* data (emergency contacts, spouse/partner names) — the very pattern the design elsewhere criticizes.

A reader should treat findings **1–3 as blockers** for a CCPA-applicable system and **4–14 as real problems to resolve before launch/migration**, several of which (4, 6, 8) bake in exposure at the one-time data load and are cheapest to fix now.

## Severity legend (reviewer-assigned)

- **blocker** — must be resolved before building from these docs.
- **should-fix** — a real problem worth correcting, not a build-stopper.
- **minor** — small clarity/consistency nit.
- **question** — the reviewer is unsure; needs a judgment call.

## Other Reviewer Notes

**Done well (privacy strengths worth preserving).** Server-side projection as the enforced floor (D5/D16/D19) and its exhaustive role-by-field test mandate (§6.6); auth token in the URL *fragment* to avoid log/history/Referer leakage (D20) — the right instinct that Finding 12 asks to extend to view state; the `audit` stream's names-not-values rule (D61) — Finding 9 asks to extend it to *all* streams; `localStorage`-not-cookies for client-only prefs (D30); `private, no-cache` so shared caches never hold a per-role payload (D76); keyless service-account auth avoiding leakable keys (D58); a well-minimized Book→Ghost push (email/name/two prefs only); `adminNote` explicitly barred from MITAA files via an allowlist (§10); and a thoughtful, honest treatment of consent copy for the intra-brotherhood toggles (D45). The instinct is consistently good *inside* the system boundary; the gaps are at the boundary crossings (Mixpanel, MITAA defaults, logs, backups, deletion).

**Cross-cutting theme.** Most findings reduce to one root cause: **the consent/projection model governs reads of the live record but not the copies and exports that leave it** — Mixpanel (1, 14), MITAA (6, 7), diagnostic/access logs (9, 12), the log agent (13), browser disk cache (10), CSV/backups (3, 11), and retained image versions (8). A single "data leaving Book" register — enumerating each egress, its consent basis, retention, and deletion behavior — would close most of these and feed the privacy notice (2) directly.

**Scope/assumptions.** Security was deliberately not assessed (no attacker assumed). CCPA assumed to apply per instruction (see Finding 19). Several documents are explicitly draft (USER-MANUAL screenshots, the Ghost-sync conflict policy, the MITAA format) — findings that depend on not-yet-written mechanics (e.g. exactly what Mixpanel events carry, Finding 14; what push-failures log, Finding 9) are framed as constraints to honor when those parts are built, and are flagged now because the design as written does not yet forbid the leak.
