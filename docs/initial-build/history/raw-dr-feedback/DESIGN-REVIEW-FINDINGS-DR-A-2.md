# PBE Address Book — Design-Review Findings

Reviewer: DR-A
Review Number: DR-A-2
Review Focus: Security
Date of Review: 2026-06-07

## Overview of Findings

This is a security-only review. I read all eight documents before forming conclusions and threat-modeled each trust boundary (STRIDE). Below is the orientation that frames the register, then the findings themselves.

### Authentication model
Book has no credential store. Every user is authenticated through a **Ghost auth bridge** (`ENGINEERING-DESIGN §2.1`): a Ghost Handlebars page at `pbe400.org/book` fetches a short-lived JWT from the same-origin `/members/api/session`, redirects to `book.pbe400.org/auth/callback` carrying the JWT in the URL **fragment**, and the SPA POSTs it to `POST /api/auth/session`. The backend verifies the JWT against Ghost's JWKS (signature/`aud`/`iss`/`exp`), resolves the `sub` **email → a `profiles` record**, and issues Book's own cookie (`HttpOnly`, `Secure`, `SameSite=Strict`, session + 4-hour server cap; no logout). First match auto-creates a `users` doc with `role: brother`; an unmatched email is denied. Identity is pluggable behind an `IdentityProvider` seam, with an env-gated `DevIdentityProvider` for non-prod.

### Authorization model
Three roles (**brother / manager / admin**) live in Book's `users` collection, independent of Ghost. The core control is a **server-side field-projection** (`D5`, `DATABASE-SCHEMA §9`) with six visibility classes (public / toggle / restricted / private / staff-internal / system). The capability matrix is in `PRD §4`. The design correctly states that SPA route guards are "UX only" and enforcement is server-side — so I have treated every SPA-only control as absent and verified whether a server check backs it.

### Entry points / external interfaces
- **Browser → `/api/*`** (cookie auth): `auth/session`, `me`, `profiles` (bulk + per-id CRUD), `profiles/{id}/verify`, `me/stars/{id}`, `users/{id}/role`, `profiles/{id}/headshot` (PUT/DELETE), `thumbnail:regenerate`, and reserved `admin/{import,backup,restore,sync-ghost}`.
- **Browser → `/img/*`** (Cloud CDN, signed cookie) for headshots/thumbnails.
- **Linter → `GET /api/roster`** (GCP service-account OIDC token, off the cookie path).
- **The Ghost bridge page** (third-party-hosted JS that handles the member token).
- **Static SPA** on Firebase Hosting.

### Trust boundaries (less-trusted → more-trusted)
1. **Browser/SPA → backend** — all client input; the primary boundary.
2. **Ghost → Book** — the JWT at sign-in (and the read-only sync audit). Ghost is a third-party SaaS that can mint identity assertions for any email.
3. **The fragment token handoff** — a cross-origin landing that establishes a session.
4. **Linter (machine) → roster endpoint.**
5. **MITAA CSV / admin uploads (CSV, backup, headshot) → Book** — explicitly low-trust data.
6. **Book → external sinks** — Ghost Admin API (push), Mixpanel (browser-side), GCS/CDN.

### Headline
The single most serious issue is **Finding 1**: the wire-compression memoization (`D75`) caches one projected payload and serves it to every caller, which collapses the server-side privacy projection that is the whole security model. Beyond that, the auth-bridge token handling, the lack of an explicit server-side write-allowlist (mass assignment), several UI-only "controls," and the absence of rate-limiting / security-headers / export-auditing are the recurring themes.

| DR-A-2 | 1 | blocker | `ENGINEERING-DESIGN §1.6 (D75)` vs `§1.4 / DATABASE-SCHEMA §9 (D5)` | **Memoized compressed bulk payload defeats the per-caller privacy projection.** D75 memoizes a *single* brotli/gzip buffer of the projected `/api/profiles` payload, invalidated only on write ("amortizes that single compression over every download until the next write"). But the payload is **per-caller**: each caller sees their *own* record in full (incl. values they hid behind off-toggles) and every *other* record projected to the caller's role and that record's privacy flags. A single shared buffer therefore serves the first reader's projection to everyone until the next write — e.g. if an admin/manager is the first GET after any write, every subsequent brother receives admin/manager data (hidden contact values, restricted flags, `adminNote`). Even a charitable "one buffer per role" reading still leaks owner-only fields across brothers and cannot represent the owner/`adminNote` special-casing. Correct partitioning is per-user (~N variants), which negates D75's stated benefit. | Project per request from the hot in-memory dataset and either (a) drop server-side memoization of the *projected* buffer (compress per request at a tuned brotli level), or (b) key any cache by the full projection identity (role **and** caller id) — and never reuse a buffer across identities. Re-evaluate the D75 cost rationale under per-user buffers. Add a test that two different callers cannot receive each other's projection. | | |
| DR-A-2 | 2 | should-fix | `ENGINEERING-DESIGN §1.6 (D76)`; `API-SPEC §1.4` | **Bulk `ETag` ignores role/identity → stale authorization.** The `/api/profiles` ETag is derived solely from the dataset's Firestore `updateTime`, so it is identical for every caller and every role. (a) A role change is a write to `users`, not `profiles`, so it does **not** change `updateTime`; a demoted manager/admin reloading gets `304 Not Modified` and keeps serving the browser-cached higher-privilege payload until an unrelated `profiles` write occurs. (b) Identical ETags across roles mean nothing but `Cache-Control: private` separates projections — a thin margin for the most sensitive response in the app. | Incorporate the projection identity (role + caller id, and a role-version/`users` updateTime) into the ETag so the token changes when authorization changes. Ensure a role downgrade immediately invalidates cached payloads. | | |
| DR-A-2 | 3 | should-fix | `ENGINEERING-DESIGN §2.1, §2.7`; `API-SPEC §2` | **JWT algorithm is not pinned.** The design says "verify signature against Ghost's JWKS (signature, `aud`, `iss`, `exp`)" but never pins the expected algorithm. The classic JWT bypasses — `alg: none`, or `alg: HS256` using the RSA public key as an HMAC secret — are not foreclosed. | Pin verification to Ghost's specific asymmetric algorithm (the `alg`/`kid` advertised in JWKS); explicitly reject `none` and any symmetric algorithm. State this in `§2.1` and unit-test it (the test plan in `§6.6` should include forged-`alg` cases). | | |
| DR-A-2 | 4 | should-fix | `ENGINEERING-DESIGN §2.1 (step 3)` | **Token audience confusion — the assertion is not minted for Book.** Book authenticates with the token from Ghost's `/members/api/session`, whose `aud`/`iss` are "set to the members API" (Ghost's own member/comments surface). Verifying `aud` therefore only confirms it is a *Ghost members* token — not one issued *for Book*. Any party able to read a member's Ghost session token (XSS or a malicious element on `pbe400.org`, the comments widget, a future Ghost integration, a shared analytics script) can POST it to `/api/auth/session` and obtain a full Book session as that member. This is a bearer token with no proof-of-possession or Book-specific audience. | Prefer a token whose audience is Book specifically (e.g. a Ghost integration/JWT scoped to `book.pbe400.org`), or bind the handoff to Book (one-time, nonce/PKCE-style, single-use). At minimum, document that Book's auth trust equals "anyone who can read a member's Ghost session token," and harden `pbe400.org` accordingly. | | |
| DR-A-2 | 5 | should-fix | `ENGINEERING-DESIGN §2.1 (steps 1–4)`; `API-SPEC §2` | **No state/nonce binding on the callback (login CSRF) and redirect-target integrity unstated.** The fragment handoff has no `state`/nonce tying the callback to a flow the user initiated, so an attacker can deliver `book.pbe400.org/auth/callback#token=<attacker token>` and force the victim's browser to establish a session as the *attacker* (forced login — the victim then edits/acts inside the attacker's identity). Separately, if any redirect target in the bridge (the Portal `return`, or the callback URL) is caller-parameterizable, it becomes an open redirect that leaks the `#token` to an attacker origin (full account takeover). | Add a single-use `state`/nonce generated by Book and verified at the callback. Hardcode/allowlist every redirect target so neither the `return` nor the token-bearing destination can be attacker-controlled. | | |
| DR-A-2 | 6 | should-fix | `ENGINEERING-DESIGN §2.1`; `DATABASE-SCHEMA §2, §8` | **Email is the auth join key but has no uniqueness or normalization rule.** Sign-in "resolves the email to a `profiles` record," yet the schema enforces uniqueness only on `id`, and validation (`§8`) has no email-uniqueness or canonicalization (case/Unicode/whitespace) rule. Two profiles sharing an email (incomplete dedup, data error, or a malicious/duplicate admin entry) make resolution ambiguous — and if one of them is a manager/admin, the wrong-identity resolution is a privilege issue. Case/normalization drift between Ghost and Book can also cause mismatched logins. | Enforce a uniqueness constraint on the login email server-side, define canonical normalization (lowercase, trim, Unicode-normalize) applied identically at write and at resolution, and make resolution fail closed (deny) on any ambiguity rather than picking a record. | | |
| DR-A-2 | 7 | should-fix | `API-SPEC §3 (PATCH/POST /api/profiles)`; `DATABASE-SCHEMA §8` | **Field-write authorization mechanism is unspecified (mass-assignment risk).** PATCH is "field-scoped … fields the caller may not edit are rejected," but the doc never specifies this as a positive server-side **allowlist**. A denylist or "ignore unknown fields" implementation would let a manager set another brother's `privacy.*`/consent flags, or any caller set system/verification fields (`lastVerifiedDate`, `verifiedBy`, `lastModified`, `headshotVersion`, `id`) — forging verification (the dedicated `verify` action exists precisely to prevent this) or worse. `ghostMemberId` is the only field documented as explicitly refused on edit. | Specify and implement a per-role **writable-field allowlist**; reject (422/403), never silently ignore, any field outside it — including all system/verification/Ghost fields on every role and consent/privacy fields for managers. Exhaustively test by role × field. | | |
| DR-A-2 | 8 | should-fix | `PRD §5.7.10 (D51)`; `API-SPEC §5 (PUT /api/users/{id}/role)` | **Last-admin safeguard appears to live only in the UI.** "A safeguard prevents demoting the last remaining administrator" is described as on-profile UI behavior; the role endpoint contract lists only admin-only + 404/422 and says nothing about it. If unenforced server-side, a direct API call can remove the final admin and lock the org out of all admin functions (backup/restore, add/delete, role changes, Ghost sync). | Enforce the last-admin invariant server-side in `PUT /api/users/{id}/role` (reject the demotion of the only admin), independent of the UI, and audit-log every role change with before/after. | | |
| DR-A-2 | 9 | should-fix | `DATABASE-SCHEMA §10`; `PRD §5.6.8`; `ENGINEERING-DESIGN §5.3, §6.6` | **CSV / formula injection in exports.** The Directory export and the MITAA export emit user-controlled free-text fields (`firstName`, `employerName`, `adminNote`, links, etc.) to CSV. A value beginning `=`, `+`, `-`, or `@` is executed as a formula when the file is opened in Excel/Sheets. `§6.6` mentions "escaping," but RFC-4180 quoting does not neutralize formula injection — and the MITAA file flows to a third party. | Neutralize leading formula characters on CSV emit (prefix with `'` or wrap per OWASP guidance) for every text cell, in both the client export and any MITAA mapping; add a test for malicious leading characters. | | |
| DR-A-2 | 10 | should-fix | `ENGINEERING-DESIGN §1.6 (D75), §2.7`; `API-SPEC §2` | **No rate-limiting / abuse controls; specific amplification vectors.** No endpoint defines throttling. Concrete abuse: (a) any brother can loop `PATCH` own profile (invalidates the compressed buffer) → `GET /api/profiles` (forces a full-dataset **brotli-11** recompression) to burn CPU — a low-priv CPU DoS; (b) JWKS "refresh on an unknown `kid`" lets crafted tokens force repeated outbound JWKS fetches; (c) `POST /api/auth/session` does unauthenticated crypto work. Alerting on denial bursts (`§6.1`) detects but does not prevent. | Add rate limits / concurrency caps on auth, writes, and the bulk read; debounce/limit JWKS refresh and cap unknown-`kid` triggered fetches; consider a cheaper compression level or decoupling recompression from the request path. | | |
| DR-A-2 | 11 | should-fix | `PRD §5.7.5`; `API-SPEC §6 (PUT /api/profiles/{id}/headshot)` | **Image pipeline attack surface under-bounded.** Uploads are "validated for type and a sane maximum size," then server-side transcoded to WEBP and downscaled. A byte-size cap does not bound *decoded* dimensions: a small, highly-compressed "pixel-flood"/decompression-bomb image can exhaust memory/CPU during decode, and the transcode library itself (libvips/sharp/ImageMagick-class) is a recurring RCE/DoS surface on attacker-controlled input. | Validate and cap decoded width×height (and total pixels) before/at decode; set decoder memory/time limits; pin and patch the imaging library; run transcoding with least privilege. Verify magic bytes, not just declared `Content-Type`. | | |
| DR-A-2 | 12 | should-fix | `ENGINEERING-DESIGN §6.1`; `PRD §5.6.8`; `ENGINEERING-DESIGN §5.3` | **Bulk PII export is unauditable.** Directory CSV export is "pure client-side" and the MITAA export reuses it; neither hits a backend endpoint, and the audit stream's enumerated events (`§6.1`) do **not** include export. So the highest-volume exfiltration path — a manager/admin downloading the whole visible directory, or an admin exporting consenting brothers' full contact set to MITAA — leaves no security audit trail (Mixpanel's client-side "export" event is analytics, not audit, and is itself client-controlled). | Route bulk/MITAA export through (or at least notify) a thin backend endpoint that writes an audit entry (actor, scope/row-count, timestamp), even if generation stays client-side; add "export" to the `§6.1` audit event list. | | |
| DR-A-2 | 13 | should-fix | `ENGINEERING-DESIGN §6.2 (D62)` | **Full-membership PII is shipped to a third party (Mixpanel) with no consent gate.** Book sets **email** as `distinct_id` and **name, Constitution ID, and role** as user properties, from the browser, with `ignore_dnt: true`. This contradicts the product's stated posture that the data "never leaves the brotherhood" (`PRD §5.3/§1`), there is no toggle governing it (unlike MITAA), and shipping `role` exposes the admin roster to Mixpanel — useful reconnaissance if that account is compromised. | Reconcile with the privacy posture: minimize what is sent (avoid name/email; use an opaque, non-reversible `distinct_id`; avoid sending `role`, or restrict it), reconsider `ignore_dnt`, and document the third-party data flow as a deliberate, governed decision. | | |
| DR-A-2 | 14 | should-fix | `ENGINEERING-DESIGN §2.7, §6.4` | **No Content-Security-Policy or standard security headers beyond HSTS.** The hardening list covers cookies, CSRF (SameSite), and HSTS, but there is no CSP, `X-Content-Type-Options`, or framing/`frame-ancestors` policy. Book stores and renders user-supplied content (names, links, `adminNote`, obituary URLs) and loads a third-party analytics script — a CSP is the key defense-in-depth against stored/reflected XSS and script injection. | Define a strict CSP (script/style/connect/img/frame-ancestors), `X-Content-Type-Options: nosniff`, and a framing policy; document them alongside HSTS in `§6.4`. | | |
| DR-A-2 | 15 | should-fix | `ENGINEERING-DESIGN §6.3 (D63)`; `API-SPEC §7 (/api/admin/restore)` | **Restore can set roles outside the audited role-change path, and backups are unguarded crown-jewels.** Backups are "a JSON export of all three collections" (incl. `users` = `role`) and restore is a "faithful whole-database replacement" that writes system/verification fields verbatim and is exempt from import rules. An admin tricked into restoring an attacker-supplied/altered backup thereby grants `role: admin` (and arbitrary `verifiedBy`/`ghostMemberId`) with no per-record role-change audit. The manual backup is also a complete off-platform PII archive whose integrity, encryption, and storage ACLs are unspecified. | Validate/authenticate restore inputs (schema + integrity check, e.g. signed manifest); audit-log the full role delta a restore applies; specify access controls/retention/encryption for both the backup bucket and the downloadable archive; consider excluding or separately gating `role` on restore. | | |
| DR-A-2 | 16 | should-fix | `DATABASE-SCHEMA §8`; `PRD §5.7.4, §5.7.7` | **Stored XSS via user-supplied URLs if scheme is not allowlisted.** `links[].url`, `obituaryUrl`, and `inMemoriamUrl` are validated as "valid `http(s)` URL," but if the check is a loose URL match rather than a strict scheme allowlist, a stored `javascript:`/`data:` URL becomes script-on-click when rendered as an anchor on the profile. | Strictly allowlist the `http`/`https` scheme on write (reject all others); on render, emit anchors with `rel="noopener noreferrer"` and never interpolate URLs into dangerous sinks. | | |
| DR-A-2 | 17 | should-fix | `ENGINEERING-DESIGN §6.6 (D72)` | **`DevIdentityProvider` is a single-gate total auth bypass.** The dev provider "issues a session for a chosen identity and role with no Ghost," guarded by one env check ("the backend refuses to start if the dev provider is combined with a production configuration"). | DR-A-2 | 18 | should-fix | `API-SPEC §8`; `ENGINEERING-DESIGN §5.2 (D58)` | **Roster endpoint authorization must pin the specific caller, not just "a valid Google token."** Auth is a Google-signed OIDC token "audience-scoped to Book," verified at the Cloud Run front door or in-code against Google's JWKS. Though MVP returns a stub, the contract is being built now and will later serve the whole name/year roster. | Require all of: Google issuer, audience = Book, **and** subject = the designated `linter` service account; least-privilege the `run.invoker` grant to exactly that SA. | DR-A-2 | 19 | minor | `ENGINEERING-DESIGN §2.1–2.2`; `PRD §1` | **A Ghost compromise is a total Book compromise, including admin.** Because Book fully trusts Ghost-signed identity assertions and derives all authorization from the resolved email, anyone who can mint/obtain a Ghost token for an admin's email (Ghost Pro breach, signing-key leak, or admin-email control) gets full admin in Book. There is no second factor or step-up for elevated actions (delete-all, restore, role grants). This is an accepted architectural dependency but is worth stating and bounding. | Document the blast radius in the threat model; consider step-up/out-of-band confirmation for the most destructive admin actions (restore, bulk delete, role grant) so they don't ride solely on a single SaaS identity assertion. | | |
| DR-A-2 | 20 | minor | `DATABASE-SCHEMA §7`; `ENGINEERING-DESIGN §2.5 (D23)`; `API-SPEC §6` | **Image access is coarse, non-revocable, and "removed" photos persist.** The CDN signed cookie grants the entire `/img/*` prefix for its 4h TTL and cannot be revoked mid-TTL; combined with versioned, immutable, indefinitely-cacheable object paths, a replaced or "Removed" headshot remains fetchable at its old `…/{version}.webp` URL (delete clears the current path; GCS versioning and CDN caches retain prior bytes). | DR-A-2 | 21 | minor | `ENGINEERING-DESIGN §2.3, §2.5, §2.7` | **Confirm CORS and cookie-domain scoping (cookie-auth depends on both).** Auth is cookie-based and CSRF defense rests on `SameSite=Strict` + same-origin. Two things must be explicit: (a) `/api/*` must not enable permissive/credentialed CORS (no `Access-Control-Allow-Origin` reflection with credentials, especially not for `pbe400.org`), or cross-origin reads of authenticated responses become possible; (b) the session cookie must be **host-only** on `book.pbe400.org` (no `Domain=.pbe400.org`), else Ghost or any sibling subdomain could read it. The CDN cookie is stated host-only; the session cookie's host-only status is not. | State and enforce a deny-by-default CORS policy on `/api/*` and an explicit host-only session cookie (no `Domain` attribute). | | |
| DR-A-2 | 22 | question | `PRD §4.1, §5.7.7`; `ENGINEERING-DESIGN §5.1` | **Managers can mark living brothers deceased — a comms-cutoff capability.** "Set deceased status" is a manager capability. Marking deceased removes a brother from the default directory, applies In Memoriam, and **forces both email preferences off and pushes the unsubscribes to Ghost**. A rogue or compromised manager account can thus silently cut a living brother's newsletter/comment email and visibility (reversible, and audited as "deceased-marking," but impactful). Is granting this to managers (vs. admin-only) the intended trust level? | Confirm intent; consider making mark-deceased admin-only, or require a second factor/approval, given its push side-effects on Ghost. | | |
| DR-A-2 | 23 | question | `API-SPEC §3 (verify), §4 (stars)`; `DATABASE-SCHEMA §6.1` | **Confirm object-level checks on `verify` and `stars` writes.** (a) `POST /api/profiles/{id}/verify` lists auth as "owner, manager, or admin" — confirm a *plain brother* cannot verify **another** brother's record (the capability matrix reserves verifying others to manager/admin); a naive "is authenticated" check would let any brother stamp provenance on anyone. (b) The `users` document holds both self-writable `stars` and admin-only `role`; confirm the stars endpoints write only the `stars` array and cannot be coerced into a `role` write (mass assignment on a shared document). | Make the verify endpoint enforce owner-vs-other by role; ensure stars writes are scoped to the `stars` field only. Add tests for both. | | |

## Other Reviewer Notes

- **What the design gets right (so triage can weight the gaps).** Several decisions are genuinely sound and reduce the attack surface: server-side projection as the single enforcement point (`D5`); making the sync audit *read-only into Book* so a Ghost compromise can't rewrite records (`D55`); a dedicated server-set `verify` action and server-enforced cycle check (`D28`, `§8`); optimistic concurrency on Firestore `updateTime` rather than client clocks (`D25`); the audit discipline of logging field *names, never values* (`D61`); keeping Ghost/CDN signing keys in Secret Manager off the client (`D27`/`§2.7`); and the off-Ghost service-account path for the Linter (`D58`). My findings are about the gaps around these, not the core instincts.
- **Projection test-criticality (relates to Findings 1–2).** The owner-must-not-see-own-`adminNote` rule (`DATABASE-SCHEMA §9`) is an easy projection bug: the natural shortcut "caller == owner ⇒ return the full record" leaks `adminNote`. The `§6.6` plan already calls this out for exhaustive testing — keep it there, and add the cross-caller no-leak test from Finding 1.
- **Email-change / Ghost-push consistency (relates to Finding 6).** Push is real-time but best-effort with retry; if a push fails after an email change, Ghost's `sub` no longer resolves in Book and the brother is **locked out** (or, worse, an old address later reassigned could resolve to the wrong profile). This is primarily availability/consistency, but the lockout window and the reassigned-email edge are worth an explicit policy.
- **Scope note.** This review is diagnosis only; I did not edit or rewrite any design document. Findings cite the documents as written — where a control may exist in implementation but is unstated in the design (e.g. allowlist enforcement, alg pinning, last-admin server check), I have flagged the *specification* gap, since the build is meant to proceed from these docs.
- **Out of security scope.** Functional/UX/consistency issues were not catalogued except where they carry a security consequence.
