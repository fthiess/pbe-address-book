# PBE Address Book — Design-Review Findings

Reviewer: DR-B
Review Number: DR-B-4
Review Focus: Operations & reliability
Date of Review: 2026-06-07

## Overview of Findings

This is a well-documented and thoughtful architectural draft. You have made excellent choices in optimizing for the slow-connection cohort by leveraging edge caching, code-splitting, and compression. Additionally, leaning on serverless architecture (Cloud Run, Firestore) keeps baseline costs extremely low. 

However, as you transition this to a production environment, there are critical distributed systems fallacies embedded in the design—specifically around the assumption of reliable networks and synchronous cross-system operations. The system currently relies on "dual writes" between Book and Ghost without transactional guarantees. If the external Ghost API degrades, times out, or rate-limits Book during an update or bulk import, data will be left in an inconsistent state, and users will be permanently locked out. Furthermore, relying on stateful long-polling (Firestore snapshot listeners) inside stateless serverless containers creates risks of silent cache staleness and read-amplification storms during scaling events. 

The findings below detail exactly how these failures will manifest and offer standard SRE patterns (like the Outbox Pattern and async workers) to resolve them before launch.

## Detailed Finding Register

| DR-B-4 | 1 | blocker | `ENGINEERING-DESIGN.md §5.1` | **Dual-Write / User Lockout Risk:** Book pushes to Ghost in real time on save. If the Firestore write succeeds but the `PUT` to Ghost fails or times out, the systems are out of sync. If this was an email change, Book now expects the new email, but Ghost still sends the magic link to the old email. The returned JWT will contain the old email, which Book will reject with a "contact an administrator" message, permanently locking the user out. | Implement the **Transactional Outbox pattern**. Write the Ghost update intent to a separate Firestore `outbox` collection in the same atomic transaction as the profile edit. Use a Cloud Firestore trigger or Cloud Task to process the outbox asynchronously with exponential backoff until Ghost acknowledges it. | | |
| DR-B-4 | 2 | blocker | `PRD.md §5.8.2`, `ENGINEERING-DESIGN.md §6.5` | **Bulk Import Timeouts & Rate Limits:** The steady-state admin bulk import is treated as a per-row non-owner edit. Processing a CSV synchronously over HTTP will hit Cloud Run's request timeout limits. Furthermore, triggering real-time Ghost API pushes for hundreds of rows synchronously will likely hit Ghost Admin API rate limits, crashing the operation midway and leaving a partially applied, un-rollbackable dataset. | Decouple the bulk import execution. Upload the CSV to GCS, and have a Cloud Task or asynchronous worker process the file in batches. This worker can respect Ghost API rate limits and track row-level success/failure without holding open the admin's browser HTTP connection. | | |
| DR-B-4 | 3 | should-fix | `ENGINEERING-DESIGN.md §1.5` | **Read Amplification & OOM on Restore:** Restore is a faithful whole-database replacement. A bulk replacement of ~2,000 documents will instantly trigger the Firestore snapshot listeners on all active Cloud Run instances. This will cause a massive synchronous download spike across the fleet, potentially exceeding container memory limits (OOM) or causing severe read-amplification latency. | Route traffic away or programmatically disable snapshot listeners during a restore operation. Alternatively, implement debouncing in the cache hydration logic so it does not attempt to process 2,000 individual snapshot events simultaneously. | | |
| DR-B-4 | 4 | should-fix | `ENGINEERING-DESIGN.md §1.5` | **Silent Cache Staleness in Serverless:** Each instance hydrates its in-memory dataset cache and keeps it fresh with a Firestore snapshot listener. In serverless environments, background threads can be throttled, and TCP connections can drop silently without firing a disconnection event. If the listener drops silently, the Cloud Run instance will serve permanently stale data until the container is destroyed. | Implement a health-check/heartbeat mechanism for the snapshot listener. If no updates or heartbeats are received within a threshold, force the listener to re-initialize or gracefully kill the container to trigger a cold start. | | |
| DR-B-4 | 5 | should-fix | `ENGINEERING-DESIGN.md §2.1` | **Ghost JWKS Cache Degradation:** The backend caches Ghost's JWKS and refreshes it on a schedule. If a Cloud Run instance cold-starts while the Ghost JWKS endpoint is temporarily unreachable, the instance cannot verify any JWTs, resulting in a total authentication outage for that container. | Persist the fetched JWKS in a highly available internal store (e.g., Firestore or GCS) as a fallback. If the external Ghost endpoint is down during a cold start, fetch the last known good keys from the internal fallback to allow authentication to continue gracefully. | | |
| DR-B-4 | 6 | should-fix | `ENGINEERING-DESIGN.md §6.5` | **Migration Execution Environment:** Schema migration uses eager batch migration. If an admin or CI runner triggers a batch migration of ~2,000 documents over the public internet, a network drop mid-execution will leave the database in a partially migrated state. Firestore limits batch writes to 500 operations, meaning this must be chunked and lacks atomic rollback. | Execute migrations exclusively via a dedicated GCP Cloud Run Job or Cloud Task that runs entirely within the Google network. Implement cursor tracking so that if a chunked batch fails, it can be safely resumed from the point of failure. | | |
