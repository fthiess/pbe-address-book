# PBE Address Book — Design-Review Findings

Reviewer: DR-A
Review Number: DR-A-4
Review Focus: Operations & reliability
Date of Review: 2026-06-07

## Overview of Findings

This is a well-reasoned design with genuine reliability instincts already in place: Firestore as a write-through system of record so memory is never authoritative (D7), optimistic concurrency on a server-authoritative `updateTime` with a non-destructive reconcile (D25), server-side field projection (D5), an audit stream that logs field *names* not *values* (D61), versioned object storage (D8), and managed TLS (D64). Those are the right bones. My job was to assume it fails in production and find where — and the failure surface is concentrated in three places the design under-treats: **(a) the Book→Ghost dual-write**, which is non-atomic, has no defined retry/outbox, and is the explicitly-deferred "Session-6 operations detail" that the delivered Operations section never actually delivers; **(b) the in-memory-cache + snapshot-listener freshness model**, whose correctness claim ("self-healing, correct at any instance count," D26) is not established against Cloud Run's idle-CPU-throttling and listener-drop behavior, and which has no staleness detection because reads never touch Firestore; and **(c) the bulk-payload caching/compression layer**, whose cache key (dataset version + encoding) does not include the *caller's projection identity*, so the owner-aware projection it sits on top of can leak one caller's data to another or strand a demoted user on an elevated cached view.

The recurring theme is **non-atomic multi-system mutations with no compensation and no observability to catch the resulting drift**. Every write of consequence touches two or more of {Firestore, GCS, Ghost} with no transaction, the failure path is "the manual admin audit will catch it eventually," and the only specified alert is on sign-in denials — not on the push failures that actually drive divergence. The single most severe user-facing consequence is a **self-inflicted lockout**: a brother changes their own email, the Ghost push silently fails, and on their next visit Book (new email) and Ghost (old email) no longer agree, so authentication resolves to no profile and they are denied until an admin manually reconciles.

Highest-priority items: Findings **1, 2, 3** (blockers). The rest are should-fix resilience/observability gaps, a set of minor concurrency/atomicity nits, and one open question about the Ghost-push execution model that several other findings depend on.

| DR-A-4 | 2 | blocker | `ENGINEERING-DESIGN §1.3, §1.6`; `DATABASE-SCHEMA §9`; `DECISIONS D5, D75, D76` | **The bulk-payload cache/ETag is keyed by dataset version + encoding, not by the caller's projection identity** — but the projection is owner-aware. | DR-A-4 | 3 | blocker | `ENGINEERING-DESIGN §1.5`; `DECISIONS D26` | **The snapshot-listener freshness model is not established against the chosen platform, and has no staleness detection.** D26 asserts each instance's cache stays correct "within a fraction of a second … self-healing and correct at any instance count." ## Other Reviewer Notes

### Failure-surface / dependency / state-mutation map (orientation for the findings above)

**External dependencies (and what dies when each fails):**

- **Ghost (Ghost Pro SaaS) — hard dependency, two surfaces.** *Auth/read:* the bridge (`/members/api/session` JWT, JWKS, Portal). Down/changed → **no one can sign in** (Findings 5, 16). *Member-management write:* the Admin API (create/update/delete). Down/slow → dual-write divergence + lockouts (Findings 1, 4, 7). The asymmetry the design notes (Book hard-depends on Ghost) is exactly the exposure: Ghost is a single point of failure for both authentication *and* data consistency.
- **Firestore — system of record + snapshot listener.** Listener stall/drop or cold-start read storm → stale cache served as fresh / latency spikes (Findings 3, 9). No read path ever revalidates, so staleness is invisible.
- **GCS + Cloud CDN — images + backups.** Partial image writes → orphans / broken images (Finding 6); signed-cookie expiry → mid-session image 403s (Finding 19).
- **Mixpanel — analytics (client-side).** Should fail open; main note is the extra third-party bytes/connection on first load, mildly at odds with the slow-connection goal. (No finding raised — verify it is genuinely non-blocking.)
- **Cloud Run platform — regional, scales to zero.** Cold starts (Finding 9); idle CPU throttling undermines the background-listener model (Finding 3); request timeouts kill long admin ops (Finding 8); single-region DR (Finding 22).

**What the design already gets right (so triage doesn't "fix" these):** write-through Firestore so a crashed instance loses nothing (§1.5); `updateTime`-based OCC with a no-silent-clobber reconcile (D25); server-side owner-aware projection (D5) — though see Finding 2 for its interaction with caching; audit logging of field *names* not *values* (D61); GCS object versioning as a recovery floor (D8); the read-only reconciliation audit's never-write-Book invariant (D55), which correctly avoids the failed-push-undo hazard (the gap is that it's the *only* drift-recovery mechanism and it's manual — Finding 11).

My review is complete.
