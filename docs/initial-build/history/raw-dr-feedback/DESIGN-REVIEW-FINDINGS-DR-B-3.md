# PBE Address Book — Design-Review Findings

Reviewer: DR-B
Review Number: DR-B-3
Review Focus: Privacy
Date of Review: 2026-06-07

## Overview of Findings
This review assesses the PBE Address Book ("Book") design for privacy risks, focusing on data minimization, leakage paths, third-party exposure, retention/deletion lifecycles, and CCPA compliance. The architecture is generally secure against external attackers, but its ordinary operation introduces severe privacy risks. The most critical issues involve sending cleartext PII to third-party analytics while explicitly ignoring Do Not Track (DNT) signals, forcing the sharing of personal data with an external organization (MITAA) even when the user explicitly opts out, and a backup-restore mechanism that will illegally resurrect deleted user data. Additionally, the design uses URL state in a way that will leak PII to external websites via the `Referer` header, and it completely omits a mechanism for users to access hidden internal notes about themselves, which is a CCPA violation.

| DR-B-3 | 1 | blocker | `ENGINEERING-DESIGN.md` §6.2, `DECISIONS.md` D62 | **Analytics Data Minimization & DNT Violation:** The design explicitly sends PII (Email, Name, Constitution ID) to Mixpanel and sets `ignore_dnt: true`. Use a salted hash of the Constitution ID or a dedicated opaque UUID for analytics. Remove `ignore_dnt: true` and honor browser privacy signals. |  |  |
| DR-B-3 | 2 | blocker | `PRD.md` §6.3, `DATABASE-SCHEMA.md` §9 | **Forced Third-Party Data Sharing on Opt-Out:** The MITAA export logic dictates that even if `allowShareWithMITAA` is false, "name, class year, and public deceased status/obituary always flow regardless". Under CCPA, if a user exercises their right to opt out of third-party sharing/sales, you cannot force the transfer of their personal identity data to that third party. | Respect the opt-out absolutely. If `allowShareWithMITAA` is false, the brother must be entirely omitted from the MITAA export payload. |  |  |
| DR-B-3 | 3 | blocker | `DATABASE-SCHEMA.md` §9, `PRD.md` §4.2 | **CCPA Right to Access vs. Hidden Admin Notes:** `adminNote` is defined as "staff-internal" and "invisible to the owner". However, under CCPA, a consumer has the right to request access to *all* personal information collected about them. The system currently provides no DSAR (Data Subject Access Request) export feature, meaning these notes would be illegally withheld during a privacy request. | Create a backend-only DSAR export endpoint that compiles a full JSON dump of a brother's profile, explicitly including `adminNote`, `ghostMemberId`, and audit logs tied to their ID. |  |  |
| DR-B-3 | 4 | should-fix | `API-SPEC.md` §3, `ENGINEERING-DESIGN.md` §6.3 | **Backup Restoration Resurrects Deleted Users:** `DELETE /api/profiles/{id}` deletes the live profile, but `ENGINEERING-DESIGN` §6.3 states a restore is a "faithful whole-database replacement" and makes no mention of CCPA deletion obligations. If the database is restored from a backup, brothers who exercised their Right to Delete will be silently resurrected. | Implement a "tombstone" table of deleted Constitution IDs. The restore script must cross-reference this list and actively scrub deleted profiles from the restored JSON before writing to Firestore. |  |  |
| DR-B-3 | 5 | should-fix | `PRD.md` §5.4, `DECISIONS.md` D31 | **URL PII Leakage via Referer Header:** Search queries live in the URL (`?q=James+Smyth`). The application contains external links (`obituaryUrl`, `links` array). When a user clicks an external link from a filtered directory page, the browser's `Referer` header will leak the user's search queries (which contain other brothers' PII) to the external website. | Configure Firebase Hosting / the SPA shell to include a strict `Referrer-Policy: strict-origin-when-cross-origin` or `no-referrer` header to strip query parameters when navigating off-site. |  |  |
| DR-B-3 | 6 | should-fix | `DATABASE-SCHEMA.md` §7, `API-SPEC.md` §6 | **Image Versioning Retains Deleted Personal Data:** GCS object versioning is enabled. When a brother updates their headshot or deletes it (e.g., to remove an old photo for privacy reasons), the old version remains accessible in GCS. Because CDN auth is coarse, anyone who bookmarked or guesses the old version string can still view the deleted photo indefinitely. | Define a lifecycle policy for GCS to permanently purge non-current image object versions after a short window (e.g., 7 days), ensuring deleted personal photos are actually destroyed. |  |  |
| DR-B-3 | 7 | should-fix | `API-SPEC.md` §2, `PRD.md` §5.2 | **Missing Notice at Collection:** The system automatically provisions a user profile upon their first successful Ghost auth bridge handshake. There is no intercept or screen providing a Privacy Policy or CCPA Notice at Collection informing the user about data processing, analytics tracking, or external sharing defaults. | Introduce a first-login interstitial page requiring the user to acknowledge the Privacy Policy and data collection practices before their session is fully established. |  |  |
