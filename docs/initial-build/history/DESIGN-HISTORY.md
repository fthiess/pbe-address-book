# PBE Address Book — Design History

This is the narrative of *how* Book's initial-build design was reached: the sequence of planning sessions, what each settled, and the turning points where an earlier assumption was overturned. It is distilled from the planning session log and is a **history**, not a specification — it is deliberately set apart under `history/` and is **not** referenced by the engineering docs. For *what* was decided and *why*, the authority is [`../DECISIONS.md`](../DECISIONS.md) (the decision log, cited below by its `D##` numbers); for the current design, the delivered documents in [`../`](../) govern.

## How the planning was run

The design was developed across roughly a dozen focused sessions over 2026-05-31 … 06-07, sequenced in dependency order: **scope** gates everything, the **schema** is what the app manages, **auth/permissions/API** govern access to the schema, the **UI** consumes all three, **integrations** extend outward, and **operations and the build plan** require everything else to be settled first. Each session began from a blank context, so the written artifacts *were* the memory between sessions: a session read the index, the decision log, and the open-questions tracker, did its work by growing the relevant delivered document section by section, then recorded its decisions and updated the trackers. Sessions split when a topic ran long (Session 4 became 4a/4b/4c; Session 6 became 6a/6b/6c) rather than being padded to a fixed cadence.

Two homes were used during planning (decision **D1**): an Obsidian vault held the *process* artifacts — this history's source (the index/roadmap and session log), the decision log, the open-questions tracker, the deferred-optimizations list, and the original seed notes — while the *delivered* documents were authored straight into the code repo. At planning close-out (Session 6c) the still-useful process docs **graduated** into the repo: the decision log and the pre-launch-tools inventory became delivered docs under `docs/initial-build/`, the seed notes were preserved as [`SEED.md`](SEED.md), and the session log was distilled into this file.

## The sessions

**Session 1 — Scope & architecture foundations.** Set the spine. Book is a from-scratch application rather than a server-centric framework (**D2**), because its defining trait is client-side ownership of the data: the browser bulk-downloads the whole permitted dataset and does all search/filter/sort locally over an in-memory copy, with a thin serverless backend (**D4**, **D6**). The privacy-leak that a client-only filter would create was closed by making the backend project every response down to what the caller may see (**D5**) — a decision that shaped everything after it. The stack was fixed as end-to-end TypeScript with a single shared `Profile` type (**D3**), React + Vite on the front, Node on Cloud Run behind Firebase Hosting/CDN (**D10**), Firestore as the system of record with an in-memory cache (**D7**), and headshots/thumbnails as versioned, CDN-served GCS objects loaded lazily (**D8**, **D9**). The MVP/deferred line was drawn (**D11**).

**Session 2 — Data model.** Grew `DATABASE-SCHEMA.md`. Three Firestore collections split private per-user state (`role`, `stars`) out of the directory data so it never enters the bulk-download (**D12**); class year is `number | null` to force explicit handling of "unknown" (**D13**); majors are a controlled vocabulary of MIT course codes (**D14**); the Canonical Name (`First Last 'YY`) is constructed, never stored, with ambiguity derived at load and disambiguated by Constitution ID (**D15**). The field-visibility taxonomy — public / toggle / restricted / private — was defined and made a server-side projection (**D16**), and image assets were specified as immutable versioned WEBP objects (**D17**), with `country` an ISO 3166-1 code from a bundled list (**D18**).

**Session 3 — Auth, permissions & API.** Authentication bridges through Ghost: a Handlebars page relays a Ghost-signed member JWT that Book verifies against Ghost's JWKS, kept swappable behind an `IdentityProvider` seam (**D20**, **D21**), with Book minting its own session cookie under a 4-hour revocation cap (**D22**). The role/capability matrix drew the manager/admin line on *actions* and narrowed the read model so a manager no longer sees through an *off* privacy toggle (**D19**, amending D16). Media is gated by a Cloud CDN signed cookie rather than signed URLs (**D23**); the REST surface was specified under `/api/*` (**D24**) with optimistic concurrency on Firestore's `updateTime` and a reject-and-reconcile UX (**D25**), cache coherence via a per-instance snapshot listener (**D26**), and the external API confirmed deferred while the Linter was promised a first-party path (**D27**). A follow-on coupled verification to content edits (**D28**).

**Session 4 — UI & UX (4a/4b/4c).** *4a* settled the cross-page layer: shadcn/ui (Radix + Tailwind) with TanStack Table/Virtual for the grid (**D29**) — chosen partly because column reordering and virtualization are free there but paywalled in MUI; client preferences in `localStorage` not cookies (**D30**); a three-bucket URL/state model (view→URL, transient context→History state, column lens→`localStorage`) with real anchors and scroll restoration (**D31**); and WCAG 2.1 AA adopted as policy with a width-based responsive baseline (**D32**). *4b* specified the Directory page end to end (**D33–D42**): the pinned/frozen column model, single-column sort with a universal canonical secondary key, the name-only "Name Search" combining typo tolerance and phonetic matching over tokenized name fields, deceased-hidden-by-default with an In Memoriam marker, the "filterable ⟺ visible" typed filter panel, universal stars, full-page profile navigation, the manager/admin action bar with client-side CSV export, and a single virtualized list with bounded thumbnail prefetch; State/Province became a US/CA controlled vocabulary (**D37**, amending D18). *4c* specified the Profile and Admin pages (**D43–D52**) — the reading-order-preserving two-up layout, one layout rendered as four role projections, privacy/consent switches that state each side's consequence and default open (curing the MITAA ambiguous-opt-out failure), the special field controls, crop-on-upload staged until Save, the verification surfacing and In Memoriam treatment, the validation/PATCH-first save with 412 reconcile, and the admin on-profile controls — plus a cross-cutting in-context-help standard whose single source is shared with the user manual (**D53**).

**Session 5 — Integrations.** The organizing realization: Book and Ghost are one composite system, Book hard-depending on Ghost while Ghost soft-depends on Book (**D54**). Ghost sync resolved to a single-master model — Book authoritative, Ghost editing disabled and redirected, real-time Book→Ghost push, and a read-only reconciliation audit that never writes Book (killing the failed-push-undo hazard) (**D55**), which added three Ghost-coupled fields and two new visibility classes (**D56**) and made the initial migration external one-time tooling that seeds email preferences *from* Ghost to honor existing opt-outs (**D57**). The Linter got a read-only roster endpoint authenticated by a service-account token off the Ghost path (**D58**); MITAA stayed manual and low-trust with `allowShareWithMITAA` redefined as a master switch (**D59**); Mailman stayed deferred with replacement directions logged (**D60**). The pre-launch-tools inventory was started this session.

**Session 6 — Operations & build (6a/6b/6c).** *6a* filled the Operations section and the last deferred decisions (**D61–D72**): three-stream structured logging with a names-not-values audit discipline, one shared Mixpanel project across Book and Ghost, two-tier backups with faithful-replacement restore, Google-managed TLS (retiring an earlier Let's Encrypt assumption), the Vitest + Playwright + committed-fake-data testing strategy, three-layer accessibility verification, the bulk-import/verification coupling, majors-vocabulary maintenance by a checked-in file, a correction that `ghostMemberId` is the Admin-API id with the uuid captured alongside (**D70**), a tiered schema-migration strategy (**D71**), and a staging-Book + `DevIdentityProvider` approach to system/UAT testing that dissolved the apparent need for a second paid Ghost (**D72**). *6b* wrote `CODING-PROJECT-PLAN.md` (guiding principles, environments, CI/CD, nine dependency-ordered build phases with exit gates, and the migration/cutover sequence) and drafted `USER-MANUAL.md`, and restructured the repo's `docs/` tree to be subdivided by build with origin/history docs set apart. A short **interstitial** session then adopted four wire-efficiency measures as MVP — bundle caching, a small code-split bundle, dataset compression, and conditional GET (**D73–D76**) — under the organizing constraint that a meaningful fraction of the brotherhood are capable-but-slow-connection users, a sibling to the accessibility-as-policy posture.

**Session 6c — Close-out.** Ran the cross-document consistency pass over the whole delivered suite in a fresh context, scrubbed all real personal information (replacing it with the fake exemplar "James Smyth '84 (#5247)"), stripped the decision log's seed-document citations so it stands alone, graduated the decision log and pre-launch-tools inventory into the repo, preserved the seed notes as `SEED.md`, distilled this design history, retired the vault scaffolding, and proposed a clean-context whole-suite design review (see [`DESIGN-REVIEW-PLAN.md`](DESIGN-REVIEW-PLAN.md)).

## Turning points worth remembering

A few decisions reversed an earlier, plausible-looking assumption — the kind of thing a future reader might otherwise re-propose:

- **Server-side projection over a client-side filter (D5).** The seed imagined the frontend hiding fields it shouldn't show; that is security theater, since the bytes still arrive. Privacy is enforced on the server, and the bulk download a caller receives is already projected to their role.
- **`localStorage`, not cookies, for client preferences (D30).** The seed assumed cookies; cookies needlessly travel to the server on every request for preferences the backend never reads.
- **A single-master Ghost sync, not a two-way merge (D55).** The seed weighed several conflict-resolution policies; making Book the sole master and the Ghost audit read-only dissolves the conflict at its root and removes the failed-push-undo hazard entirely.
- **Managers do not see through an *off* privacy toggle (D19, narrowing D16).** The first taxonomy let managers see restricted values like admins; the housekeeper-and-homeowner model corrected this — a manager sees *that* a field is private, only an admin sees the value.
- **Google-managed TLS, not Let's Encrypt (D64).** An early assumption of certbot-style certificate management did not survive the serverless hosting choice.
