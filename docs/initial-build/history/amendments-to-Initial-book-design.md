The following are new amendments to the initial design of the PBE Address Book, being added after the design review completed.

* On rare occasions, a brother of PBE can be “de-brothered” in extreme circumstances. We need a flag in the profile schema to indicate whether a member has been de-brothered, and an accompanying admin-only toggle in the UI, that removes visibility of the record from brothers, and visually flags it for managers and admins (perhaps a strikethrough on their name, so it’s visible and clear that they’re de-brothered, while still allowing basic visibility of the profile record?). Setting this flag should also prevent the de-brothered user’s access to *any* Book or Ghost data. If the de-brother flag is set, none of their data should be shared with brothers (other than managers and admins).
	* How do we disable access to Ghost?
		* Option 1: Delete them in Ghost but keep them in Book; this ensures they can’t login and that they won’t receive email, but requires gymnastics to code it in Book because it’s such an exception to normal brother handling.
		* Option 2: Change their email address in both Book and Ghost to something fake
			* If they’re still in Ghost, we have to also turn off their subscriptions to all newsletters so newsletters don’t get sent to their fake address and bounce.
			* This seems much easier to implement than Option 1 and maintains our principle that Book and Ghost data stay in sync.
* Put a link to the Ghost privacy policy in the Book UI. Probably in a footer at the bottom of the page: it’s important it be there, but we don’t have to hit people over the head with it. (The need for the link may already have been captured—check—but not the position on the page).
* There needs to be an optional system message banner across the top of all pages in Book. Only admin can set it or remove it. It has the same function, and should have the same appearance, as the optional banner Ghost has (though the messages would be used for different things at different times—there’s no need for banner messages from Book to propagate to Ghost, or vice versa).
* There needs to be a way to display a “down for maintenance, check back later” display (not just a system message banner—the system itself is down).
	* How would we trigger it? A script, or an Admin page feature? (I think a script).
	* It would be ideal if this page would show up if something is broken, but I’d settle for something an admin could turn on/off.
* Need to display a “Loading” pop-up, maybe with an animated revolving icon, displayed to the user during both initial backend cold-start and the download of data from the backend to the frontend in their browser. If all’s well it won’t show up at all, but if the frontend detects it’s going to take more than a short time (1s? less?), it would display this message. The message would be taken down once the server has downloaded the full dataset to the browser and the user can begin using Book.
* Ghost already logs when a newsletter email sent to a brother bounces. I’d like to evaluate whether we can send that bounce information to Book so it can display a BOUNCING warning in the UI, perhaps across the top of the page, when a user’s own email is bouncing, and on the directory and profile views for managers and admins. An alternative: make generation and display of a Ghost bounce report across all users an option on the admin page? I’d prefer the first, but if it’s too complex, I’d settle for the second.
* There needs to be a visible and easy-to-use “report a bug” link at the top of all Book pages, perhaps next to the user’s avatar icon/menu. If the link sends an email to the admin, we need to be careful of the security implications (e.g. as a vector for DoS attacks).
