# PBE Address Book — Coding Project Plan

This is the build-and-launch plan for Book. It assumes the design is settled: the product is specified in `PRD.md`, the data model in `DATABASE-SCHEMA.md`, the architecture and operations in `ENGINEERING-DESIGN.md`, and the HTTP surface in `API-SPEC.md`. Nothing here re-decides design; it sequences the work of building what those documents describe, and it names the deployment and data-migration steps that turn a finished codebase into a running service at `book.pbe400.org`.

## 1. How to read this plan

The plan is organized as guiding principles, then the things that exist for the whole life of the build (the repository layout, the environments and infrastructure, and the CI/CD pipeline), then the build itself as a sequence of phases, and finally the one-time pre-launch migration and cutover. The phases are dependency-ordered and each ends with a gate that must be green before the next begins; the gate discipline is described in §5 and restated at each phase. Cross-references point at the design docs rather than repeating them — when this plan says "the privacy projection," the authority for what that means is `ENGINEERING-DESIGN.md` §2 and `DATABASE-SCHEMA.md` §9, not this file.

The plan is written to be executed by a small team — realistically one developer working with AI assistance — so it favors a single coherent thread of work over parallel tracks, and it front-loads the integration risks that are cheapest to discover early and most expensive to discover late.

## 2. Guiding principles

**Walking skeleton first.** Before any feature is built out, a razor-thin slice runs end to end through every real architectural component and is deployed to a real environment. That slice does almost nothing — sign in, read one profile, render a list — but it exercises the Ghost auth bridge, the Cloud Run backend, the Firestore read, the privacy projection seam, the SPA shell, and the full deploy pipeline with managed TLS. The hardest and least-reversible risks in this project are the Ghost integration and the hosting/DNS/TLS wiring; the walking skeleton forces them into the open in the first days of work rather than the last.

**Vertical slices over horizontal layers.** After the skeleton, features are built page by page as complete vertical slices (schema → validation → API → projection → UI → tests), not layer by layer (all endpoints, then all UI). Each phase therefore ends with something a person can actually use and a reviewer can actually exercise, which keeps integration continuous instead of deferred to a big-bang assembly at the end.

**Security core early.** The server-side privacy projection, the role/capability matrix, and optimistic-concurrency control are the parts where a mistake is a data-exposure or data-loss incident, not a cosmetic bug. They are built and tested as their own phase (Phase 2) immediately after the skeleton, before the feature pages that depend on them, so that every later phase sits on an already-proven access-control floor.

**A tests-green gate between phases.** Per the testing strategy (`ENGINEERING-DESIGN.md` §6.6; DECISIONS D65), the rule "test anything with non-trivial logic" applies as the code is written, not after. A phase is not complete until its tests — unit/integration under Vitest against the Firestore emulator, plus the relevant Playwright end-to-end paths — pass in CI, the accessibility checks for any new pages pass (DECISIONS D67), and the branch deploys cleanly to staging. The next phase starts from that green state.

**Test against fake data and a dev identity from day one.** The committed deterministic fake-data generator (DECISIONS D65) and the `DevIdentityProvider` (DECISIONS D72) are built in Phase 0, before anything that needs them, because everything that follows is developed and tested against them. They are the substrate the whole build runs on locally and in CI.

**Lean delivery for slow connections.** A meaningful fraction of the brotherhood are older members on slow links (the constraint recorded with DECISIONS D73–D76), so the build is frugal about bytes from the start rather than as an afterthought: the static bundle is browser-cached with content-hash immutability (D73), kept small via minification, tree-shaking, and per-page code-splitting that lazy-loads the heavy surfaces — Admin, the crop modal, the phonetic libraries — off the Directory landing page (D74), and the bulk dataset is served compressed (brotli, precomputed off the request path; D75/D84) but `no-store`, because the projected payload is real PII that must not persist on a shared machine's disk (D95, superseding D76's read-side conditional-GET). These are wired as their underlying pieces are built (read path, Hosting config, CI) and verified at scale in Phase 7.

## 3. Repository and project layout

Book is a single Git repository (created at the start of Phase 0 — the design-planning folder is deliberately not yet a repo). It is a TypeScript monorepo holding the frontend SPA, the backend service, the code they share, and the supporting tooling, so that the shared validation module and the type definitions are imported by both sides rather than duplicated.

The intended top-level shape:

| Path | Contents |
|---|---|
| `apps/web/` | The React + Vite SPA (shadcn/ui, TanStack Table/Virtual, React Router + nuqs). |
| `apps/api/` | The Node + TypeScript backend on Cloud Run (REST under `/api/*`, the Firestore snapshot-listener cache, the privacy projection, the Ghost push/reconcile, the roster endpoint). |
| `packages/shared/` | Types and the **shared client/server validation module** (DECISIONS D50), the field-visibility taxonomy, the Canonical Name construction, and other logic that must be identical on both sides. |
| `packages/help-content/` | The single-source help-content entries (DECISIONS D53; §10 here) — imported by the SPA for in-page help and consumed by the manual assembly step. |
| `tools/fake-data/` | The deterministic seeded fake-data generator and its small committed fixtures (DECISIONS D65). |
| `tools/migration/` | The one-time pre-launch migration utilities (see §9 and `PRE-LAUNCH-TOOLS.md`) — kept in-repo for history but never part of the deployed service (DECISIONS D57). |
| `docs/` | Documentation, subdivided **by build** so future features get their own coherent doc sets, with a top-level `README.md` indexing the builds. This initial release lives in `docs/initial-build/` — the delivered design docs (`PRD.md`, `DATABASE-SCHEMA.md`, `ENGINEERING-DESIGN.md`, `API-SPEC.md`, this plan, `USER-MANUAL.md`, the `DECISIONS.md` log, and `PRE-LAUNCH-TOOLS.md`), checked in beside the code per the global "docs ship with code" rule — with origin/history documents (`SEED.md`, `DESIGN-HISTORY.md`, `DESIGN-REVIEW-PLAN.md`, the consolidated `DESIGN-REVIEW-COMPOSITE.md`, and the triage/resolution records) set apart under `docs/initial-build/history/` so the engineering docs don't depend on them. |
| `.github/workflows/` | The CI/CD pipeline (§5). |

Tooling is uniform across the repo: TypeScript end-to-end, Biome for format and lint (DECISIONS, Session 1), Vitest for unit/integration tests, Playwright for end-to-end, and the Firestore emulator for any test that touches the datastore.

## 4. Environments and infrastructure

There are three environments, and the distinction between them is load-bearing for both testing and safety.

**Local** is a developer's machine: Vite dev server, the backend run locally, the Firestore emulator, the fake-data generator, and the `DevIdentityProvider` for Ghost-free role-switchable sign-in. No cloud resources and no real member data are involved.

**Staging** is a complete, independent cloud deployment — its own GCP project, its own Firestore, its own GCS bucket — holding **fake data only** (DECISIONS D72), and it is **ephemeral and script-provisioned** (DECISIONS D102): rather than a standing second environment, setup/teardown infrastructure-as-code stands a throwaway Book up on demand (the lightest form being a persistent-but-*empty* staging project into which the scripts provision an ephemeral Firestore database and bucket), and those same scripts double as the single-region DR runbook and the periodic backup-integrity harness (§6.3 of `ENGINEERING-DESIGN.md`; inventoried in `PRE-LAUNCH-TOOLS.md`). It exists for system testing, user-acceptance testing with real alumni, and pre-launch migration dry runs. Staging authenticates through the **real Ghost auth bridge** by way of an additive route on the live Ghost Pro site (the D72 mechanism that gets full-fidelity auth testing without a second paid Ghost instance), while its Book→Ghost **write** path is pointed at a stub-and-log or a short-lived throwaway Ghost — **never at production Ghost**, which would mutate live members.

**Production** is the live service at `book.pbe400.org`: its own GCP project, Firestore, and GCS bucket, the real Ghost integration both ways, and the real member data.

The infrastructure is the same shape in staging and production:

| Component | Role |
|---|---|
| **Firebase Hosting** | Serves the SPA static bundle, fronts the custom domain (Google-managed TLS), and rewrites `/api/*` and `/img/*` to Cloud Run — so the whole app is one origin with **no external load balancer** (DECISIONS D126). |
| **Cloud Run** | The backend API service; scales to zero and is **capped at one instance** (`max-instances=1`), so the single authoritative instance's in-memory cache is the source of truth, with the Firestore snapshot-listener demoted to a deploy-window safety net (DECISIONS D83/D26). |
| **Firestore** | System of record: the `profiles`, `users`, and `majors` collections. |
| **Cloud Storage (GCS)** | Headshot and thumbnail objects (versioned, immutable URLs; **private** — served by Cloud Run, not a CDN, DECISIONS D126) and backup archives. |
| **Cloud Logging** | The three structured log streams — access, diagnostic, audit (DECISIONS D61). |
| **Google-managed TLS** | Certificates on Hosting and Cloud Run; HSTS and forced HTTPS; no Let's Encrypt/certbot (DECISIONS D64/D126). |

Identity and secrets follow the design's two distinct paths. User auth rides the Ghost bridge (`IdentityProvider` seam; DECISIONS D20/D21). Machine access — the Linter's roster pull (DECISIONS D58) and the log-reader agent (DECISIONS D61) — uses **GCP service-account OIDC identity tokens off the Ghost path**, with no static API keys to leak. Application secrets live in Secret Manager and are injected into Cloud Run at deploy time.

DNS for `book.pbe400.org` is the last piece wired in production: a staging hostname is used throughout the build, and the production custom domain is pointed at Firebase Hosting only at cutover (§9), after the service behind it is proven.

## 5. CI/CD pipeline

CI runs on GitHub Actions on every push and pull request. The pipeline is the mechanical enforcement of the tests-green gate: format and lint (Biome), type-check, the Vitest unit/integration suite against the Firestore emulator, the Playwright end-to-end suite, the accessibility checks (axe-core via Playwright, plus the CI contrast gate; DECISIONS D67), a **bundle-size budget** that fails the build if the shipped JS grows past a set threshold (DECISIONS D74), and a **build assertion that the production artifact cannot instantiate the `DevIdentityProvider`** (DECISIONS D108). A red pipeline blocks merge.

Deployment is continuous to staging and gated to production. Merging to the main branch builds and deploys to **staging** automatically, so staging always reflects the current main branch. **Production** deploys are a deliberate, manually-approved promotion of an already-staged build — never an automatic consequence of a merge — because production carries real member data and the live Ghost write path.

Database changes follow the migration discipline from DECISIONS D71: additive/optional schema changes need no migration (the read path and the shared validation module supply defaults), while breaking changes ship as numbered, idempotent, dry-run-able migration scripts that are emulator-tested against fake data, run only after a backup, logged to the audit stream, and ordered by the `_meta` schema-version marker.

## 6. Testing in the build

The testing strategy itself is specified in `ENGINEERING-DESIGN.md` §6.6 (DECISIONS D65–D67); this section only says *when* each kind of test enters the build. Unit and integration tests are written with the code in every phase, never deferred — the gate depends on them. End-to-end Playwright coverage grows page by page: each feature phase adds the happy-path and key edge-case flows for the page it builds, and Phase 7 completes the suite into a comprehensive regression net. Accessibility verification (the three layers of D67) applies from the first page that ships and gates every phase that adds UI. The phonetic-matching A/B (DECISIONS D66) is a build-time offline-harness experiment run during the Directory phase, not a runtime feature. Browser MCP tools are for interactive, exploratory verification during development; the committed automated suite is Playwright.

## 7. Build phases

Each phase below states its goal, its main work, its exit gate, and how it divides into **context-sized implementation sessions** — the unit of work that fits comfortably in a single AI-assisted coding session, named in the same `a`/`b`/`c` spirit the planning phase used (and just as revisable: a session may split or merge as the work reveals its true size). The per-phase session counts are gathered in the §8 summary; the splits are first cuts, not commitments. Every gate implicitly includes the standing rule: CI green (format, lint, type-check, unit/integration, end-to-end, a11y — including the page's **baseline labels and any instructions a control needs to be usable**, which ship with the page rather than waiting for Phase 6, per DECISIONS D111), and a clean deploy to staging.

### Phase 0 — Scaffolding and substrate

**Goal:** an empty but fully wired project that can be tested and deployed.

Create the Git repository and the monorepo layout (§3). Stand up TypeScript, Biome, Vitest with the Firestore emulator, and the Playwright harness. Establish the GitHub Actions pipeline (§5) so the gate exists from the first commit. Build the two pieces of substrate everything else relies on: the **deterministic seeded fake-data generator** (DECISIONS D65) — committing the generator and small fixtures, not a data blob, producing ~600–2000 obviously-fake profiles that span every feature — and the **`DevIdentityProvider`** (DECISIONS D72), built behind the **four independent production lockouts of D108** (compiled out of the prod bundle as the load-bearing layer, plus the retained env gate, the CI assertion, and a startup alert) so it can never run in production. Scaffold the shared validation module and the help-content package as empty-but-importable packages. Wire the **visual-design token layer** — [`visual-design/tokens.css`](visual-design/tokens.css) into the shadcn `globals.css` (`:root` / `.dark`, the D29 CSS-variable token layer) — so every screen built in later phases inherits the final look from the start; the completed design pass in [`visual-design/`](visual-design/) (README + `COMPONENTS.md` + `tokens.css` + the HTML prototypes) is the visual source of truth, recreated with shadcn primitives rather than shipped (VISUAL-DESIGN-BRIEF).

**Gate:** the pipeline runs and is green on a trivial test; the fake-data generator produces a deterministic dataset into the emulator; `DevIdentityProvider` yields a role-switchable local session.

**Sessions (1):** a single session — S0 — carries the whole of scaffolding.

### Phase 1 — The walking skeleton

**Goal:** a thin end-to-end slice signed-in and deployed to staging.

Stand up the Cloud Run backend with a single read path (a minimal `GET /api/profiles` from the single instance's in-memory cache, returning projected data; `API-SPEC.md`) — **served compressed and `no-store` from the start** (the projected payload is real PII that must not persist on a shared machine's disk; DECISIONS D75/D84/D95), and with the static-asset cache-header policy (DECISIONS D73) set as part of the Hosting config. Wire the `IdentityProvider` seam (DECISIONS D20/D21) with the `DevIdentityProvider` locally and the **real Ghost auth bridge via the additive staging route** (DECISIONS D72) — verifying the Ghost-issued JWT against Ghost's JWKS, the token carried in the URL fragment as designed — with Book's **session record and the single-use login `state` nonce persisted in Firestore** (the `sessions` and `authNonces` collections, each TTL-expired on `expiresAt`, with a read-through in-memory cache for warm requests) so a scale-to-zero cold start doesn't invalidate sessions or force re-login, the 4-hour absolute cap unchanged (DECISIONS D125). Build the persistent SPA shell (app shell, identity, role badge, routing via React Router + nuqs) — including the **persistent privacy-notice footer** (DECISIONS D116), the **system-banner slot** the shell renders when one is set (D117), and the **threshold-gated cold-start loading overlay** (D119) — and render the directory list of fake brothers. Provision staging end to end: GCP project, Firestore, GCS, Firebase Hosting (rewriting `/api/*` and `/img/*` to Cloud Run — no external load balancer or Cloud CDN, DECISIONS D126), Cloud Run, Google-managed TLS, on a staging hostname.

**Gate:** a user can sign in on staging through the real Ghost bridge (and locally through Dev), see a list of fake brothers served through the projection, with the session surviving a scale-to-zero cold start (Firestore-persisted; D125), and the whole thing is deployed with HTTPS. The integration spine is proven.

**Sessions (2):** 1a — the backend, the Firestore wiring, the single read path, and the staging infrastructure bring-up (Cloud Run, Hosting with `/api`+`/img` rewrites to Cloud Run, GCS, managed TLS, staging DNS — no external load balancer or Cloud CDN, DECISIONS D126); 1b — the Ghost auth bridge (the Handlebars relay page, JWKS verification, the fragment-carried token, the `IdentityProvider` seam wired to both Dev and the real Ghost staging route, with Firestore-persisted sessions and login nonces — D125) plus the SPA shell and the rendered list. The bridge gets its own session because it is the single fiddliest, least-reversible integration in the project.

### Phase 2 — Data and permission core

**Goal:** the security-critical floor every feature stands on.

Implement the full `Profile` schema and sub-types, the **shared client/server validation module** (DECISIONS D50), and the Canonical Name construction with load-time ambiguity detection (DECISIONS D15/D12). Build the **server-side privacy projection** enforcing the full field-visibility taxonomy — public / toggle / restricted / private, plus the staff-internal and system/internal classes (DECISIONS D16/D44/D56) — including the **`unlisted` whole-record hide** that omits a profile from the brother-role projection by reusing the de-brothering projection mechanism (DECISIONS D124/D115) while managers/admins still see it — and the **role/capability matrix** for brother / manager / admin (DECISIONS D19, `ENGINEERING-DESIGN.md` §2), whose write rules let an owner set their own `unlisted` (PATCH self-service) and an admin set another's, but not a manager (D124). Implement **optimistic concurrency** on the Firestore `updateTime` token (`ETag`/`If-Match`/`412`; DECISIONS D25) on the write paths, and the **audit log stream** recording actor, target, action, and changed-field *names* — never values (DECISIONS D61). This phase is unit-test-heavy: the projection and the capability matrix get exhaustive role-by-field coverage, because they are the exposure surface.

**Gate:** the projection and capability matrix pass exhaustive tests across all roles and visibility classes, including an `unlisted` record absent from the brother projection but visible to managers/admins (D124); OCC correctly rejects stale writes with a 412; audit entries are emitted with names-not-values.

**Sessions (3):** 2a — the full schema and sub-types, the shared validation module, and the Canonical Name construction with ambiguity detection; 2b — the server-side privacy projection across all visibility classes (including the `unlisted` whole-record hide reusing the de-brothering mechanism — D124/D115) and the role/capability matrix, with exhaustive role-by-field tests; 2c — optimistic concurrency (`If-Match`/`412` and the reconcile seam) and the audit log stream. (2c could fold into 2b if both come in light, but keeping 2b purely on the exposure surface is the safer default.)

## 9. Pre-launch migration and cutover

The application being finished is necessary but not sufficient for launch: Book has to come up holding real, cleaned member data, and the Ghost site has to be flipped from "members edit themselves in Ghost" to "members are managed in Book." Both are one-time operations performed by standalone tooling, not by Book (DECISIONS D57), and they are inventoried in `PRE-LAUNCH-TOOLS.md`.

**The data merge and load.** The source data is messy and spread across several systems, so the merge is substantially a manual human-plus-AI effort, with the tools doing the mechanical pulls and the final write. The hard ordering constraint is that the **Ghost-account dedup must come first** — collapsing the multiple Ghost accounts many brothers hold today down to one primary email per brother (the mystery-email project), because Book's whole design assumes a single primary email per brother. Only then can the **pull-and-seed** utility read each brother's Ghost member id (→ `ghostMemberId`), notes (→ `adminNote`), and current newsletter/comment-reply state (seeding those preferences *from* Ghost so existing opt-outs are honored; DECISIONS D57). The merged dataset — Constitution roster, MITAA data, Ghost fields, family-tree edges for `bigBrotherId`, and yearbook headshots/early class years — is then written by the **initial bulk-loader**, a one-time external tool (Book itself has no online bulk-write path, D100) that, as the genesis load, is permitted to write `ghostMemberId` and the verification/system fields directly, exempt from the in-app write rules (DECISIONS D57/D68).

**Dry runs on staging.** The entire migration is rehearsed against staging (fake or anonymized data, never production Ghost) until it runs clean, so that the production run is a repeat of a proven sequence rather than a first attempt.

**User-acceptance testing.** Real alumni exercise staging through the `DevIdentityProvider` for role-switchable access (DECISIONS D72), validating the product against actual users — the cohort that includes brothers 60+ with vision and fine-motor needs — before the production flip.

**Cutover.** When staging is proven and the migration rehearsed, production comes up in order: bring up the production environment and run the real data load; perform the **one-time initial Ghost↔Book sync**; **disable Ghost member editing and redirect its account UI to Book** (the theme override, with the JS-injection fallback; DECISIONS D55); point `book.pbe400.org` DNS at Firebase Hosting and let Google-managed TLS provision; and go live. After go-live the synchronous Book→Ghost push is the steady-state path, and the admin "Sync with Ghost" audit (also run on a schedule, D99) is available to catch any drift.

## 10. The help-content single source (contract for Phases 3–6)

Per `PRD.md` §5.9 and DECISIONS D53, the in-page help and the `USER-MANUAL` must share one source so they cannot drift. The mechanism is a structured **help-content module** (`packages/help-content/`): a set of entries keyed by a stable control identifier, each entry carrying the control's **label**, its **helper text** (the always-visible `aria-describedby` line — e.g. the class-year range/list hint), an optional **placeholder** example, and the **toggle-tip body** (the deeper "what is this and how do I use it" explanation revealed by the `CircleHelp` Popover). The SPA imports these entries directly to render the layered help. The **wiring is split along the WCAG line** (DECISIONS D111): the **baseline label and helper-text/instruction** entries a page's controls need to be usable wire in **with each page** in Phases 3–5, so each page's a11y gate is honest, while the **enriched `?` toggle-tips and the manual assembly** land in **Phase 6**. The manual's per-control reference section is **assembled from the same entries** by a build step, wrapped in the manual's narrative sections (which are authored separately because they describe flows, not individual controls). A help string therefore exists in exactly one place; both the running UI and the manual read it from there. The full shape of an entry and the assembly step are specified alongside `USER-MANUAL.md`.

## 11. Risks and sequencing notes

The riskiest dependency is the **Ghost integration**, which is why the auth bridge is in the walking skeleton (Phase 1) and the write path is isolated to a non-production Ghost throughout the build (Phase 5; DECISIONS D72) — the one thing that must never happen during development is pointing the Book→Ghost write path at production Ghost and mutating live members. The second risk is the **pre-launch data merge**, which is manual and cross-source; it is mitigated by rehearsing the entire migration on staging until it is repeatable before the production run. The **dedup-before-load** ordering in §9 is a hard prerequisite, not a preference. Finally, the **privacy projection and capability matrix** are built and exhaustively tested as their own early phase precisely because they are the highest-consequence code in the system; no feature phase is allowed to weaken or bypass them.
