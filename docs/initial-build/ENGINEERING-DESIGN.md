## 1. Architecture Overview

Book is a custom-built single-page application (SPA) over a small dataset (fewer than ~2,000 records, growing 30–50/year). The defining design choice: the browser downloads the entire dataset the user is permitted to see and performs all search, filter, sort, and display locally, while the backend is a thin serverless API handling authentication, per-user field projection, validation, persistence, media, and external sync. This yields instant client-side interactivity, a minimal and cheap backend (free-tier-feasible), and a clean fit for serverless plus document storage.

Book is built from scratch rather than on a server-centric framework such as Django. At this scale the client owns the data and does the querying, which drains most of such a framework's value; adapting one to a serverless + bulk-download + SPA shape would add brittle complexity for little benefit. AI-assisted development collapses the build-cost advantage frameworks once offered.

### 1.1 Technology stack

- **Frontend:** React + TypeScript, built with Vite. A pure client SPA — no SSR or meta-framework (e.g. Next.js), which this app does not need.
- **Backend:** Node.js + TypeScript on Google Cloud Run (serverless).
- **System of record:** Firestore (one document per profile, plus a users/permissions collection).
- **Object storage:** Google Cloud Storage for headshots, thumbnails, and database backups, with object versioning enabled.
- **Static hosting:** Firebase Hosting / CDN serves the SPA bundle.
- **Shared types:** a single `Profile` TypeScript type is defined once and imported by both frontend and backend, so the record shape cannot drift across the wire.
- **Tooling:** Biome for both formatting and linting, enforced via format-on-save, a pre-commit hook, and a CI check.
- **External services:** Ghost (authentication bridge and membership sync), Mixpanel (analytics), Google-managed certificates (TLS — §6.4).

### 1.3 Data flow

- **Authentication:** unauthenticated users are bridged through pbe400.org/book (Ghost) to obtain a token; Book then sets its own cookie under book.pbe400.org. Mechanism detailed in Session 3.
- **Bulk load:** on start, the SPA calls `GET /profiles`; the backend returns the entire dataset projected down to the fields the requesting user is authorized to see, as one JSON payload — a **uniform per-role projection**, with the caller's own full record fetched separately via `GET /api/me` and overlaid by the SPA (the split read, decision D82). It serves this from an in-memory cache, so client bulk-loads incur zero Firestore reads — reads occur only on cold start (a single GCS-snapshot hydration, §1.5/decision D85) and on writes. The payload is compressed on the wire (§1.6) but served **`no-store`** — it is real PII and must never persist to a shared machine's disk, so it lives only in memory while the tab is open and is re-fetched on the next fresh load (decision D95, superseding the earlier conditional-GET/`304` scheme).
- **Local operations:** all search, filter, sort, and column selection happen in the browser over the in-memory dataset. Because filtering and sorting can touch *any* field (not only displayed columns), the full permitted record set is downloaded rather than a partial projection.
- **Thumbnails:** served by the backend from the private GCS bucket (same origin, via the `/img/*` Firebase Hosting → Cloud Run rewrite) as individually addressable, immutable, browser-cacheable objects, lazy-loaded with a viewport lookahead margin and optional idle-time prefetch of the current result set in scroll order. Precomputed on headshot upload. (Exact prefetch policy is tuned in Session 4; the app-served image model is decision D126, which replaced the Cloud CDN of D23.)
- **Headshots:** fetched per profile via `GET /headshot/:id` when a profile is opened.
- **Writes:** `POST` / `PATCH` / `DELETE /profile`, written through to Firestore synchronously — memory is never the system of record. Optimistic concurrency via Firestore's `updateTime` token (`ETag` / `If-Match`), designed in Session 3 (§1.5/§2.6, decision D25).
- **System banner:** the SPA fetches the global system banner (`GET /api/banner`) on load and renders it across the top of every page; an admin sets or clears it via `PUT /api/admin/banner` (a small `config` singleton written through the single instance — decision D117). Bug reports submitted from the app (`POST /api/bug-report`, decision D121) are stored and audited, never emailed.

### 1.4 Privacy enforcement

Per-field visibility is enforced on the server. Each response is projected to the fields permitted by the requester's role and the record's privacy flags, so the frontend never receives data the user may not see. This closes the DevTools / network-inspection leak that a frontend-only filter would expose.

**The split read sharpens this single enforcement point (decision D82).** `GET /api/profiles` returns a *uniform per-role* projection of every record — identical for all callers of the same role, including a plain role-projection of the caller's own record — and the caller's own full record is delivered out-of-band by `GET /api/me`. Factoring the per-*user* part (your own record in full) out of the bulk payload makes that payload cacheable and removes the failure mode where the first post-write caller's owner-aware projection could be served to the next caller (the most serious review finding, S1): no two callers of different entitlement can receive each other's projection (asserted by a cross-caller isolation test, §6.6).

**Writes are governed by the same single point, on the write side (decision D106).** A **positive per-role writable-field allowlist** — the write-side dual of the read projection, enumerated alongside the capability matrix (PRD §4) — rejects any out-of-allowlist field rather than silently dropping it, while an object-level predicate (`profileId == session.profileId OR role ∈ {manager, admin}`) blocks the IDOR that a bare "is authenticated" check would allow on contiguous, guessable Constitution IDs. System/verification/Ghost fields are unwritable through PATCH by every role (set only by dedicated server actions), and consent/privacy fields are owner-only. This extends the credited server-side projection rather than bolting on a second enforcement mechanism.

**De-brothering is the one whole-record projection exception (decision D115).** A profile flagged `debrothered.isDebrothered` is **dropped from the brother-role projection entirely** — it is absent from the brother bulk payload, search, filters, and export, and a brother requesting it by id gets `404` — while managers and admins still receive it (with the flag set, so the UI strikes the name through). The `debrothered` state is set only by a **dedicated admin action** (`PUT /api/profiles/{id}/debrothered`, API-SPEC §3), never via PATCH, because raising it also deletes the Ghost member and denies sign-in (§2.1/§5.1); the consent/verification snapshot it captures is system-internal. This is the one *admin-action* case where the projection removes an entire record, not just fields, for a role.

**Unlisting is the second whole-record projection exception (decision D124).** A profile with `unlisted = true` is likewise dropped from the brother-role projection entirely — absent from the bulk payload, search, filters, and export, with a by-id request returning `404`/`403` — while managers and admins still receive it (flag set, so the UI badges it "UNLISTED"). Unlike de-brothering it is **not** a dedicated admin action and carries **no Ghost-member or sign-in side-effects**: `unlisted` is an ordinary **owner-writable privacy field on PATCH** (the owner sets his own; an admin may set another's; a manager may not — the owner-only consent/privacy rule above), and the member stays fully active, still flowing to MITAA per `allowShareWithMITAA`. It governs directory *listing* only.

### 1.5 Concurrency & caching

Writes are written through to Firestore so durability never depends on flushing memory at shutdown (a Cloud Run instance can crash or be lost without a clean SIGTERM).

**Single authoritative instance (decision D83).** Cloud Run is capped at **`max-instances = 1`** while keeping **scale-to-zero** (`min-instances = 0`). With at most one instance there is no second writer, so the single instance — which updates its own cache synchronously on every write and is the only path writes flow through — is **authoritative by construction**. This replaces D26's "self-healing and correct at any instance count" framing: on scale-to-zero Cloud Run an idle instance is CPU-throttled, so a background snapshot listener could stall and then serve *stale* data while `304`-confirming it (the review's most serious operational finding, R2); the single-instance cap dissolves that at the root, for **$0**, rather than buying an always-warm instance (≈$180/yr). The conscious tradeoff: Book uses Cloud Run for its **scale-to-zero cost floor, not for autoscaling** — ample capacity at ≈700 brothers with bursty bi-annual traffic. The **Firestore snapshot listener is retained only as a convergence safety net** for the brief two-revision overlap a deploy creates and for out-of-band writers (an offline restore, §6.3), no longer as the correctness mechanism; deploys cut traffic to the new revision immediately, and a **listener-health watchdog** (resume-token / last-event age) plus a **cache-age / listener-disconnect alert** (§6.1) remain as cheap insurance.

**Cold-start hydration from one GCS read (decision D85).** Because scale-to-zero makes cold starts common, each one previously re-read all ~2,000 profile documents from Firestore and recompressed — seconds of latency for exactly the slow-connection cohort, at the worst moment (the first click-through after a newsletter send). Instead, the backend maintains a **backend-internal GCS snapshot object** holding the raw dataset, the precomputed brother-role brotli buffer (§1.6, decision D84), and the dataset version/ETag, **regenerated on every write**; a cold-starting instance **hydrates from this one GCS read** (and rebuilds its in-memory email index, §5.1/D97) and can serve the brother buffer immediately. The snapshot is strictly backend-internal — clients never read it; profile data is always served through `GET /api/profiles` under the per-role projection and session auth — so access control is never delegated to GCS ACLs. A write that commits to Firestore but fails to refresh the snapshot is caught by reconciling the snapshot version against Firestore's latest on cold start and falling back to a full read on divergence. The small `majors` vocabulary rides the same snapshot; the per-user `users` documents are small and low-traffic and read on demand.

Concurrent edits to the *same* record are handled by optimistic concurrency: Firestore's `updateTime` is the concurrency token, surfaced to clients as an `ETag` and required back as `If-Match` on writes, with a stale write rejected as `412` (see §2.6 and API-SPEC §1.4).

### 1.6 Delivery & wire efficiency

A meaningful fraction of the brotherhood are older members on slow connections (the product requirement is in `PRD.md` §5.5; decisions D73–D76 record the constraint and the choices), so Book is deliberately frugal about bytes on the wire — both the code it ships and the data it serves. Four measures, all in MVP:

**Browser caching of the SPA bundle (D73).** Vite emits content-hashed asset filenames, so every asset URL is immutable by construction. Hashed JS/CSS/asset files are served with `Cache-Control: public, max-age=31536000, immutable` and the single unhashed `index.html` with `no-cache` / `must-revalidate`, configured in the `firebase.json` headers block. A returning visitor re-downloads only the chunks whose hash changed, while the always-revalidated HTML — which points at the current hashes — makes a deploy propagate with no stale-bundle/fresh-API skew. There is no service worker / offline PWA cache: persisting member contact data on-device is a privacy and staleness cost unjustified for an online-only directory.

**A small JS bundle (D74).** Production builds rely on Vite's minification (esbuild) and Rollup tree-shaking, with narrow imports (e.g. per-icon `lucide-react`) so unused code is dropped. The initial load is **code-split** so the Directory — the universal landing page — ships lean, lazy-loading the heavier, rarely-first-touched surfaces: the Admin page, the react-easy-crop headshot modal (§ Profile, D47), and the talisman phonetic-matching libraries (Name Search, D35). A CI bundle-size budget guards against a heavyweight dependency slipping in. (Firebase Hosting also compresses static assets over the wire, so the bundle ships minified *and* brotli'd.)

**Compression of the bulk dataset (D75, off the event loop per D84).** The `GET /profiles` payload is compressed at the application — Cloud Run does not transparently compress responses. The backend serves **brotli (level 11)** to clients advertising `Accept-Encoding: br` and **gzip** to the rest, via standard `Content-Encoding` / `Vary: Accept-Encoding` negotiation, so the browser inflates the body transparently and the SPA's `fetch()` only ever sees JSON. The original design compressed lazily on the **request path**, which on the single-threaded Node event loop meant a latency spike for the unlucky first-caller-after-a-write and, under repeated edits, a low-privilege CPU DoS (finding R3). Compression therefore moves **off the event loop (decision D84)**: on each write the instance recomputes the **brother-role** projection and brotli-11-compresses it on the libuv threadpool (async `zlib.brotliCompress`), swaps the buffer in atomically, and writes it into the GCS snapshot (§1.5/D85) — so a cold-starting instance *loads* a ready-to-serve buffer rather than computing one, and **brotli-11 is retained** (maximum ratio for the slow-connection cohort) with no request-path or cold-start penalty. Rebuilds are coalesced by a time-based **debounce** for organic bursts and an explicit **"regenerate once after the batch"** hook for any known-batch operation, so a multi-row change triggers one recompression, not N. The **manager and admin** projections — computed fresh per request (D82), few callers — are compressed async at a **moderate level (~brotli-5)**, gzip fallback for non-`br` clients. At ~2,000 records of repeated keys the payload compresses ~80–90%+; because it gates app startup this is the highest-leverage byte reduction in the system. The Firestore↔backend hop is deliberately left uncompressed: rare (cold start), intra-datacenter, billed per-operation not per-byte.

**No on-disk PII: the bulk read is served `no-store` (decision D95, superseding D76).** The earlier design served `GET /profiles` `private, no-cache` with an `ETag`/`304` conditional-GET so a returning browser could revalidate cheaply. But that persisted the projected dataset — every brother's name plus shared emails/phones, **real PII** — to a shared or public machine's disk, with no reliable browser hook to purge it on tab close (finding P11). `GET /profiles` is therefore served **`Cache-Control: no-store`**: never written to disk, held only in memory while the tab is open, re-fetched on the next fresh load. This **supersedes D76's read-side conditional GET** (and dissolves the S6 "cache key must include role" concern, since there is no read-side cache to key); the **write-side** `If-Match`/`updateTime` optimistic-concurrency token of §2.6 is unaffected. `GET /api/me`, which now carries the caller's own contact values, is served `no-store` for the same reason. (A planned **Phase 7.5** — §1.7, decisions N62/N63 — recovers the revalidation *bandwidth* this decision gave up, by reintroducing a conditional read at the **application** layer over a heap-only store while keeping `no-store` and never persisting PII to disk; it does not reinstate D76's browser-cache conditional GET.)

**Client-side index built in a Web Worker (decision D110).** The Name-Search fuzzy/phonetic index — Fuse.js plus phonetic codes over ~2,000 profiles × several name fields (D35) — is built **in a Web Worker**, not on the main thread, so SPA init never janks on the older hardware the rest of the design optimizes for. (The phonetic algorithm was settled by the Phase-3b A/B as **Beider-Morse**, sourced from the `bmpm` package since talisman ships no BM; Double Metaphone — the runner-up — remains talisman's. See N19/D66.) The directory grid renders immediately with **exact/substring matching on the main thread**, and **fuzzy + phonetic search switches on when the worker posts "index ready"** (progressive enhancement; the dataset reaches the worker by a structured-clone `postMessage` of the already-in-memory profiles). D35's IndexedDB memo is **dropped** — persisting name-keyed phonetic codes would re-introduce the on-disk PII residue `no-store` (D95) exists to prevent — so the codes are recomputed in the worker each load (cheap off the main thread). The same worker index also applies a **curated diminutives expansion** (decision D123): given-name tokens, and the query, are expanded through a small bundled nickname dictionary so "Bill" matches "William" in both directions — a config-flagged, A/B-tunable layer (like the Double-Metaphone↔Beider-Morse choice) over the same tokenize-and-index pass, adding no main-thread work and no per-brother data.

**Stale-tab refresh across a deploy (decision D112).** D73's content-hashed assets make a *fresh* load current, but cannot force an *already-open* background tab to update across a deploy/migration. The server **advertises the current client/contract version** (a header on `/api/*` or a tiny `/api/version`); the SPA compares it to its own build hash and, if stale, shows a **non-blocking "a new version is available — refresh" overlay** and **fails a knowingly-stale write gracefully** rather than sending malformed data. Deliberately lightweight — no full versioning regime — for a 1–2-admin internal tool.

**Cold-start feedback: a threshold-gated loading overlay (decision D119).** Scale-to-zero (D83/D85) makes cold starts common, so the first load — worst for the slow-connection cohort, and at the worst moment (the first click-through after a newsletter send) — can take several seconds. The SPA shows a **"Loading…" spinner overlay only if the initial bulk fetch exceeds a short threshold (~500 ms)** (so a fast load shows nothing), and adds a **"Waking the server — this can take a few seconds…" reassurance line after ~3–4 s** so a cold start reads as expected behaviour rather than a hang. It is dismissed once the dataset has arrived and the Directory can render — the Web-Worker index (above) lights up fuzzy/phonetic search slightly later, while exact/substring matching is available immediately. Purely a frontend affordance over the existing bulk load; no backend change.

### 1.7 Client data layer & roster cache coherence (planned — Phase 7.5)

*Forward design, not yet built.* The current behaviour — every Directory mount re-downloads the whole `no-store` per-role payload (§1.3/§1.6, D95), and the Profile page holds a **separate**, session-lived roster copy for its Big/Little-Brother derivation — remains authoritative until Phase 7.5 ships. This section fixes the target now so the direction is on the record (CODING-PROJECT-PLAN §Phase 7.5; decisions N62/N63).

**The problem: two caches, redundant downloads, and a cross-page inconsistency.** The SPA holds the bulk dataset **twice**. The Directory fetches `GET /api/profiles` into mount-scoped component state and re-fetches **unconditionally** on every remount (the payload is `no-store` and carries no read `ETag`, so there is no `304` path — a full brotli transfer each time, even when nothing changed). The Profile page's relationship derivation reads a **second** copy — a module-singleton roster loaded once per tab. Both hit the same endpoint for the same per-role projection, so a cold Directory→Profile→Directory trip can pull the full payload two-to-three times into two caches. Because the two refresh on different schedules, they can also **disagree**: after an edit the Directory (which re-fetches on remount) shows new data while the Profile page's roster-derived Big/Little-Brother edges (held for the tab) stay stale.

**One shared in-memory store (decision N62).** Phase 7.5 collapses both into a **single module-level in-memory store** subscribed by the Directory and the Profile pages alike (extending today's roster singleton into the canonical store; the Directory reads it instead of running its own fetch). The existing optimistic fold that reflects the editor's own save into the roster is retained and now feeds the one store, so the two views can no longer drift.

**An app-level, versioned conditional read (decision N62).** The store holds an opaque dataset **version token**. On revalidation it issues `GET /api/profiles` with `If-None-Match` (or a `?v=` param); the server returns **`304 Not Modified`** (empty body) when the token matches or **`200`** + the already-compressed per-role buffer + a fresh token when it differs. Server-side this is nearly free: the backend already maintains a dataset **version/ETag in the GCS snapshot, regenerated on every write** (§1.5/D85), and the brother buffer is precomputed (§1.6/D84) — so a match is a token compare and a miss serves a ready buffer. The read `ETag` is **qualified by effective role** (`{datasetVersion}.{role}`) so a "View as" switch cannot be served a mismatched projection as a `304`, and a role switch naturally forces a fresh pull. Revalidation fires on **Directory mount** (replacing today's unconditional refetch) and on tab **`visibilitychange`/`focus`** — which also narrows the cross-user staleness window (a second reader converges on the next mount or tab-focus rather than only on a full reload) for the cost of a tiny conditional request, **without** any WebSocket/SSE push (out of scope — over-built for ≈700 brothers on bursty bi-annual traffic). This is the **data-version analogue of D112's build/contract-version advertisement** (which refreshes stale *code*); the two are complementary and may share a `/api/version`-style surface.

**No on-disk PII — this does *not* resurrect D76 (decision N63).** D95 (superseding D76) removed the read-side conditional GET precisely because D76 served the bulk read `private, no-cache` and let the **browser HTTP cache** store the revalidatable body — persisting real PII to a shared machine's disk with no reliable purge-on-close hook (finding P11). The Phase 7.5 conditional read reintroduces revalidation at a **different layer** and keeps that invariant: (1) the response stays **`Cache-Control: no-store`** — the browser HTTP cache is never involved; (2) the conditional logic lives in **application code** — the fetch layer sends the token and inspects `res.status === 304` itself; and (3) the dataset is retained **only in the JS heap** (the module singleton), wiped on tab close and actively cleared on sign-out. So it recovers the *bandwidth* win D95 gave up while preserving the *on-disk-PII* invariant D95 exists to protect — an app-layer conditional read over a heap-only store, orthogonal to browser caching.

**Preserved invariants and non-goals.** The **per-role projection** (§1.4/D82/D5) is untouched — a `304` only skips re-sending the *same* projection, it never widens it. The **write-side** optimistic-concurrency token (§2.6: per-record `updateTime`/`If-Match`) is a different `ETag` at a different granularity from the bulk read's dataset version and must not be conflated with it. The system stays **pull-based, not real-time**: convergence is bounded by revalidation cadence, not by server push. **Precondition:** OFC-212 (verify the *delivered* `Cache-Control` on the `no-store` PII endpoints is not silently downgraded by the Firebase Hosting header layer) is resolved in Phase 7 first, because the true on-the-wire cache posture shapes this implementation.

## 2. Authentication & Security

### 2.1 The Ghost auth bridge

Book has no password store of its own; it authenticates members against the Ghost membership at pbe400.org through a redirect bridge. Ghost Pro rules out the lower-friction alternatives — a subdirectory reverse proxy or a CORS-enabled cross-origin call both require control over the Ghost web server (Nginx config, CORS headers) that Ghost Pro does not expose. What Ghost Pro *does* allow — a custom theme template plus an editable `routes.yaml` — is exactly what the bridge needs.

The flow:

1. A link on pbe400.org, or Book's backend redirecting an unauthenticated visitor, sends the user to **`pbe400.org/book`** — a custom Handlebars template mapped via `routes.yaml`.
2. **Not logged into Ghost:** the page redirects to Ghost's Portal sign-in with a `return` back to `/book`; Ghost runs its own magic-link email login and returns the user to the bridge page.
3. **Logged in:** before redirecting to Ghost, the SPA obtained a **single-use `state` nonce** from Book (stored server-side, §2.7); the page's script does a *same-origin* `fetch('/members/api/session')` — so Ghost's `ghost-members-ssr` cookies attach — and receives a short-lived (~10-minute) JWT signed by Ghost, with the member's email in `sub` and `aud`/`iss` set to the members API. It redirects to **`book.pbe400.org/auth/callback`** with the JWT in the **URL fragment** (`#token=…`) and the `state` nonce; the fragment — unlike a query string — never reaches server logs, browser history, or the `Referer` header, and every redirect target in the bridge (the Portal `return`, the callback URL) is **hardcoded/allowlisted**, not caller-parameterizable.
4. The `/auth/callback` SPA page reads the fragment, clears it, and POSTs the token (and `state`) to `POST /api/auth/session`. The backend verifies the JWT against Ghost's JWKS (`pbe400.org/members/.well-known/jwks.json`) — signature, `aud`, `iss`, `exp` — with the **algorithm and key id pinned** to Ghost's asymmetric key, **rejecting `alg:none` and every symmetric algorithm** (decision D104); checks and consumes the **single-use `state` nonce**; resolves the email to a profile; sets Book's own cookies; and lands the user in the app.

This is a genuine cryptographic verification, not a trust-the-client handoff: the JWT is signed by Ghost and validated against Ghost's published public keys. The alg/kid pin forecloses the two classic forges (an unsigned `none` token; an `HS256` token HMAC'd with the RSA *public* key); the nonce ties the callback to a Book-initiated flow (anti-forgery), composing with the fragment handoff's anti-leakage (decision D104).

**Identity resolution.** The JWT carries only the email; Book keys its users by Constitution ID, so the callback **normalizes** the `sub` email (lowercase, trim, Unicode-NFC — the same normalization stored emails use, decision D97) and resolves it against the single authoritative instance's **in-memory email→profile index** (primary + `alternateEmail` in one namespace; §5.1). Resolution **fails closed**: an email that maps to more than one profile is denied (`ambiguous_member`) rather than guessed. A first successful match creates the brother's `users` document (which since decision D128 holds only `stars`) using **create-if-absent** semantics (a transaction, or `create()` treating "already exists" as success) so two near-simultaneous first logins cannot race (finding R20). The caller's **role is read from the resolved `Profile`** — where `role` now lives (public read; a missing or unrecognized value normalized to `brother` at the hydration boundary) — and **snapshotted onto the Book session** at sign-in, not from the `users` document (decision D128). An email that matches no profile — a not-yet-loaded new initiate, or one of the historical unidentified Ghost addresses — is **denied with a "contact an administrator" message** rather than auto-provisioned; this keeps unknowns out and funnels them to an admin who can identify and link them. A resolved profile that is **de-brothered** (decision D115) is **denied a session** (`debrothered`), the same hard stop — the de-brothering already deleted the Ghost member (§5.1), so this Book-side check is the belt-and-suspenders half that blocks any lingering token and is the part fully under Book's control regardless of Ghost. Richer member attributes, if ever needed, come from the Ghost Admin API using a key held server-side (§2.7).

### 2.2 Pluggable identity

Although Ghost is the authenticator today, the design keeps it swappable (a project goal from the outset). The backend exposes a single `IdentityProvider` seam whose only contract is `verify(rawToken) → { email }`; `GhostIdentityProvider` implements it with the JWKS check above. Everything downstream — resolving the email to a Book user, reading the role, minting Book's session — is provider-agnostic, so a future switch to, say, Google OIDC is a new provider plus a new bridge entry point, leaving the session and authorization layers untouched. Book keeps its own role table regardless of provider.

**Threat model: Ghost is an accepted single point of total compromise (decision D105).** The token Book consumes is minted by Ghost's `/members/api/session` with `aud`/`iss` of the *Ghost members API*, not Book — a bearer credential not issued *for* Book — and Book derives *all* authorization (including admin) from the resolved email. The finding (S5) is resolved by type:
- **Replay / forced-login — closed.** The D104 nonce binds the handoff to a Book-initiated flow, the D104 alg-pin forecloses forged tokens, and the 4-hour session cap (§2.3) bounds a stolen token's useful life. The residual — an attacker who steals a *live* member token via XSS or a malicious element on the shared `pbe400.org` origin — is closed by **hardening `pbe400.org`**, named as an **explicit dependency** in this threat model. A Book-specific-audience token would be ideal, but Ghost Pro exposes no way to mint one (D20), so handoff-binding is the realistic mechanism.
- **Blast radius — accepted, documented.** That anyone who obtains a member's — or an admin's — Ghost token thereby obtains the corresponding Book access is an architectural given of D54 (Ghost *is* the authentication; there is nothing below it to appeal to). Recorded **as-designed** with the written note: **"Book's authorization trust equals Ghost Pro's account security."** This note also absorbs D101's socially-engineered-restore residual.
- **Step-up — rejected.** **No step-up authentication is added.** Session 4 shrank the online destructive surface to **role grants and single-profile delete** (the email-collision takeover path was closed by D97/D99; restore went offline and Directory bulk-delete was dropped, D100/D101), both already gated by a typed-acknowledgment confirmation and audited with before/after (D106). An *in-Book* step-up authenticates nothing new, since every factor Book holds derives from the same Ghost identity; only an *out-of-band* channel would be meaningful, and that is unjustified friction for a 1–2-admin volunteer directory whose remaining destructive actions are already confirmed, reversible, and audited. Detect-and-reverse (the D106 audit + D101 forensic log) is the right-sized control at this scale.

### 2.3 Book session & revocation

After verification the backend issues **Book's own** session cookie on `book.pbe400.org`, independent of Ghost's. Book's session cookie is an **opaque, high-entropy session identifier** backed by a server-side session record **persisted in Firestore** (a `sessions` document with a native TTL on `expiresAt`, **not** held in instance memory) — **not** a self-contained or self-signed JWT — so the 4-hour cap and revocation are enforced server-side rather than trusted from a token the client holds (the Ghost JWT of §2.1 is consumed once at the handoff and never used as Book's session). It is **host-only** (no `Domain` attribute, so no sibling `pbe400.org` subdomain — Ghost included — can read it, decision D107), a *session* cookie (cleared on browser close), **and** carries a **4-hour server-side lifetime cap**. The cap is the security-relevant half: a session cookie alone is a soft guarantee (browsers with "continue where you left off" restore session cookies, and a browser left open never re-authenticates), so the server-side expiry is what bounds the revocation window — the goal being that a brother removed from Ghost loses Book access promptly. When the session lapses, the next request bounces back through the bridge: silently if the user is still a Ghost member, denied if they have been removed. Four hours balances security against not nagging users, since a re-auth is a silent redirect rather than a fresh magic-link round-trip.

**Active revocation on withdrawn trust (decision N53).** The 4-hour cap bounds the *worst case*, but three staff actions withdraw a brother's trust and must not wait for it: **de-brothering**, **deletion**, and a **role change** (a session snapshots the caller's real role, so a demoted admin who kept a live session could otherwise re-promote themselves via the role endpoint before the cap). Each of those endpoints therefore actively revokes the target's live sessions the moment it commits — `destroyAllForProfile(profileId)` deletes every matching `sessions` document (queried on the nested `session.identity.profileId`, which Firestore single-field-indexes automatically) and prunes the in-memory read-through cache in lock-step (the single instance of D83 owns both), recording the count as `sessionsRevoked` on the action's audit entry (§6.1). Reinstating (de-brother reverse) and a no-op role reassignment revoke nothing; mark-deceased does not revoke (revocation tracks withdrawn trust, not death). As a structural backstop, the session **gate** additionally rejects and destroys any session whose own record is *present-and-de-brothered* — a free in-memory lookup on every request, keyed on the positive de-brothered flag rather than on record *absence* (an absent record is ambiguous — a hard delete, a not-yet-hydrated initiate, or a record skipped as malformed — so 401'ing on absence would risk locking out a live brother; the delete path's active revocation covers the hard-delete case instead). The gate **also** catches a **role downgrade** (decision D130/OFC-239): now that `role` lives on the profile cache (D128), a free per-request rank comparison against the session's snapshotted real role rejects-and-destroys a session whose caller has since been demoted (`isRoleDowngrade` — only a downgrade; an upgrade merely under-privileges the stale session, harmlessly). This is the defense-in-depth backstop to the role-change route's active revocation above — the same relationship the de-brothered check has to the de-brother route — closing the window in which a demoted admin's stale session would otherwise keep admin powers if that best-effort revocation failed.

**Persisting the session record in Firestore is load-bearing under scale-to-zero (decision D125).** The single instance (D83) shuts down whenever idle, and a *browsing* brother makes very few backend calls after the initial load (search/filter/sort are client-side; image reads come in brief bursts that leave long quiet stretches — decision D126), so the instance routinely scales to zero **while a brother is still reading**. An in-memory session table would make every cold start invalidate every session — bouncing the next action back through the bridge, forcing a full SPA reload and re-download of the whole dataset over a slow link, and turning the rare-by-design mid-edit re-auth (§2.6/D109) into a constant event. Firestore-backing decouples session validity from instance lifecycle while keeping exactly what §2.3 wanted from a server-side record — the cap and revocation stay server-enforced (the cap rides in the stored record) — and a read-through in-memory cache keeps warm requests free, so only a cold start pays a Firestore read. The single-use `state` nonce (§2.1/§2.7) is persisted the same way, so a cold start landing between sign-in initiation and the callback cannot fail the nonce check. Genuine expiry still happens (the 4-hour cap; removal from Ghost) and is surfaced with a clear "your session has expired — please sign in again" message; re-auth/expiry events are logged (§6.1) so the true frequency is measurable.

**Non-destructive recovery on an absolute-cap lapse (decision D109).** Because the cap is **absolute**, not a sliding idle timeout, a genuinely long edit *will* hit the wall mid-Save — a case silent renewal cannot prevent, so a non-destructive recovery is mandatory. On a `401` from a state-changing request the SPA does **not** follow a redirect mid-XHR and discard the work: it preserves the in-progress form, re-authenticates in a **child window** (the editor tab never navigates, so the in-memory form survives with **zero PII written to disk** — a hidden-iframe variant is foreclosed by D107's `frame-ancestors`, so it is a child *window*), and on success **resumes the same Save carrying its original `If-Match`** (so a real concurrency conflict still reconciles via §2.6). A `sessionStorage` draft stash is the **popup-blocked fallback only** (tab-scoped, cleared on tab close — the closest analog to the no-disk posture). This extends §2.6's preserve-edits-on-`412` to the previously-unspecified `401` path; the matching image-`403` case is handled in §2.5.

### 2.4 Roles, capabilities, and projection

Book has three roles — **brother**, **manager**, **admin** — held on the `Profile` (since decision D128; formerly the `users` collection), independent of Ghost. `role` is a **public** projected field with a **protected** write (set only by the change-role action), normalized to `brother` at the hydration boundary; the session snapshots the caller's role from the resolved profile at sign-in (§2.1). The full capability matrix is in PRD §4; the field-visibility model it enforces is in DATABASE-SCHEMA §9. In summary, the manager/admin line is drawn on *actions*, not raw visibility: managers maintain the directory (edit any visible field, set deceased status, export, verify) but cannot perform the destructive or consent-altering operations — delete, add, bulk import, role changes, Ghost sync, backup, or changing a brother's privacy/consent settings — which are admin-only. On *reading*, a brother and a manager see the same contact values (an off-toggle hides a value from managers too); managers additionally see the restricted flags, preferences, and dates (read-only); only admins see the contact values hidden behind an off-toggle. Projection is enforced server-side on every response (decision D5), so the frontend never receives data the caller may not see — closing the DevTools/network-inspection leak a client-side filter would expose. The bulk read is **split** (decision D82, §1.4): `GET /api/profiles` returns a uniform *per-role* projection of all records (the brother buffer cached, manager/admin computed fresh), and the caller's own full record comes from `GET /api/me` — so the bulk payload is identical for all callers of a role and cannot leak one caller's owner-level fields to another.

### 2.5 Media authentication

Headshots and thumbnails are served by the **Cloud Run backend** from a **private GCS bucket** (§1; DATABASE-SCHEMA §7), under the Book origin (`/img/*`, via the Firebase Hosting → Cloud Run rewrite) and gated by the **ordinary Book session cookie** — the same cookie that authenticates `/api/*`. The backend returns versioned URLs with `Cache-Control: private, max-age=31536000, immutable`, so each image is browser-cached permanently and re-fetched only when its `headshotVersion` changes. The grant is coarse — any authenticated brother may view any photo, because photos carry no per-brother privacy (decision D23) — but because the app mediates every read, image visibility also **tracks record visibility**: an `unlisted` or de-brothered brother's image is not served to a brother who guessed the URL. The bucket is **private** (read only by the backend's service account; no public objects, and object ACLs are never the access surface), and the image origin is `'self'` in the CSP `img-src` (§6.4). This is **decision D126**, which replaced D23's **Cloud CDN signed-cookie** scheme — and with it the **external HTTPS load balancer** the CDN required — after that load balancer's fixed ~$18–25/month proved disproportionate at Book's scale (the corpus is ≈15–20 MB; the slow-link win is immutable *browser* caching, which survives the change; the per-image cost through Cloud Run rounds to ~$0 within the free tier). Signed URLs remain rejected (D23): they force a backend round-trip per image or fight the immutable-cacheable-URL model.

**Image `401`/`403` on cap lapse — re-auth and retry (decision D109 / finding R18).** Because image reads carry the session cookie (§2.3), a long-open tab can hit a `401`/`403` on an image read at the absolute 4-hour wall — physically the *same* expiry event as the mid-edit `401`. The SPA triggers the **same** child-window re-auth and **retries the image load** (the new session cookie rides the retry) rather than rendering a broken image; one cap-lapse, one re-auth, both the pending Save and the failed image fetches resume. (There is nothing to "reissue ahead of expiry" — the session is what gates the image, and it is bounded by the absolute wall.)

**Superseded-version purge (decision D94).** On a headshot replace/remove/delete, the brother's prior GCS object versions are **purged after a 3-month lifecycle window** (aligned with the backup/audit retention, §6.1/§6.3); because reads go through the app and each version has its own immutable URL, a superseded object stops being served the moment the `headshotVersion` pointer advances — so a removed photo actually goes away, with a brief recovery margin.

### 2.6 Optimistic concurrency & conflict handling

Profile writes use optimistic concurrency keyed on Firestore's server-authoritative `updateTime` — a stronger token than a wall-clock `lastModified`, and one Firestore can enforce as a native write precondition, making the check a single atomic conditional write with no read-modify-write race. The token is surfaced to clients as an `ETag` and required back as `If-Match`; a stale write is rejected with `412` (API-SPEC §1.4), at document-level granularity.

On a `412` the client neither silently discards the user's work nor blindly overwrites: it repulls the current record, **keeps the user's unsaved edits in the form**, and shows a read-only "these fields changed underneath you" notice so the user can reconcile by hand and re-save (now carrying the fresh token). This middle ground — between throwing the user's work away and building a full auto-merge engine — fits the rarity of real conflicts at this scale while never losing data and never clobbering silently.

The same "never lose the user's work" instinct extends to the **`401`** (expired-session) path: a mid-edit session lapse preserves the in-progress form and re-authenticates in a child window, then resumes the Save carrying its **original `If-Match`** — so an expiry that coincides with a real concurrency conflict still reconciles through the `412` flow above (decision D109, detailed in §2.3).

### 2.7 Hardening

- **Cookies.** The session cookie is `HttpOnly`, `Secure`, `SameSite=Strict`, and **host-only** (no `Domain` attribute — decision D107, §2.3); it is the single credential, authenticating both `/api/*` and the same-origin `/img/*` image reads (decision D126 — there is no separate CDN cookie). `SameSite=Strict` is viable because the bridge passes its token in the URL fragment, not via the cookie, so the cross-site landing does not depend on cookie attachment.
- **CSRF.** Cookie-based auth plus `SameSite=Strict` and same-origin JSON requests cover the state-changing endpoints; a cross-site form cannot attach the session cookie.
- **CORS.** `/api/*` is **deny-by-default** (decision D107): no credentialed `Access-Control-Allow-Origin` reflection — in particular `pbe400.org` is not granted cross-origin credentialed access to Book's API.
- **Callback binding.** A **single-use `state` nonce** is generated and stored in **Firestore** (a short-TTL `authNonces` document — decision D125, so a cold start mid-handoff cannot drop it) at sign-in flow initiation and verified (then consumed) at `POST /api/auth/session`; every redirect target in the bridge is hardcoded/allowlisted, not caller-parameterizable (decision D104, §2.1).
- **Rate limits & abuse controls (decision D86).** Rate limits / concurrency caps are applied on the **authentication endpoint** (capping the unauthenticated crypto work in `POST /api/auth/session`), on **mutating endpoints**, and on the **bulk read**; unknown-`kid` JWKS refetches are single-flighted and capped (shared with the JWKS item below), and compression rebuilds are debounced (§1.6/D84). These matter more now that a single instance (D83) serves everyone; detection (the §6.1 sign-in-denial alert) is not prevention.
- **Secrets.** The Ghost Admin API key (used for member create/update/delete sync, §5.1) lives in Secret Manager, never in the SPA or the theme template. (There is no Cloud CDN signing key — images are app-served, decision D126.)
- **JWKS (persisted/seeded — decision D87).** Book caches Ghost's JWKS and refreshes it on a schedule and on an unknown `kid` (refetches **single-flighted and capped**), so key rotation does not break verification. Because scale-to-zero (D83) makes cold starts common, the JWKS is additionally **persisted/seeded across cold starts** (or keys pinned): a freshly cold-started instance can verify against last-known-good keys for a short grace window even if Ghost/JWKS is momentarily unreachable — otherwise every cold start during a brief Ghost hiccup would lock out *all* sign-ins (finding R6). This is the **Ghost** members-API JWKS on the sign-in path; the **Google** JWKS for the Linter roster (§5.2, D78) gets the same single-flight discipline.
- **Security headers.** A strict **CSP**, `nosniff`, a framing policy, and `Referrer-Policy` are served alongside HSTS — see §6.4 (decision D107).
- **Transport.** HTTPS throughout, with Google-managed certificates (§6.4).

## 5. Integrations

Book is not a standalone application. It is one half of a **composite system with Ghost**: Ghost (pbe400.org) provides Book's authentication (§2), and Book provides Ghost's member management — and a deferred third tie, Ghost content search (PRD §3.2), would deepen the coupling further. The dependency is asymmetric: **Book hard-depends on Ghost** (without Ghost there is no sign-in and Book is unusable), while **Ghost soft-depends on Book** (member data is maintained through Book by design, but Ghost's own admin panel remains a break-glass path if Book is ever unavailable). This section specifies the four integration surfaces: the Ghost membership sync, the PBE News Linter's data feed, the MITAA exchange, and the deferred Mailman lists.

### 5.1 Ghost membership sync

**Single master.** Book is the authoritative system of record for member data; Ghost is downstream. Member-data editing in Ghost is **disabled and redirected to Book**: Ghost's account UI (the avatar → Account → email / email-preferences modals) is repointed at Book's profile editor, either by overriding the theme's account-link target (`data-portal="account"` / `#/portal/account`) or, failing that, via Ghost Pro's JavaScript Code Injection. Concentrating every edit in one place removes the two-master conflict at its root.

**Fields pushed to Ghost — synchronously, diff-based, Ghost-first-gated (decisions D96/N65).** Book pushes a small set to Ghost **synchronously on save** (the save calls the Ghost Admin API and commits Firestore **only on its success**, before returning — Ghost-first-gated, DECISIONS N65, which amended D96's original write-Firestore-then-push ordering so the update path matches the create/delete/de-brother paths), reversing the Session-0 async-outbox direction: an outbox fights the scale-to-zero choice (D83 — nothing drains a queue while the single instance sleeps), while synchronous push surfaces failures loudly at the moment they occur and a human is almost always present to retry. The timing matters because email is the authentication join key (below), so Ghost must learn an email change before the brother's next sign-in. Pushes are **diff-based** — only the pushed fields that actually changed in a given save are sent — so an unrelated edit never re-asserts a consent value; this specifically prevents silently **re-subscribing a brother who used Ghost's own unsubscribe link** (which sets his subscription off in Ghost while Book still reads `true`; a whole-set push would clobber that opt-out). The pushed set:

- **email** — the brother's primary email (the auth join key).
- **name** — the constructed **Canonical Name** (`James Smyth '84`), matching existing practice so it appears in article bylines; the rare `(#5247)` disambiguation suffix is omitted from the pushed value.
- **newsletter subscription** ← `allowNewsletterEmail` (Ghost v5 models this as the `newsletters[]` relation — subscribe = `[{ id }]`, unsubscribe = `[]`; the newsletter id is pinned via `GHOST_NEWSLETTER_ID`).

No other Book field is pushed; in particular `adminNote` is not (were Book ever retired, the notes would be migrated then). **Two things a brother has in Ghost are not pushable — Ghost platform limitations, not Book choices (DECISIONS N66).** Photos: the member `avatar_image` is read-only (Gravatar-derived or `null`; no upload path), and staff byline photos sit behind an API surface outside the documented stable Admin API (the byline/staff-photo automation is backlogged as Linear OFC-213). **The comment-reply-notification preference** (`enable_comment_notifications`): verified against ghost-staging (2026-07-08) as **not settable via the documented Admin API** — the serializer never returns it, Ghost silently drops it on write, and a live portal cross-check confirmed no effect. Because Book could not make that setting take effect, the former `allowCommentReplyEmail` field was **removed from Book entirely** rather than shipped as a dead toggle (decision N68); the upstream Ghost issue is tracked as **OFC-216** and a re-add as **OFC-215**. The pushed set is therefore these three fields.

**Member lifecycle (Book → Ghost).**

- **Create** (an admin adds a brother) — **Ghost-first**: a `POST` to the Ghost Admin API creates the member and **captures the returned Ghost Admin-API member id** into `ghostMemberId`, *then* Book writes Firestore (Ghost's automatic signup email **suppressed** — the brother simply becomes able to magic-link in). Ghost-first ordering means a failure cannot strand a brother who can never sign in; the rare reverse (Ghost member created, then the Firestore write fails) leaves a *cleanable* Ghost orphan, caught by the reconciliation audit (decision D98). (`ghostMemberUuid` is no longer captured — decision D81.)
- **Update** (email / a Canonical-Name input — `firstName`/`lastName`/`classYear` / either email-preference changes): the save `PUT`s the member addressed by `ghostMemberId` **first** (Ghost addresses members by id, not by email — addressing by the stored id is what makes an *email change* a safe, unambiguous operation, since the member is never looked up by the very field being changed), and **only on Ghost's acceptance does Book write Firestore** (Ghost-first-gated, DECISIONS N65, amending D96's original ordering). Book therefore never holds a pushed-field value Ghost has not accepted — no divergence window opens and no tolerance mechanism is needed. On a clear Ghost failure the **whole save fails loudly** (`502 ghost_update_failed`, Book untouched) and the user retries later; a save touching none of the pushed fields makes **no** Ghost call and is unaffected by a Ghost outage. A profile with no `ghostMemberId` skips the push (there is no member to update; the reconciliation audit reports it). The `If-Match` token is preflighted before the Ghost call and same-record writes are serialized in-process (sound at `max-instances=1`), so a concurrent edit cannot commit between the Ghost `PUT` and the Firestore write; residual cross-system races fall to the reconciliation audit as `fieldDrift`. The rare reverse failure (Ghost accepted, then the Firestore write fails) leaves Ghost briefly ahead of Book — the failure is loud, the editor's session is live, and an idempotent retry converges (re-`PUT` by id is a no-op; a PATCH treats a conflicting `updateTime` that is its own just-applied write as success — finding R13); an abandoned case surfaces in the audit for an admin re-push.
- **Delete** (an admin deletes a brother) — **Ghost-first-abort-clean** (decision D98): the Ghost member is deleted **first**; if that fails the delete aborts cleanly (Book untouched, admin retries). Then the idempotent Book-side steps run (GCS objects, `users` doc, profile doc), and inbound references are scrubbed (`bigBrotherId == id`, `stars` containing `id`) so no dangling reference is left. A removed brother loses sign-in promptly.
- **Deceased** (a brother is marked deceased): a deterministic consequence forces `allowNewsletterEmail` to `false` (stamping `newsletterConsentChangedAt`) and pushes the unsubscribe to Ghost **Ghost-first-gated** (N65), so a deceased brother receives no PBE News email; the Ghost account is otherwise left intact. The prior consent/verification state is snapshotted so an un-marking restores it (decision D80; DATABASE-SCHEMA §8). *(The former companion flag `allowCommentReplyEmail` was removed — N68.)*
- **De-brother** (an admin de-brothers a member — decision D115): unlike deceased, which leaves the Ghost account intact, de-brothering **deletes the Ghost member via the Ghost-first delete path** (decision D98 — abort-clean on failure), blocking Ghost login and stopping all mail in one move, while the Book profile is **retained** but projected away for brothers (§1.4) and sign-in denied (§2.1). The prior consent/verification state is snapshotted; **un-de-brothering re-creates the Ghost member** via the Ghost-first create path and restores it — symmetric with un-decease (D80). The reconciliation audit (below) knows a de-brothered profile is *expected* to have no Ghost member.

**The email join key — uniqueness and the change-lockout.** Book resolves the Ghost-issued login JWT's email to a profile (§2.1), matching on the **primary email only**; `alternateEmail` is never used for matching and is never pushed to Ghost — a contact-of-last-resort fallback we deliberately do not encourage anyone to rely on. Email is enforced **unique** by the single authoritative instance's **in-memory email→profile index** (decision D97), derived from the profiles themselves and rebuilt on cold-start hydration (§1.5/D85): because Node runs single-threaded, the synchronous check-and-claim cannot interleave, so no Firestore reservation collection is needed (D83's single writer is what makes this sufficient — there is no second writer to race, except an offline restore, which re-validates uniqueness structurally on a down system, §6.3/D101). Emails are normalized destructively on save (lowercase/trim/NFC); primary and `alternateEmail` share **one namespace**; resolution **fails closed** on any duplicate. The email-change lockout is closed **structurally** by the Ghost-first-gated update (DECISIONS N65, amending D96): Book never commits an email Ghost has not accepted, so sign-in — which matches on Book's stored primary email — and Ghost's magic-link delivery can never disagree about the current address. The transient `pendingPriorEmail` tolerance D96 originally specified (a backend-only prior-email alias in the index, set on an uncertain push outcome and cleared by reconciliation) is **dropped as no longer needed** — there is no disagreement window left to tolerate. The one residual (Ghost accepted the new email but the Firestore write failed) is loud — the save reports failure while the editor's session is still live — and converges on retry or via the reconciliation audit's re-push.

**The reconciliation audit (the "Book / Ghost alignment audit"; read-only into Book — decision N69, amending D103).** The admin triggers it from the Admin page (`GET /api/admin/ghost-audit`, PRD §5.8.3), and the same audit **also runs on a schedule** inside the consolidated health-check job (decision D99, below). It is a **reconciliation audit, not a writer into Book at all** — decision N69 removed even D103's scoped newsletter write-back, so the audit **overwrites nothing** in Book, in any category; this restores D55's read-only-into-Book invariant in full and permanently removes the failed-push-undo hazard. Keyed on `ghostMemberId`, it produces a discrepancy report (the JSON shape is in API-SPEC §7) across these categories, **every one report-only**: **`unmatchedGhostMember`** — a Ghost member referenced by no Book profile (a self-signup, or a historical unidentified address) → identify/link; **`fieldDrift`** — a pushed field (email / Canonical Name) differs, typically a push that failed while Ghost was unreachable → admin re-saves to re-push; **`missingGhostMember`** — a Book profile with an email whose `ghostMemberId` no longer resolves, or that has none → re-create (a **de-brothered** profile is *expected* to have no Ghost member and is **excluded** — decision D115; a no-email/unidentified profile is likewise not expected to have one and is not flagged); **`bookInternalOrphan`** — a dangling `bigBrotherId` or a `users` doc with no live profile (decision D98, folding orphan detection into this audit rather than a dedicated sweep); and **`newsletterDrift`** — reported like the rest, enriched with the change timestamps so a human can resolve it (below). The report is **download-only** — the SPA fetches the JSON and saves it as Markdown (N69); nothing is rendered in Book's UI.

**The newsletter flag is reported, not reconciled (decision N69, amending D103).** `allowNewsletterEmail` is the one field with a legitimate two-way source: Book can change it (a profile edit, or the mark-deceased force) and **Ghost can change it** (the brother clicks "unsubscribe" in a PBE News email). D103 originally had the audit *resolve* this by most-recent-change-wins — writing Ghost→Book on a later Ghost-side unsubscribe and re-pushing Book→Ghost on a later Book change. Decision N69 **removes that write-back**: the Book→Ghost arm was already made moot by the 5b-1 Ghost-first-gated write (N65, which stops Book ever getting silently ahead of Ghost), and dropping the Ghost→Book arm restores the read-only-into-Book invariant everywhere. So a `newsletterDrift` row is **reported like any other**, carrying both values plus **`bookChangedAt`** (Book's `newsletterConsentChangedAt`) and **`ghostChangedAt`** (the latest Ghost `newsletter_event` timestamp) so the reviewer — a human, or the future "AI system administrator" (Linear OFC-214) — can see which side changed later and flip the correct toggle by hand with the controls that already exist. The Ghost timestamp is thus **advisory context, not a control input**: it is read and shown, but a run with no event available simply omits `ghostChangedAt` rather than failing. Ghost's member-activity API was **verified** (OFC-230) to expose the subscribe/unsubscribe timestamp programmatically (`GET /members/events/?filter=type:newsletter_event` → `{ member_id, subscribed, created_at }`), and that endpoint joins the pinned/canaried Ghost surfaces (below).

**Observability and contract monitoring (decision D99).** Every log line of a save, its Ghost call, and the resulting audit entry carry the Cloud-Run-injected **trace/correlation id** (`X-Cloud-Trace-Context`) — simplified by the synchronous model, since a save is now one request rather than an async span. Because synchronous push surfaces individual failures at the source, there is **no standalone push-failure-rate alert**; periodic drift-detection and contract-monitoring are instead consolidated into **one scheduled sysadmin health-check job** (Cloud Scheduler → an admin endpoint — one wake-up under scale-to-zero, daily by default): it runs the reconciliation audit (above), a **login canary** (the bridge's dominant failure is *sign-in failure → Book unusable*, most valuable to catch during quiet stretches before a newsletter send), and the `IdentityProvider` **contract test** (D72) on a schedule, alerting on findings and on a failed run. The depended-on Ghost surfaces — the member-session JWT/JWKS, the Portal redirect, the Admin-API shapes, and the member-activity unsubscribe timestamp (D103) — are documented and pinned (finding R14). It was **verified that Ghost Portal pages cannot be disabled server-side**, so a member who types the obscure Portal profile URL can still edit name/email/newsletter in Ghost; this is an accepted low residual — the scheduled audit catches the drift, and a Ghost-side email change is **self-surfacing** (the member is locked out of Book under the new address, prompting an admin contact), so no dedicated account-redirect canary is built.

**Email-bounce reporting is a separate download (decision D120, as amended by N69).** Ghost logs when a PBE News email to a brother bounces. This is its **own** admin report — `GET /api/admin/bounce-report`, its own Admin-page card — not something riding each audit run (N69 split it out: it is a different job, email-address maintenance rather than drift). It reuses the proven `ghost-member-export/export-bounces.js` join: fetch Ghost's `email_failed_event` member-events, join them to members by `member_id`→email, and aggregate per brother to `email / bounce_count / last_bounce_at / last_bounce_newsletter` (working around Ghost's `/admin/emails/` 403 via the posts endpoint — best-effort, so an unavailable title degrades to the raw email id — and skipping events for hard-deleted members). It is **not surfaced anywhere in Book's UI** (no per-user "BOUNCING" banner, no directory/profile indicator, no panel — D120): the SPA fetches the JSON and **downloads a CSV** the admin opens in a spreadsheet (N69 chose a CSV download over a Cloud Logging line, which is *less* accessible to the one admin who needs it). A live in-app bounce warning was considered and **dropped as disproportionate** — PBE News is bi-annual and scale-to-zero offers no live-polling path, so a periodic report is the right granularity. The bounce-report generator is inventoried in `PRE-LAUNCH-TOOLS.md` §B.

**Why `ghostMemberId`, and not Book's ID in Ghost.** Storing Ghost's member ID in Book gives the audit a stable, email-independent join in both directions, which is sufficient for reconciliation. The symmetric idea — embedding Book's Constitution ID *in* Ghost (e.g. a second canonical form) — was considered and rejected as redundant; the only scenario that would revive it (deep-linking PBE News bylines to Book profiles, PRD §3.2) is itself deferred.

**Initial migration is external, one-time tooling.** Two facts make the pre-launch data load a job for standalone utilities rather than Book code (decision D57): it runs exactly once, and it is a messy manual merge of Constitution-roster, MITAA, Ghost, family-tree, and yearbook data. So:

- A **one-time pull-and-seed utility** reads from Ghost the member id (→ `ghostMemberId`), the notes (→ `adminNote`), and — critically — the **current newsletter subscription state**, seeding `allowNewsletterEmail` *from Ghost* so existing opt-outs are honored rather than silently overwritten by the `true` default (and stamping an initial `newsletterConsentChangedAt`). (The member `uuid` is no longer captured — decision D81; the comment-notification preference is no longer a Book field — N68.)
- The **initial bulk-loader** writes the merged dataset, including `ghostMemberId`, which the ordinary admin bulk-CSV import does not handle.

This is the only time Book data is written *from* Ghost. In steady state Book reads Ghost only for authentication (§2.1) and for the read-only audit above; it never imports member data from Ghost again. The growing inventory of these tools lives in `PRE-LAUNCH-TOOLS.md`.

### 5.2 PBE News Linter

The Linter is a first-party, non-browser consumer that canonicalizes brother names in draft articles (e.g. `Jim Smyth` → `James Smyth '84`). It needs Book's name-and-year roster, and nothing else.

- **Data — a read-only roster.** `GET /api/roster` returns the least-sensitive slice of Book: `id`, the name fields (`firstName`, `middleName`, `lastName`, `fullLegalName`, `mugName`), `classYear`, `isDeceased`, and the **server-constructed `canonicalName`**. No contact information, no restricted fields.
- **Book serves the canonical name; the Linter does not recompute it.** Decision D15 envisioned one *shared canonical-name function* used by both Book and the Linter — which holds only if they share a language. The Linter is likely Python (local-LLM tooling such as Gemma) while Book is TypeScript, so to preserve a single source of truth the **roster endpoint emits the finished `canonicalName`** (including ambiguity disambiguation, computed once over the whole in-memory dataset). The Linter still receives the raw name parts so it can match messy article text ("Vic" → Victor), but the authoritative canonical string comes from Book. This refines D15 from "share the function" to "serve the output."
- **Access pattern.** The Linter pulls the **whole roster once per run** and matches locally — the same bulk-download-and-work-locally model as Book's own frontend (D4), a natural fit for a tool processing 20–30 articles per issue. It is strictly **read-only**: it flags possible-but-unmatched names in the editor, never writing back to Book.
- **Authentication — a Google service-account identity token, verified in-code (decisions D58, D78).** The roster endpoint is authorized not by the Ghost session cookie but by a **GCP service-account OIDC identity token** (satisfying D27's "scoped service path, not the public API and not the Ghost bridge"). The Linter runs as a dedicated `linter` service account and presents a short-lived, Google-signed token as `Authorization: Bearer …`, **verified in-code against Google's JWKS requiring issuer = Google, audience = Book, *and* subject = the exact `linter` service account** — the same shape as the Ghost JWKS check (§2.1), with Google as a second trusted issuer. The earlier "Cloud Run front-door IAM" option is **dropped (decision D78)**: `run.invoker` is enforced per-*service*, and this service also hosts the **unauthenticated** sign-in endpoint, so requiring IAM would lock out sign-in and `/api/profiles`; the **subject pin is essential**, since issuer+audience alone would accept any Google-issued token for that audience. This avoids any static API key that could leak into source control: on GCP the runtime's metadata server mints tokens keylessly, and for a locally-run Linter the keyless path is impersonation through the operator's own `gcloud` login (a downloaded key file is the avoidable fallback). The scheme is **runtime-agnostic** — desktop or GCP VM, local files or Ghost — so Book's obligation is fixed regardless of how the Linter's own design settles.
- **Contract versioning (decision D112).** The roster carries an explicit **`contractVersion`** (a field and an `X-Contract-Version` header); the Linter **pins** a version, and `contractVersion` is **bumped on any breaking shape change** with a deprecation note recorded in API-SPEC §8 — closing the silent-break risk of an independently-deployed, cross-language consumer.

### 5.3 MITAA exchange

The MIT Alumni Association exchange stays **manual, occasional, and low-trust**; automated exchange is deferred (decision D11). MITAA data is a *suggestion source*, never authoritative — Book exists precisely because MITAA's data was inadequate — and Book remains the system of record. Both directions reuse Book's existing tools; no MITAA-specific feature is built for MVP.

- **Import (MITAA → Book).** Its highest-value job in practice is **catching deaths**: for older brothers whose families and surviving peers no longer notify PBE, a status change in MITAA's export is often how PBE learns a brother has died. Because MITAA rows carry no Constitution ID and are the less-trustworthy source, **matching and judgment happen outside Book**: external tooling plus a human (drawing on the same fuzzy-matching work as the mystery-email project) match MITAA rows to Constitution IDs and curate a Book-format CSV of only the vetted changes. That CSV then passes through the ordinary admin **dry-run bulk import** (PRD §5.8.2) as the final confirmation. A MITAA-surfaced death becomes an admin running the mark-deceased flow (PRD §5.7.7), which fires the deceased side-effects above.
- **Export (Book → MITAA).** Gated by `allowShareWithMITAA`, redefined as a **master contact-sharing switch**: when `true`, MITAA receives the brother's full own-contact set (email, phone, postal address) regardless of the brother-facing share-toggles, which target a different audience (other brothers, not MIT); when `false`, no contact information is shared. **Emergency-contact data is never shared with MITAA** (third-party data). **Identity (name, class year) and public deceased status / date of death / obituary link always flow regardless of the flag** — they are MIT's own data and the join key, and the death information is the reciprocal of the import's death-catching value (PBE sometimes knows of a death before MITAA does). Because the export must include contact values that may sit behind a brother's *off* toggle, a complete export requires the admin (see-through) projection, so **MITAA export is an administrator operation**. It is a **full snapshot of consenting brothers** each time; a changes-only delta is deferred (it would need a "last-sent" baseline Book does not keep). *(Linear: OFC-169)* The exact MITAA column format is kept flexible — the mapping lives in an external script — since MITAA's import requirements are not yet known and their export layout may not be fixed.

### 5.4 Mailman (deferred)

PBE's two mailing lists — **pbe-brothers-official** (heavily moderated) and **pbe-connect** (unmoderated; any brother may broadcast to all subscribers) — run on a **MIT-*hosted*** GNU **Mailman 2.1.34** instance: MIT provides Mailman as a service, but PBE manages the lists' membership and content. Integration stays **out of MVP** (decisions D11, D60). The barrier is not control of the lists (PBE has that) but control of the *software*: Mailman 2 has no usable API, and the instance cannot be modified. These lists are entirely separate from PBE News, which is sent through Ghost and governed by `allowNewsletterEmail` — "the newsletter" and "the Mailman lists" are different plumbing.

If integration is ever taken up, the likely mechanism is **lightweight HTTP form POSTs against Mailman 2's admin web forms with HTML-response parsing** (not heavyweight browser automation), with the usual caveats — brittleness to HTML changes, and admin-credential handling; whether it lives in Book or in a separate tool is left open. Two **replacement** directions are recorded as deferred future work, without choosing between them (PRD §3.2): (a) migrate **pbe-brothers-official into Ghost** as a second newsletter — viable because it is a moderated broadcast, which fits Ghost's model, though it would not cover pbe-connect's any-brother-to-all-subscribers behavior; or (b) **self-host a PBE-owned Mailman 3** and integrate Book to it over Mailman 3's REST API, replacing both lists. Direction (a) would make PBE News one of *several* Ghost newsletters, which is why **multiple-newsletter support** (per-newsletter subscription toggles) is itself a recorded deferred item — the MVP's single `allowNewsletterEmail` boolean is a deliberate simplification of Ghost's native multi-newsletter subscription model.

## 6. Operations

Book's operational surface is small by design — a static SPA, a thin stateless API, a document store, and an object store — but five concerns need explicit treatment: how the system is observed (logging, analytics), how its data is protected (backups), how it is secured in transit (TLS), how its data is maintained (the majors vocabulary, schema migration, and the deferred bulk import), and how the whole thing is tested. Deployment and the phased build sequence are the subject of the separate `CODING-PROJECT-PLAN.md`; this section specifies the standing operational design those plans assume.

### 6.1 Logging & audit

Cloud Run captures a container's `stdout`/`stderr` into Cloud Logging automatically, so Book logs as **structured JSON to the standard streams** — no logging SDK, no extra dependency — following the project convention of progress to `stdout` and errors to `stderr` (errors emitted at `ERROR` severity). Three conceptual streams:

- **HTTP access logs** — request method, path, status, and latency are already emitted by the Cloud Run layer (and Firebase Hosting's request logs). Book does not reimplement them.
- **Diagnostic logs** — severity-tagged application logs (`DEBUG`…`ERROR`) for troubleshooting: cache hydration and snapshot-listener events, Ghost-push successes and failures, slow operations, unexpected states.
- **An audit stream** — a durable record of mutations and security-relevant events, each entry labeled `audit` and carrying actor ID, target ID, action, and outcome: sign-in success and denial (session-expiry re-authentications included, so the real frequency of cap-driven re-logins is measurable, decision D125), JWKS-verification failures, profile create/update/delete, role changes (with before/after roles, decision D106), deceased-marking, **CSV export** (decision D92), backup downloads, the **offline-restore forensic privileged-roster log** (decision D101), **de-brother / un-de-brother** actions (decision D115), **system-banner** set/clear (decision D117), **bug-report** submissions (decision D121), and Ghost-sync pushes. (Online bulk import is deferred — decision D100 — so it is not an MVP audit event.)

**Logging discipline: identifiers and field *names*, never field *values* — on *all* streams (decision P10, amending D61).** An audit entry records *that* brother #5247's address fields changed and *who* changed them — not the address itself. Logs persist under different access controls than the directory, so writing contact values or restricted fields into them would quietly route protected data around the server-side privacy projection (§1.4). This discipline extends beyond the audit stream to the **diagnostic and error streams too**: request/response bodies and Ghost payloads are **never** logged, and PII is scrubbed from diagnostic/error output (tested). The one acknowledged edge is that for some actions the *target identifier* is itself the value (a `bigBrotherId` set, a brother starred); these are IDs, not contact data, and stay within the boundary.

**Retention and access.** Diagnostic logs keep Cloud Logging's default retention; the `audit` stream is routed to a dedicated log bucket with a retention horizon pinned at **3 months** (decision P16 — no compliance schedule is required under D77, so this is a concrete *operational* figure, aligned with the backup and headshot-version windows, §6.3/D94), inexpensive at Book's volume. Because everything is structured, an operator — or an automated **log-reader agent** — can query precisely (`jsonPayload.action`, severity, actor) rather than scraping text. That agent is a planned ops consumer, constrained to **first-party / on-premise processing on a local model** (no audit content egresses to an external LLM — decision D91), authenticated by the **same off-Ghost-path service-account pattern as the Linter** (§5.2, D58): a dedicated service account holding `roles/logging.viewer`, no new auth surface.

**Alerting and the scheduled health-check job.** Security-relevant events (e.g. a burst of sign-in denials) drive **log-based metrics and alerting**; the **cache-age / snapshot-listener-disconnect watchdog** (resume-token / last-event age, §1.5/D83) alerts if the safety-net listener stalls; the **rate-limit / abuse-control policy** (§2.7/D86) bounds load on auth, writes, and the bulk read; and an anomalous-newsletter-correction-volume alert guards the D103 reconcile. These run alongside the **one consolidated scheduled sysadmin job** (decision D99, §5.1) — reconciliation audit + login canary + `IdentityProvider` contract test — a single Cloud Scheduler wake-up under scale-to-zero, reporting/alerting on what it finds.

### 6.2 Analytics (Mixpanel)

Book is instrumented for Mixpanel, the analytics platform already serving the PBE News Ghost site. Because Book and Ghost are one composite system (§5) — and because the planned cross-app movement is real (the Linter will deep-link bylines to Book profiles; a Book profile may someday list the Ghost articles that mention a brother) — they share **one Mixpanel project**, so a brother's journey across both halves can be analyzed as one funnel.

- **Identity.** Both apps `identify()` on the **email** join key (the same key that joins them for authentication, §2.1/§5.1), with Mixpanel's **Identity Merge** enabled so the rare, already-propagated email change (§5.1) folds the old and new identities into one user rather than forking history. Email-as-key was chosen over the Constitution ID — Book's true person key — only because Ghost has no clean way to carry the Constitution ID into the browser where its Mixpanel runs: Ghost's client-readable member object exposes essentially email/name/uuid, the pushed name deliberately omits the `(#5247)` form (§5.1), and the `note`/`label` fields are admin-side and not exposed to theme JavaScript. The Constitution ID is not lost, however —
- **User properties.** Book sets **Constitution ID and role as Mixpanel user properties**; **`name` is dropped** (decision D88 — pure data-minimization under D77, at zero analytics cost). Because user properties attach to the merged user profile, the Constitution ID becomes visible and segmentable on every user even though only Book knows it; **role** (brother / manager / admin) makes every metric segmentable by privilege — the authenticated-app advantage the anonymous Ghost-site analytics lack.
- **Event discriminator.** Every event carries an **`app` property (`book` vs. `news`)**, so per-app boards and cross-app funnels both draw from the same data; this is what keeps a shared project legible.
- **Taxonomy.** Beyond SPA page-views (tracked on route change: Directory, Profile-view, Profile-edit, Admin, Add-Brother), Book tracks the feature events that reveal *what brothers actually use*: search performed, filter applied (which field), sort changed, column-lens changed, star toggled, "Starred only", "Include deceased", profile saved, photo changed, verify, export, Ghost sync, backup download. (Online bulk import is deferred, D100, and restore is an offline event, D101, so neither is an MVP analytics event.)
- **Event properties must not carry PII (decision P6, enforcing D62).** Properties carry only *structural* context — *which* filter field, *which* sort — and **never** the search term itself, the viewed/starred/edited brother's identifier, or any record value. Writing those would leak the who-searched-for-whom / who-starred-whom graph through analytics, around the server-side projection. Enforced by **code discipline** (the harm is low and the detection cost high, so no automated test is mandated).
- **Do-not-track.** **`ignore_dnt: true` is kept** (decision D88): a browser DNT/GPC signal is a *global* "don't let websites track me," a poor proxy for intent about pbe400.org measuring its own members' use of its own consensual, members-only internal tool — the same over-broad-flag misreading as the MITAA "do not email" case. Consistent with the Ghost site.

A small follow-on this implies, recorded in `PRE-LAUNCH-TOOLS.md`: confirm/adjust how the *live* Ghost theme currently identifies members, so both halves resolve the same Mixpanel identity.

### 6.3 Backups & restore

Book's durable data lives in two stores that need different treatment: Firestore (structured text — `profiles`, `users`, `majors`) and GCS (binary headshots/thumbnails, already protected by object versioning, D8). The backup story is two-tier:

- **Automated operational backups.** A scheduled job (Cloud Scheduler → a backup endpoint or function) writes a **timestamped JSON export of all three collections** to a backup bucket on a regular cadence (daily is ample at this scale). Images are *not* re-copied each cycle — GCS object versioning already preserves their history — but each JSON snapshot includes an **image-version manifest** (the current `headshotVersion` per profile) so a restore can pin exactly the right object versions. GCS lifecycle rules age snapshots out at a **3-month retention** horizon (decision P16, aligned with the audit-log and headshot-version windows). The backup bucket is **ACL-restricted** (service-account / admin only, never public) and relies on **GCS default encryption at rest** (CMEK judged overkill for a volunteer nonprofit) — it holds off-platform PII and is treated as crown-jewels (decision D101).
- **Manual portable backup.** The Admin page's "Download backup" control (PRD §5.8.1) produces a single downloadable archive — the collections JSON **plus a zipped folder of the current headshot/thumbnail objects** — giving the admin a complete, off-platform copy in which images are separate files rather than embedded in the data. Because this archive leaves Book's controls, the **admin is named as its custodian** (USER-MANUAL; decision D101). *(As shipped in Phase 5a-1 the control downloads the **collections JSON only**; the image-object zip and the automated nightly job both land in Phase 7. The current export fully materializes each collection into one JSON reply on the single `max-instances=1` instance — acceptable at this roster size for a rare admin action, but the Phase-7 rework should **stream** the response (NDJSON/chunked) and/or reuse the already-hydrated `ProfileCache` for the `profiles` slice instead of a second full Firestore read + copy, so a download never competes for heap with live `/api/profiles` serving; OFC-188.)*

**Restore is an offline maintenance event (decisions D100/D101).** Restore is no longer an online async operation: Book enters **hard maintenance** (serves a "down for maintenance" response; the single instance neither writes nor serves its cache as authoritative), the three collections are replaced from the backup, and Book restarts and **cold-hydrates** its cache and in-memory email index from the restored data (§1.5/§5.1). Because the system is down during the replace, the listener-storm, in-flight-coherence, and out-of-band-uniqueness concerns all vanish — there is no live instance to be a second writer. It remains a **faithful whole-database replacement** ("be exactly this snapshot," *not* "merge corrections"), so the field-level edit rules of §6.5 stay bypassed — but it must pass **structural validation before the write**: big-brother cycle detection, ID uniqueness, email uniqueness (normalized, one namespace, per D97), and reference integrity, so a corrupt or tampered backup cannot reintroduce a structurally broken database. **Roles are restored verbatim, no gating** — in-Book role-gating is theater against an actor who already holds (or can mint) admin, and it would force an incoherent "be this snapshot *except* roles" semantic; the residual (a socially-engineered restore that plants `role: admin`) is pushed down to backup-bucket security and named in the threat model (§2.2/D105). Restore brings back `ghostMemberId` and verification fields verbatim (`ghostMemberUuid` no longer exists — D81). On restart, Book writes a **forensic privileged-roster log entry** (the resulting admin roster, plus a delta against the prior state when readable — non-preventive, but it makes a planted admin visible), and the **reconciliation audit is triggered immediately** (not left to the next scheduled run) to surface any Book↔Ghost divergence a rollback created, which the admin approves as a reviewed bulk re-push (never a blind mass push). A signed backup manifest is skipped for MVP (structural validation forecloses structural corruption; cryptographic input-authentication is the natural upgrade if backup *authentication* is ever wanted). It is the most destructive operation in Book, which is why its UI carries Book's strongest confirmation.

**DR / continuity posture (decision D102).** **RPO ≈ 24 hours** from the daily backup — and at Book's write volume (≈2 edits/month) the *effective* loss on a restore is usually zero. **Single-region is accepted explicitly:** Cloud Run / Firestore / GCS are regional, and multi-region cost is unjustified for a bi-annual-traffic volunteer directory; the region-failure **RTO is hours** (redeploy + restore), bounded in practice by admin-awareness latency, not the sub-hour restore. **Continuous backup-integrity verification** runs via **ephemeral, script-provisioned staging**: setup/teardown infrastructure-as-code stands a throwaway environment up on demand, and a periodic job (weekly by default) restores the latest backup into it, runs structural validation + a hydrate-and-count smoke check + checksums, then tears it down — nothing idle to rot between runs. Those same scripts **are** the single-region DR recovery runbook, so every backup test rehearses "stand Book up from nothing," de-risking the RTO. They join the sysadmin tooling inventory (PRE-LAUNCH-TOOLS) and run separately from the §6.1 daily production health-check.

**Maintenance and outage signalling (decision D118).** The "hard maintenance" state restore invokes is generalized into a first-class capability covering both *planned* downtime (restore, migration, a deploy) and *unplanned* outage. Because the backend may itself be the thing that is down, two mechanisms cover the full matrix. For **planned** downtime an operator **script swaps a static `maintenance.html` in at the Firebase Hosting / edge layer** (§6.4), served independently of Cloud Run, so a visitor loading Book fresh gets an honest "down for maintenance, check back later" page even with the backend stopped. For **unplanned** outage — and for an already-open tab during planned downtime — the **cached SPA shell** (D73) detects that its API calls fail, time out, or return a maintenance signal and **renders a maintenance/outage fallback itself** rather than a broken UI. To let the cached SPA tell the two apart, the backend (when reachable) returns a **maintenance-flavored `503`** with `Retry-After` during planned downtime (API-SPEC §1.5/§10), so it shows "down for maintenance" copy; an unannounced hard failure falls back to generic-outage copy. Operator-script triggering (over an admin-page toggle) keeps the planned path working precisely when the app is intentionally down, and matches how restore and migration already run offline. The script and the static page are inventoried in `PRE-LAUNCH-TOOLS.md`.

### 6.4 Transport security (TLS)

TLS requires no manual certificate management on the chosen stack (D10): on **Firebase Hosting + Cloud Run, certificates are Google-managed and auto-renewing.** Firebase Hosting provisions and renews the certificate for `book.pbe400.org` automatically and fronts the whole origin, rewriting `/api/*` and `/img/*` to Cloud Run (decision D126 — there is no Cloud CDN and no external load balancer, so no separate certificate to manage there). There is no certbot and no renewal cron. Book adds the standard hardening on top: forced HTTP→HTTPS redirection (automatic on Firebase Hosting) and the already-`Secure`, `HttpOnly`, `SameSite=Strict`, host-only session cookie (§2.3). Custom-domain DNS bring-up is a deployment step (`CODING-PROJECT-PLAN.md`); certificate provisioning follows automatically once the domain verifies.

**Security headers (decision D107, growing D64's HSTS-only set; implemented per decision N54).** Alongside **HSTS** (an explicit `Strict-Transport-Security` header, so `book.pbe400.org` is covered at cutover rather than relying on the `.app`-TLD preload the `web.app` staging host rides), Book serves:
- a **strict Content-Security-Policy** — `script-src` / `style-src` / `connect-src` / `img-src` / `worker-src` / `frame-ancestors`. **`script-src` carries no `'unsafe-inline'`**: the only inline scripts are the two in-`<head>` no-FOUC setters (theme + font size, §5.3/D30), admitted by their **SHA-256 hashes** instead, and a CI guard (`assert:csp-hashes`, run after the build) fails the build if those hashes drift from the **shipped** `apps/web/dist/index.html`. `style-src` **does** allow `'unsafe-inline'` — unavoidable for React inline styles, the virtualized grid's per-row transforms, and the theme/font-scale CSS variables. `img-src` is `'self' data: blob:` (headshots/thumbnails are app-served same-origin per D126, plus the Avatar gradient and the crop preview); `connect-src 'self'`; `worker-src 'self' blob:` (the Name-Search worker). When Mixpanel lands (§6.2/D62, Phase 7) its script and connect origins are **added to the allowlist** so the strict policy doesn't silently break analytics. (Firebase Hosting applies this `**` header set to the `/api/**` and `/img/**` rewrite responses too, and it **takes precedence** over what Cloud Run sets — verified against the live staging deploy — so in production those responses carry the SPA policy; the backend's own `onSend` header hook is defense-in-depth for direct-to-Cloud-Run access that bypasses Hosting, and a CSP on a JSON/image response is inert regardless);
- **`X-Content-Type-Options: nosniff`**;
- a **framing policy** (`frame-ancestors 'none'` / `X-Frame-Options: DENY`) — this is also what forecloses a hidden-iframe re-auth, making the §2.3 recovery a child *window* (D109);
- **`Referrer-Policy: strict-origin-when-cross-origin`** (or stricter), landing finding **P9**, so the Name-Search term and `/brother/:id` path do not leak as `Referer` to `/api`, `/img`, or external obituary links.

These compose with the deny-by-default CORS and the URL-scheme allowlist (§2.7, DATABASE-SCHEMA §8).

### 6.5 Data-maintenance operations

Three recurring data operations, beyond the per-record editing the app provides, carry operational rules:

- **Bulk-CSV import and verification (online import descoped — decision D100).** Book has **no online bulk-write path**: every write is a single authenticated edit through the one live instance, so an online admin bulk-import endpoint is **not built for MVP** (amends D52/D68). The realistic driver — applying externally-vetted MITAA-sourced corrections (§5.3) — is, by measured cadence (≈1 reconcilable MITAA death/month), a handful of single manual edits per year, not a bulk job; a **MITAA-specific bulk import is deferred to the post-MVP backlog**, and any genuinely unforeseen bulk update is a rare **offline** operator task (Book taken down, the change applied, service resumed). Whenever such an import does run, it retains the **per-row non-owner edit** semantics that resolve D28: a row whose content changes **unverifies** that profile (clearing `lastVerifiedDate`/`verifiedBy`); unchanged and deceased rows keep status; the CSV **does not accept the verification fields** (system-managed; ignored on import, exportable as read-only columns); and a **dry-run preview** reports "will update X, add Y, **unverify Z**" so the blast radius is visible. (The one-time pre-launch bulk-loader, D57, is a separate external tool exempt from these rules — the genesis load.)
- **Majors-vocabulary maintenance.** The `majors` controlled vocabulary (DATABASE-SCHEMA §6.2, D14) changes rarely. Rather than build a runtime admin UI for MVP, the canonical list is a **version-controlled source file in the repo**, applied by a small **idempotent seed/reconcile script** that creates or updates the `majors` collection from it. The same script performs the initial load, so it is both a pre-launch tool and the steady-state maintenance path — "edit the file, run the script" — which keeps the vocabulary reviewable in git and removes the footgun of hand-editing production documents (a mistyped code silently breaks every profile that references it). A runtime majors-curation UI on the Admin page remains a recorded future tenant.
- **Schema migration.** Firestore is schemaless, but Book is not — it carries a strict shared `Profile` type (D3), and as that type evolves (bug fixes, new features) stored documents drift from it. Migration is handled in tiers by change kind. **Additive / optional changes need no migration:** a new optional field is simply absent on older documents, and the read path and shared validation module (D50) supply the default — the common case, at zero cost. **Breaking or structural changes** (renames, retypes, restructures, or backfilling a now-required field) get a **numbered, idempotent migration script**, checked into the repo, dry-run-able, tested against the Firestore emulator with the fake dataset (§6.6), and run against production **only after a backup** (§6.3), with the run recorded to the audit stream (§6.1). A schema-version marker — a small global `_meta` document — orders the migrations and records what has been applied. As **operator / offline tooling, not an online Book feature** (decision D100), migrations carry R15's execution hardening: they run **in-network as a Cloud Run Job/Task with cursor-resume** (never from a laptop), and are either **backward-compatible-read-then-migrate** or run with the **old revision drained first**, closing the mid-deploy old-code race. At Book's scale (~2,000 documents, all loaded into the in-memory cache on cold start) **eager batch migration** is the right model: it completes in seconds; lazy, per-document on-read migration is the escape hatch only if a future change ever made a batch run impractical. These scripts differ from the one-time pre-launch loaders (D57) in being a **recurring** category that lives in the repo with the code.

### 6.6 Testing strategy

This section fixes the testing *methodology*; the *sequence* in which tests and fixtures are built, the "tests-green gate between phases," and the CI pipeline belong to the build plan (`CODING-PROJECT-PLAN.md`). The guiding principle is **test anything with non-trivial logic** — branching, transformation, parsing, derivation, or a state machine, i.e. any function where a bug is plausible and the TypeScript type system will not catch it — while skipping trivial passthrough and presentational glue. That is deliberately broader than "high-risk only" and narrower than coverage-for-its-own-sake.

**Unit and integration tests — Vitest.** The runner is **Vitest**: Vite-native, fast, Jest-compatible, and one tool across the shared frontend/backend TypeScript (D3/D10). Pure-logic units are tested directly; backend code that touches the data store is tested against the **Firestore emulator** (local, free, faithful query/concurrency behavior, no cloud cost or flakiness). Priorities, by the principle above:

- *Security- and integrity-critical, tested exhaustively:* the **shared validation module** (every DATABASE-SCHEMA §8 rule, including the strict URL-scheme allowlist and the magic-byte / ~40 MP image bound, D107); the **per-role projection** (§1.4) — every visibility class against every role, confirming off-toggle values are hidden from peers and managers but not admins, `adminNote` is hidden from the owner, and `ghostMemberId` never leaves the backend — plus a **cross-caller isolation** test asserting no two callers of different entitlement receive each other's bulk projection (the split-read invariant, D82); the **write-field allowlist** (out-of-allowlist and system/consent fields rejected per role) and the **object-level predicate** / IDOR check (D106); **authentication** — JWT verification against a mocked JWKS *including forged-`alg` cases* (`alg:none` and a symmetric token HMAC'd with the public key, both rejected — D104), the single-use callback nonce, session establishment and the 4-hour cap, email→user resolution (normalized, deny-unknown-email, fail-closed-on-duplicate, first-match role assignment, create-if-absent), and the `IdentityProvider` seam; the **server-enforced last-admin invariant**; the **image read path's session gate and per-record visibility** (decision D126); and the **big-brother cycle check**.
- *Non-trivial logic, tested solidly:* **Canonical Name** construction (ambiguity detection, ID disambiguation, suffix folding — load-bearing for the Directory and the Linter roster); the **client query engine** (name tokenization, AND-across-tokens, the filter grammar's OR-within / AND-across, the universal canonical secondary sort key); the **verification coupling** (owner auto-verify, non-owner unverify, deceased exemption) and the **import diff** of §6.5; **optimistic concurrency** (`If-Match` / 412); **Ghost-sync push** lifecycle and field mapping; **CSV export** (role-projected columns, escaping, selection scoping) and **CSV import** parsing; the **image pipeline** (crop → 512×512 WEBP + 96×96 thumbnail derivation; no regenerate feature, D114); **address** logic (country-conditional state/province, code↔name, invalid-subdivision clearing); **class-year** null / `'84` / range-grammar handling; **majors** lookup and primary-first ordering; **stars** optimistic toggle and revert; the **reconciliation audit** categories (including `bookInternalOrphan`) and its read-only-into-Book invariant **except** the scoped newsletter most-recent-change-wins reconcile (both directions, timestamp-driven — D103); the **CSV formula-injection neutralization** (a malicious leading `= + - @` cell is prefixed, in both the general and MITAA exports — S9); the **backup/restore round-trip** including pre-write **structural validation** (cycle / ID-uniqueness / email-uniqueness / reference-integrity, D101); the **roster endpoint** field subset, the subject-pinned auth path (wrong-issuer/-audience/-subject all rejected, D78), and the optional-absent serialization (C11); and **URL/state** serialization with scroll restoration.
- *Targeted frontend component tests:* the per-role conditional rendering; the privacy switches showing the **active-side consequence inline with the counterfactual in the `?` tip** (D113); the **non-destructive 401/403 session-expiry recovery** (form preserved, child-window re-auth, Save resumed with the original `If-Match` — D109); and the help toggle-tips.

**End-to-end tests — Playwright.** Browser-level flows are exercised with **Playwright** — the standard for AI-agent-driven, CI-friendly, multi-browser testing, and the carrier of the accessibility checks below. (The Claude-in-Chrome / Preview MCP tools are for interactive, exploratory verification during development; the *committed, repeatable* suite is Playwright.) E2E covers the core journeys: the sign-in bridge, search/filter/sort, opening and editing a profile (including the crop flow and the 412 reconcile), starring, export, the admin operations, and the responsive card/grid switch.

**Fake test dataset — a committed generator.** Tests run against a **deterministic, seeded generator** that produces ~600–2,000 obviously-fake-but-structurally-realistic profiles (silly names, fake domains — never confusable with real people). The data deliberately spans every feature: null class years, double majors, deceased brothers, missing headshots, every privacy-toggle combination, big/little-brother trees, international (free-text state) versus US/CA (coded) addresses, emergency contacts, `adminNote`s, and verified/stale/unverified states — with international, hyphenated, middle-name, and mug-name cases to exercise phonetic search and virtualization-at-scale, and fake placeholder headshots to drive the image pipeline. The repo holds the **generator and a small fixed fixture set**, not a large generated blob (a fixed seed reproduces identical data on demand, and the blob would be pure diff noise); E2E runs generate the full set into the Firestore emulator at setup. The generator is an **early build deliverable** — almost everything else needs it to be testable. It is strictly a development artifact and never touches production data.

**Phonetic-matching A/B (D35).** Whether the Name Search uses **Double Metaphone** (the incumbent — English-centric, compact codes) or **Beider-Morse** (stronger on the non-English names of a globally-drawn brotherhood, but far more codes per name) is an *experiment to run at build time*, not a planning decision. What is fixed now is the method and the criterion: an **offline evaluation harness** (both algorithms come from the talisman library) run over a curated set of (query → expected-match) pairs built from the error classes actually expected — transliteration variants, sound-alikes, hyphenated/middle-name cases — seeded from the fake dataset plus hand-picked international names. The decision **favors recall** (for a small directory it is better to surface a couple of extra near-matches than to miss the brother) provided precision stays tolerable; index size and latency are tie-breakers only, both trivial at this scale. The chosen algorithm is recorded when search is built. The index build and phonetic-code generation run **in a Web Worker** (decision D110, §1.6) — the grid renders immediately with exact/substring matching, and fuzzy/phonetic switches on when the worker signals "index ready" — and the codes are **recomputed each load rather than memoized to IndexedDB** (no name-keyed PII on disk, consistent with `no-store`, D95); a test covers the worker-ready progressive-enhancement transition.

**Accessibility testing.** Because **WCAG 2.2 AA** is policy (decisions D32, **D79** — raised from 2.1 AA; 2.2 is a strict superset), conformance is *verified, not asserted*, in three layers: **automated** — `@axe-core/playwright` run against every page and key state, failing CI on violations (contrast, ARIA, names, roles); **contrast gating** — the AA palette from the visual-design pass enforced by automated contrast checks in CI; and a **manual per-page checklist** for what automation cannot catch — a keyboard-only walkthrough confirming tab order matches reading order (D43), visible focus throughout, focus trapping in the crop modal and dialogs, a screen-reader smoke test of the core flows, `prefers-reduced-motion` honored, and usability at 200% zoom and the largest font scale. The checklist folds in 2.2's relevant new success criteria — **2.4.11 Focus Not Obscured (Minimum)** and **3.3.7 Redundant Entry** add genuine new checks (the sticky-header virtualized grid; the multi-step edit/import flows); **2.5.8 Target Size**, **3.3.8 Accessible Authentication**, and **3.2.6 Consistent Help** are already met by D32's ≥44 px targets, the magic-link bridge, and the D53 single-source help — and **drops the obsolete 4.1.1 Parsing** (removed in 2.2). The virtualized grid is additionally required to carry and is **tested for `aria-rowcount` / `aria-rowindex` / `aria-setsize`** (finding U6). Since shadcn/Radix supply accessibility by construction (D29), most of this guards against regressions in the custom surfaces — the majors chips, the data grid, and the virtualized list's keyboard access.

**System & acceptance testing.** Before launch, real users (brothers, managers, an admin) exercise the complete system on a **staging deployment** loaded with the fake dataset, never real member data — where they try features, give feedback, and try to break things. Staging is **ephemeral and script-provisioned** (decision D102): rather than a standing second environment, setup/teardown infrastructure-as-code stands a throwaway Book up on demand (the lightest viable form is a persistent-but-*empty* staging project into which the scripts provision an ephemeral Firestore database + bucket), and those same scripts double as the single-region DR runbook and the backup-integrity harness (§6.3). The Ghost dependency, which might seem to force a paid second Ghost Pro instance, is handled by separating two concerns. **User-acceptance testing of Book** uses a **`DevIdentityProvider`** — the test implementation of the `IdentityProvider` seam (§2.2, D21) — which issues a session for a chosen identity and role with no Ghost in the loop, so testers can freely assume brother / manager / admin roles and exercise every tier; the same provider lets local development and the Playwright suite run without Ghost. It is locked out of production by **four independent layers (decision D108)**, not one env gate: **(1)** it is **compiled out of the production bundle entirely** (a separate entry point / build-time exclusion), so the prod artifact does not *contain* its code — the load-bearing layer, since "cannot be loaded" dominates "is disabled"; **(2)** D72's **runtime env gate** is retained; **(3)** a **CI/build assertion** fails the build if the prod artifact can instantiate the dev provider; **(4)** a **startup alert** fires if it is ever loaded under a prod-like configuration. Ephemeral, script-provisioned staging is now its only legitimate home. **Verifying the real Ghost integration** is then a narrower exercise that stays free. The auth / read path (the Handlebars relay, the member JWT, JWKS verification, §2.1) is tested at full fidelity by adding an *additive* staging bridge route on the **real Ghost Pro** (e.g. `pbe400.org/book-staging` → the staging Book), exercising the genuine production handshake without a second instance or disturbing the live newsletter; the verification logic is additionally unit-tested against JWTs signed by a controlled test key. The only piece that genuinely needs a non-production Ghost is the member-management **write** path (Book → Ghost create/update/delete, §5.1), which staging must **never** point at production Ghost — it would mutate live members — so it is stubbed and logged for UAT ("would create member X"), with a short-lived self-hosted open-source Ghost or a time-boxed Ghost Pro trial reserved for a faithful write-path run. The staging environment's bring-up and the pre-launch UAT phase are sequenced in `CODING-PROJECT-PLAN.md`.
