## 1. Overview

Book is a members-only online address book for Phi Beta Epsilon (PBE), an MIT fraternal living group founded in 1890 with ~40 active undergraduates and ~800 alumni. It replaces the defunct paper address books (last published in 1990) and the reliance on inaccurate MIT Alumni Association data, providing an accurate, modern, self-maintained directory of the brotherhood. Book integrates with the existing PBE News publication (Ghost, pbe400.org) and is tentatively hosted at book.pbe400.org. At its core it is a CRUD application over a small collection of brother "profiles."

**Book and Ghost are one composite system.** Book is not designed as a standalone application: it is wired into Ghost so the two operate as a single whole. Ghost authenticates every Book user (ENGINEERING-DESIGN §2), and Book is the management surface for Ghost's members (§6.1) — and a deferred third tie, Ghost content search (§3.2), would bind them further. The dependency is asymmetric: Book cannot function without Ghost (no authentication, no access), whereas Ghost can fall back to its own admin panel for member management if Book is ever unavailable. This framing explains why member-data editing is removed from Ghost and redirected to Book, and why the two systems' data is kept in sync.

**Population and two staffing limitations.** Every brother in Book is an initiated PBE member — an MIT undergraduate at initiation or an alumnus — so the population is **adults (18+)**; Book carries no minor-consent machinery (P19). And because sign-in is the Ghost email bridge, two small groups cannot self-serve and are **staff-maintained** by design: a brother with **no email address** on file, and a brother who exists only as an **unidentified historical Ghost address** not yet linked to a profile. Neither can authenticate, so an administrator maintains their records and links them when an identity is established (a corollary of the deny-unknown-email resolution, ENGINEERING-DESIGN §2.1; USER-MANUAL §11).

## 2. Terminology

- **PBE / Phi Beta Epsilon** — the fraternity Book serves.
- **Brother** — a member of PBE.
- **PBE News** — PBE's bi-annual online alumni publication, members-only at pbe400.org, built on Ghost (Ghost Pro hosting).
- **Book** — the working name of this application; tentatively book.pbe400.org.
- **Profile** — the record holding all information about one brother.

## 3. Scope (MVP vs. Deferred)

The first release ("MVP") delivers a complete, usable address book. Features tagged "nice-to-have" or "future work" are deferred so that planning and build effort concentrate on the core. Deferral means "not in the first release," not "rejected."

### 3.1 In scope (MVP)

- Brother profile data model and full CRUD.
- Directory page — search / filter / sort, performed client-side over the full dataset.
- Profile view/edit page.
- Authentication via a Ghost auth bridge.
- Three permission levels: Brother, Manager, Administrator.
- REST API under `/api/*`: the bulk `GET /api/profiles` (per-role projection) and `GET /api/me` self-fetch, single-record CRUD on `/api/profiles/{id}` with a dedicated verify action, self-service stars, the headshot upload/remove endpoints (reads are served by the backend from a private bucket, decision D126), the admin role-change endpoint, and the first-party Linter `GET /api/roster` (the full contract is in `API-SPEC.md`).
- SPA frontend that bulk-downloads the dataset the user is permitted to see.
- Headshots with crop-on-upload; precomputed thumbnails.
- Stars (personal favorites list).
- Per-field privacy controls, enforced server-side.
- Admin functions: **download** a database backup, add/delete brothers (single-record), and the read-only Ghost reconciliation audit. Book has **no online bulk-write path** (D100): online bulk-CSV import is deferred (§3.2), and restore is an **offline operator procedure**. (There is no thumbnail-regenerate feature at all — D114.)
- Manual and automated backups (restore is an offline maintenance event, ENGINEERING-DESIGN §6.3).
- Logging and Mixpanel analytics.
- Deployment to GitHub, served over HTTPS.

### 3.2 Deferred (post-MVP)

- Report / brother-analytics page — statistics and charts (e.g. living vs. deceased, brothers by class-year decade) and, enabled by the ISO-coded `country` field, geographic analytics such as a world map of brothers by country.
- Ghost content search, and the "Search News" per-brother link that depends on it.
- Big Brother graph browser. (Entering Big Brother data on a profile remains in MVP; only the graphical browser is deferred.)
- Mailman mailing-list integration.
- External / third-party API access.
- Automated MITAA data exchange. Any automated interface is deferred; reconciliation is a handful of manual single edits per year (see the MITAA bulk-CSV import item below).
- **MITAA bulk-CSV import** — a batch upsert of externally-vetted MITAA-sourced corrections. Deferred to the post-MVP backlog (D100/R11): by actuarial cadence (≈2 deaths/month, more than half never reported to MITAA), realistic reconciliation is a few **single** manual edits per year, not a bulk job — so there is no online bulk import to build for MVP. Pulled forward only if a real bulk need emerges; any rare bulk update is otherwise an offline operator task (ENGINEERING-DESIGN §6.5).
- Physical-address validation / autocomplete on entry — adopt only if a free, reliable service is available.
- "Share this selection" — sharing an explicit, viewer-independent set of brothers via a URL list of IDs (e.g. `?brothers=810,814,927`), so an opened link shows the same rows for everyone. Three parts of rising cost: the URL encoding and display logic (small); a manager/admin "Share selected" action reusing the existing row-selection checkboxes (small–moderate); and a general, brother-facing multi-select mechanism to *designate* such a set (the real lift, and the reason this is post-MVP). Distinct from the personal starred-only toggle, which remains a live client-side filter (ENGINEERING-DESIGN; DECISIONS D31).
- Mailman integration, and Mailman *replacement* — direct integration with PBE's two MIT-hosted Mailman 2.1.34 lists (pbe-brothers-official, pbe-connect) is deferred (no usable API; the instance cannot be modified — ENGINEERING-DESIGN §5.4). Two replacement directions are noted for the future without choosing between them: migrating the moderated **pbe-brothers-official** list into Ghost as a second newsletter, or self-hosting a PBE-owned **Mailman 3** and integrating Book to it directly (which could replace both lists).
- Multiple-newsletter support — generalizing the single `allowNewsletterEmail` boolean into a per-newsletter subscription set over a small "newsletters" vocabulary, mirroring Ghost's native model. Needed only if PBE adds newsletters beyond PBE News (e.g. if a Mailman list migrates into Ghost, above); MVP supports the one PBE News newsletter.
- Byline → Book profile deep-link — making a brother's name in a PBE News byline link to their Book profile page. Conceptually distinct from Ghost content search (it needs no indexing, only the brother's Constitution ID in the Ghost template), and the one feature that would justify making Book's primary key visible inside Ghost.
- General nickname field — a brother's everyday nickname when it differs from the `mugName` (some brothers have both), of possible use to Linter matching and to display. Deferred pending a decision on need.

### 3.3 Notes on near-the-line items

- **Ghost membership sync** is in MVP and now fully designed (§6.1; ENGINEERING-DESIGN §5.1): Book is the single master, Ghost editing is disabled and redirected to Book, Book pushes changes to Ghost **synchronously on save** (diff-based, D96), and the "Sync with Ghost" audit reconciles — read-only into Book for every field except the scoped most-recent-wins newsletter-subscription reconcile (D103) — and now runs on a schedule as well as on demand (D99). The one-time pre-launch data load is handled by standalone migration tools, not Book.

## 4. Permissions

Book has three permission levels, stored in Book's own `users` collection independently of Ghost: **Brother**, **Manager**, and **Administrator**. An unauthenticated visitor has no access and is redirected to the Ghost auth bridge to sign in (ENGINEERING-DESIGN §2). The guiding model is a housekeeper-and-homeowner one: a manager tidies what is in the open — maintains the directory — but does not open private drawers (cannot see a value a brother has hidden) or knock down walls (cannot delete, add, or restructure); the administrator, as the homeowner, can do anything.

### 4.1 Capability matrix

| Capability | Brother | Manager | Admin |
|---|:---:|:---:|:---:|
| Sign in via the Ghost bridge; use the app | ✓ | ✓ | ✓ |
| Search / filter / sort; view any profile | ✓ | ✓ | ✓ |
| Edit own profile; star / unstar; verify own record | ✓ | ✓ | ✓ |
| Make own record **Unlisted** (hide self from the directory) | ✓ | ✓ | ✓ |
| See another record's restricted flags, preferences, and dates (read-only) | — | ✓ | ✓ |
| See and edit staff notes (`adminNote`) on any profile | — | ✓ | ✓ |
| Edit any brother's *visible* directory fields | — | ✓ | ✓ |
| Add / change / remove any brother's headshot | own | ✓ | ✓ |
| Set deceased status and deceased fields | — | ✓ | ✓ |
| Verify another brother's record (with on-screen caution) | — | ✓ | ✓ |
| Export CSV of the current view | — | ✓ | ✓ |
| See a contact value a brother has hidden behind an *off* toggle | — | — | ✓ |
| Change another brother's privacy / consent settings (incl. Unlisting them) | — | — | ✓ |
| Add a brother | — | — | ✓ |
| Delete a brother (single-record) | — | — | ✓ |
| De-brother / reinstate a member (extreme circumstances) | — | — | ✓ |
| Change a brother's role | — | — | ✓ |
| Run the Ghost↔Book reconciliation audit | — | — | ✓ |
| Export data for MITAA sharing | — | — | ✓ |
| Download a database backup | — | — | ✓ |

The **headshot** row reflects C6: a brother manages his own photo, and managers and admins may add/change/remove *any* brother's headshot (editing others' photos is intentional, part of directory upkeep). **"Change a brother's role"** is the renamed action (formerly "Toggle Privileges," C13). Add-Brother is administrator-only by design: the entire Constitution roster is pre-loaded, so new records appear only at the annual initiation or to correct a rare, serious problem — both of which warrant the administrator's attention.

**No online bulk-write operations (D100).** The matrix no longer lists online **bulk-CSV import** or **bulk delete**: Book has no online bulk-write path — every write is a single authenticated edit through the one live instance. The MITAA bulk-CSV import is deferred (§3.2) and **restore** is an offline maintenance event (ENGINEERING-DESIGN §6.3) — so the admin retains backup *download* but not an online restore button; single-record delete (on the Profile page, §5.7.10) covers the admin error-correction case. The **thumbnail-regenerate feature is dropped entirely** (D114): headshot and thumbnail are written as a matched pair (D98) so they never drift, and any future re-derivation is a one-off offline script.

**The one manager action with a consent/Ghost side-effect (C5).** Setting **deceased status** is the single manager-permitted action that also alters consent and pushes to Ghost — marking a brother deceased forces both email-consent flags off and unsubscribes him in Ghost (§6.1; DATABASE-SCHEMA §8). This is a deliberate exception to the "managers don't change consent" boundary, kept manager-allowed because it is **reversible** (the prior consent/verification state is snapshotted and restored on un-marking, D80) and **audited** (D61); it is the housekeeper noticing a vacancy, not opening a private drawer.

**Server-side enforcement is the write-side dual of the read projection (D106).** Every capability above is enforced server-side by a **per-role write-field allowlist** plus an object-level owner-or-staff predicate (the write-side counterpart of the §4.2 read projection); out-of-allowlist fields are rejected, not silently ignored; and a **server-enforced last-admin invariant** prevents demoting the only remaining administrator (with every role change audited before/after). The UI role-guards are conveniences over this authoritative server check (ENGINEERING-DESIGN §1.4; API-SPEC §3).

**De-brothering (decision D115).** In rare, extreme circumstances an administrator may **de-brother** a member. The effect is comprehensive: the record is **hidden from brothers entirely** (it leaves their directory, search, and export, and a direct link returns "not found"), while **managers and admins still see it, with the name struck through**; the member's **Ghost account is deleted** (blocking both Ghost and Book sign-in and stopping all PBE News mail); and the record is **excluded from the MITAA export** regardless of consent — a de-brothered person's data is shared with no one outside staff. It is **admin-only** and **reversible**: a mark-time snapshot restores the brother's consent and verification state and **re-creates the Ghost member** on reinstatement, exactly symmetric with un-marking deceased (D80). The control lives on the brother's Profile page (§5.7.10); the directory and profile presentation are in §5.6.5 and §5.7.10. De-brothered and deceased are independent states.

### 4.2 What each role sees

Field visibility is enforced server-side (DATABASE-SCHEMA §9); the frontend never receives data the caller may not see. The four directory-data visibility classes (public, toggle, restricted, private) yield three reading projections — the two further classes, staff-internal and system, sit outside this brother/manager/admin model (described below):

- A **Brother** sees public fields, plus another brother's contact fields (email, phone, address, emergency contacts) only where that brother's share-toggle is on; their own record in full.
- A **Manager** sees everything a Brother sees, **plus** every record's restricted flags, preferences, and dates — the share-toggles, `allowNewsletterEmail`, `allowCommentReplyEmail`, `allowShareWithMITAA`, `lastVerifiedDate`/`verifiedBy`, and `lastModified` — as read-only columns, **plus** the staff-internal `adminNote` (read/write). Crucially, a value a brother has hidden behind an *off* toggle stays hidden from managers too; they can see *that* a field is private without seeing the value. This lets managers coach brothers on their settings and chase stale records without exposing the protected data.
- An **Administrator** sees everything, including the contact values hidden behind off-toggles, and is the only role that can change another brother's privacy or consent settings.

Two field groups sit outside this brother/manager/admin reading model. The **staff-internal** `adminNote` — a free-text note for coordinating among staff and recording the history of manual changes — is visible to and editable by **managers and administrators only**; the brother themselves (the owner) and peers never see it, making it the first field a brother cannot see on their own profile. And `ghostMemberId` is a backend-only system field, never sent to any client. See DATABASE-SCHEMA §9.

### 4.3 Verifying a record

The "Verify" action stamps `lastVerifiedDate` and records `verifiedBy` (the Constitution ID of whoever verified) — both server-set so they cannot be forged. The owner, managers, and administrators may all verify, but the button carries role-specific copy to keep the meaning unambiguous: for the owner, a simple confirmation that their information is correct; for managers and administrators, a caution to verify only after they have directly communicated with the brother and the brother has confirmed their information. `verifiedBy` preserves the provenance — self-attested versus staff-attested — for every verification. Owner-vs-other is **enforced server-side** (D106): an ordinary brother may stamp provenance only on his own record, while verifying *another* brother is reserved to managers and admins.

Verification is coupled to content, not independent of it. A profile is "verified" exactly when it carries a `lastVerifiedDate`; there is no separate flag and no history. Because an owner editing their own profile is itself an act of confirmation, any owner edit auto-(re)verifies the record. Conversely, when a manager or administrator edits *another* brother's profile, a previously verified record becomes unverified — the data no longer reflects the owner's last confirmation — and the editor sees a warning to that effect at save time (Cancel, or Save anyway); they may re-verify afterward, if confident, using the cautioned button above. An unverified profile simply stays unverified when edited. The one exception is deceased brothers: once a profile is marked deceased there is no living owner to attest, so manager/administrator edits leave its verification state untouched.

## 5. UI Requirements

Book is a React single-page application built on shadcn/ui (ENGINEERING-DESIGN; DECISIONS D29). It has four primary pages — the **Directory** (search/filter/sort), the **Profile**, the **Admin** page, and the deferred Report page (§3.2) — plus a fifth, admin-only **Add-Brother** page at `/brother/new` (D31, §5.4): a focused entry form for the rare new record, counted explicitly here so the route is not overlooked (finding C12; USER-MANUAL). Session 4a settled the foundational, cross-page UX below; the Directory page is specified in Session 4b and the Profile and Admin pages in Session 4c. A cross-cutting in-context-help standard (§5.9), applying across all pages, was added as a Session-4c addendum.

### 5.1 Component stack and theming

The UI is composed from shadcn/ui — accessible Radix UI primitives styled with Tailwind CSS, their source vendored into the repo — with TanStack Table and TanStack Virtual powering the Directory grid (D29). Theming is expressed through Tailwind's CSS-variable token layer (one set of light/dark tokens), so the entire look can be re-skinned by changing token values without touching component logic. The concrete visual design — palette, typography, and overall aesthetic — is produced separately in a dedicated design pass with Claude Design, working from reference sites, once the page structures are locked; until then the app is built against shadcn's default theme. Any chosen palette must pass WCAG AA contrast in both themes (§5.5).

### 5.2 App shell

Every page sits inside a persistent slim top bar. Its left side carries the app name/logo, which returns to the Directory (the app's home). Its right side carries the user's identity and the global controls: a small person icon — the user's own headshot downscaled, falling back to a generic avatar — that links to the user's own profile, and, beside it for elevated roles, a role badge reading "MANAGER" or "ADMINISTRATOR" (ordinary brothers see no badge). On phone-width viewports the right-hand cluster collapses into a single menu.

**Session expiry (decisions D109, D125).** Book's session carries an absolute time cap, so a session does eventually end; when it does — or when a brother has been removed from Ghost — Book shows a clear *"Your session has expired — please sign in again"* message rather than failing silently (a silent re-bridge when he is still a Ghost member; a denial when he is not). Because session state is **persisted server-side** (ENGINEERING-DESIGN §2.3, D125), an ordinary scale-to-zero server restart does **not** end a session — so this is a genuinely rare event, not something that recurs every time the backend sleeps — and a session that lapses **mid-edit** preserves the in-progress form and re-authenticates without losing the work (D109).

**Report a bug (decision D121).** Beside the person icon in the right-hand cluster sits a visible **"Report a bug"** control. It opens a small form — the current page is captured automatically — that files the report to the backend, where it is **stored and audited** for an administrator to review. **No email is sent**, which keeps the admin's inbox out of the attack surface; the submitter is always an authenticated brother, and the endpoint is rate-limited. On phone widths the control folds into the same overflow menu as the rest of the cluster.

**Optional system banner (decision D117).** Above the top bar, an **optional admin-set system message banner** can span the full width of every page — the same function and appearance as Ghost's announcement bar, though the two are independent (neither propagates to the other). Only administrators set or clear it (§5.8); it carries an info/warning severity and **persists until cleared** (it is not per-user dismissible). It is distinct from the **maintenance page** (decision D118) — a full-page "down for maintenance, check back later" state shown when the system itself is down, not merely a message while Book is running.

**Persistent footer (decision D116).** Every page carries a slim **footer** at the bottom with a link to **PBE's privacy notice** (`https://pbe400.org/privacy/` — the same notice linked from the login/landing surface, D77). It is deliberately quiet — present so the notice is always reachable, not prominent.

### 5.3 Preferences

Three client-only preferences persist in `localStorage` (not cookies — D30): dark mode, font size, and the Directory's column selection/order (the last detailed in 4b). **Dark mode** is three-state — Light, Dark, or System, where System follows the OS `prefers-color-scheme` — applied by toggling a `.dark` class on the document root, with a tiny inline script setting it before React starts to prevent a flash of the wrong theme. **Font size** offers three discrete steps (Normal / Large / Larger) for older-alum readability, implemented by scaling the root font-size with all sizing in `rem` so the whole UI scales proportionally.

### 5.4 Navigation, URLs, and browser behavior

Routing uses React Router with nuqs for type-safe URL search-parameter state (D31). The routes are `/` (Directory), `/brother/:id` (Profile), `/brother/:id/edit` (Profile in edit mode, guarded by an unsaved-changes prompt), `/brother/new` (Add Brother; admin-only), and `/admin` (admin-only); role guards on the latter two are UX conveniences, with real enforcement server-side (ENGINEERING-DESIGN §2). Application state is partitioned into three stores by contract:

- **The view** — search text, active filters, and sort — lives in the **URL query string**, updated silently via the History API, so any view is shareable, bookmarkable, and fully reconstructable by a freshly opened tab.
- **Transient per-entry context** — the starred-only toggle and the scroll position — lives in **History API state**, so the back button restores it but it never travels in a shared link. The starred toggle is deliberately kept out of the URL because stars are per-user (DATABASE-SCHEMA): a shared `?starred=1` would otherwise show the recipient's stars, not the sender's.
- **The column lens** — column selection and order — lives in **`localStorage`** as a personal preference, intentionally not in the URL, so a shared link reproduces *what* the sender sees without imposing the sender's columns.

All navigation uses real `<a href>` anchors (React Router `<Link>`), so native shift/ctrl/cmd-click opens a new browser tab that boots an independent SPA instance reconstructing its state from the URL. Returning to the Directory via the back button restores the previous scroll position — a firm requirement, implemented by saving and restoring the virtualized list's scroll offset rather than relying on native browser scroll memory. Dates are displayed, and accepted as input, only in `YYYY-MM-DD` form, and user input is validated before acceptance (e.g. email-format checks).

### 5.5 Accessibility and responsive design

Accessibility is a first-class requirement: Book targets **WCAG 2.2 AA** as a matter of policy (decisions D32, **D79** — raised from 2.1 AA; 2.2 is a strict superset, and Book already meets its new target-size, accessible-authentication, and consistent-help criteria). The semantic-HTML + Radix foundation supplies keyboard operability, focus management, and correct ARIA roles by construction; on top of that Book commits to a visible (styled, never removed) focus indicator; AA color contrast verified in **both** light and dark themes; never encoding meaning in color alone (the star also changes shape, the role badge carries text, "In Memoriam" is rendered as words); form errors linked programmatically and announced; honoring `prefers-reduced-motion`; a skip-to-content link; alt text on headshots and thumbnails set to the brother's canonical name; and — for the virtualized Directory grid — programmatic **`aria-rowcount` / `aria-rowindex` / `aria-setsize`** so assistive technology reports true position within the full result set despite only the near-viewport rows being in the DOM (finding U6; verified in the ENGINEERING-DESIGN §6.6 checklist).

The layout is responsive by **viewport width**, not device class. Below Tailwind's `md` breakpoint (~768px) the Directory grid collapses into stacked, tappable cards; at `md` and above (tablets and desktops) the full grid is shown, with horizontal scrolling when the chosen columns exceed the width. Touch targets are at least ~44×44px with adequate spacing, and all layouts must remain usable at the largest font-size setting. (WCAG's reflow rule exempts data tables, so the horizontally-scrolling grid is conformant; the card collapse is a UX choice for small screens.)

The same audience that motivates the accessibility commitments — a brotherhood with many older members — also shapes a **performance** requirement: a meaningful fraction of brothers are technically capable but on **slow internet connections** (an email list still heavy with `@aol.com`-era domains, and some users on DSL). Book therefore treats fast loading, especially the first load, as a first-class requirement rather than an afterthought, and is deliberately frugal about the bytes it sends over the wire. The product commitments are that the application code is cached in the browser so a return visit re-downloads little or nothing; that the initial download is kept small and the heavier, rarely-first-used parts of the app load only when needed; and that the member dataset is sent compressed and is not re-sent when it has not changed. These are the same accommodations of an older, capable-but-not-cutting-edge audience as the font-size control, the large touch targets, and the click-to-open help tips (which replace hover tooltips for exactly this reason, §5.9). The mechanisms and the engineering rationale are specified in `ENGINEERING-DESIGN.md` §1.6 (decisions D73–D76). And because the backend scales to zero, the first load after a quiet stretch can take several seconds; to keep that wait legible rather than alarming, Book shows a **"Loading…" overlay** only if the load exceeds a short threshold, adding a "waking the server…" reassurance line if it runs longer, and removes it the moment the directory can render (decision D119; §5.6, ENGINEERING-DESIGN §1.6).

### 5.6 Directory page

The Directory is Book's home page and primary workspace: a table (the "grid") of brother rows that any user searches, filters, and sorts entirely client-side over the in-memory dataset (D4). Below the `md` breakpoint the grid collapses to stacked cards (§5.5); the description below is the grid form. This section covers the columns, sorting, the Name Search box, and the handling of deceased brothers; the structured filter grammar, star interactions, the row-open behavior, the manager/administrator action buttons, and the virtualization and thumbnail-prefetch tuning are specified in the remaining Session-4b work (placeholder at the end of this section).

#### 5.6.1 Columns

Four columns are always present and fixed at the left edge, in order: **Select**, **Star**, **Thumbnail**, and **Canonical Name**. The remaining columns are user-selectable and drag-reorderable (by dragging their headers), and default — the *same set for every role* — to **Class Year**, **Major**, **Email**, **Telephone**, **City**, **State/Province**, and **Country**. A user's column selection and order is the personal "column lens," persisted in `localStorage` independently of the shareable view (D30/D31).

- **Select** is a row-selection checkbox for marking rows for **Export** (§5.6.8); it is not a data field. (The bulk-delete action it once also fed was removed by D100, and the thumbnail-regenerate action by D114.) In MVP it is shown only to managers and administrators, for whom it has a function. Its visibility is governed by a *single capability predicate* (today, "manager or administrator") rather than role checks scattered through the grid, so the deferred "Share this selection" feature (§3.2) — which would let any brother designate a set of rows — can later enable the column for all users with a one-line change.
- **Star** shows a filled gold star when the brother is on the viewing user's personal star list and an outline star otherwise; clicking toggles it (§5.6.x).
- **Thumbnail** shows the brother's 96×96 thumbnail (or a generic avatar), lazily loaded (D9).
- **Canonical Name** shows the constructed `First Last 'YY` form (DATABASE-SCHEMA §5.1).
- **Major** shows the derived display name of the brother's *primary* major — defined as the first entry in their `majors` list, an order the brother controls on their own profile. This makes the profile edit UI's major-ordering (§5.7) load-bearing for what appears here. (Filtering still matches any of a brother's majors.)

Class Year appears as its own column even though the year is already part of Canonical Name: the separate column is what makes the year numerically sortable and range-filterable (e.g. `1980-1989`), which a year embedded in a display string is not.

When the chosen columns exceed the viewport width the grid scrolls horizontally, and the four fixed left columns **freeze as a contiguous block** and stay visible during that scroll, so a brother's identity and the star/select controls remain anchored as the user reads across. Rows are zebra-striped and highlight on hover and keyboard focus.

The restricted-class fields a manager or administrator is permitted to see (the share-toggles, `allowNewsletterEmail`, `allowCommentReplyEmail`, `allowShareWithMITAA`, `lastVerifiedDate`/`verifiedBy`, `lastModified`) are available to those roles as *additional selectable columns, off by default* — keeping the first-load view clean and identical for all roles while letting a manager add, say, a "Last Verified" column when chasing stale records. (`allowCommentReplyEmail` is included for consistency with the other consent flags — finding C3.)

#### 5.6.2 Sorting

Clicking a column header sorts by that column ascending; clicking again toggles to descending. Sorting is single-column. The initial sort on first load is **Canonical Name, ascending**. The Canonical Name column sorts by **last name, then first name, then class year** — the expected directory order — not by the `First Last` display string. Every sort uses that same canonical order (last, first, year) as its **secondary** key, so ties within any primary column (Class Year, City, Country, …) resolve into readable name order. The active sort is part of the shareable view and lives in the URL query string (D31).

#### 5.6.3 Name Search

A single search box — labeled **"Name Search"** to set the expectation that it matches people by name, not employer or free text — performs fuzzy, typo-tolerant, and phonetic matching over brothers' names. It searches the name fields only — `firstName`, `middleName`, `lastName`, `fullLegalName`, `mugName`, and the constructed canonical name — leaving other fields to the structured filters. (Widening the searched set later is a localized change, since search runs client-side.) The matcher rests on four ideas:

- **Tokenization.** Each brother's name fields are split into individual word tokens at load, on both spaces and hyphens, so a hyphenated `Smith-Jones` is found by "Smith" or "Jones," a brother who goes by his middle name or by a multi-word mug name ("Hilbert Space Pilot") is found by any of those words, and a multi-word query requires **all** of its words to match (AND across query tokens, each matched fuzzily/phonetically).
- **Typo tolerance** via **Fuse.js**, weighted toward last and first name, with mug name weighted comparably (for some brothers the mug name *is* their everyday name) — catching misspellings like "Smith" → "Smyth."
- **Phonetic similarity** via the **talisman** library, catching sound-alikes like "Maier" → "Meyer" that diverge too far in spelling for typo tolerance — important for a brotherhood drawn from all over the world. MVP starts on **Double Metaphone**, with **Beider-Morse** (a surname matcher built for mixed language origins) held in reserve and A/B-tested against the realistic test dataset (Session 6); talisman bundles both, so switching is a configuration change.
- **Common nicknames** via a curated **diminutives dictionary** (decision D123). The given-name tokens (first / middle / full-legal) are expanded into their nickname group at index time, and the query is expanded the same way, so **"Bill" finds "William" and "Tom" finds "Thomas," in both directions** — a *semantic* name mapping that neither typo tolerance (edit distance) nor phonetic matching (sound-alike) catches. It is a small bundled static asset built into the same Web Worker index (below), needs **no per-brother data entry**, and is **A/B-tunable** like the phonetic choice — it ships on by default, and over-matches like "Jack" → both John and Jackson are tolerable given AND-across-query-tokens, the short result list, and match-highlighting. Western names only, but it does not impair non-Western names (phonetic still covers those).

Phonetic codes are **computed at load, not stored** in the profile: they are derived data (like the canonical name, D15), storing them would freeze the algorithm we intend to tune and would bloat the bulk-download payload every tab fetches. The computation — and the Fuse.js index build — runs **in a Web Worker** (D110), so SPA init never janks on older hardware: the grid renders immediately with exact/substring matching, and fuzzy/phonetic search switches on when the worker signals "index ready" (progressive enhancement). D35's IndexedDB-memo escape hatch is **dropped** — persisting name-keyed phonetic codes would be on-disk PII residue, contrary to the `no-store` posture (D95) — so the codes are recomputed in the worker each load (cheap off the main thread) rather than memoized.

Search is **live/incremental** (results update as the user types, at in-memory speed), **AND-composed** with the active filters and the deceased default, and its query text lives in the URL (D31) so a search is shareable and restored by the back button. The user's chosen sort is **always honored** — search never silently reorders by relevance — and the matched characters are highlighted in results so the eye finds the hit even within a name-sorted list.

#### 5.6.4 Filtering

A collapsible **filter panel above the grid** holds the structured filters; column headers stay reserved for sorting (§5.6.2), keeping the two interactions cleanly separated. The governing principle is that **a field is filterable by a role exactly when that role can see it as a column** (§5.6.1; DATABASE-SCHEMA §9) — filtering inherits the server-side visibility projection, adding no new exposure, and runs client-side over the already-projected dataset. Filter controls are typed to the field:

- **Numeric grammar** — a text input parsing comma-separated values and dash-separated ranges, freely combined (e.g. `1980, 1985-1989, 1992`), for **Class Year** and **Constitution ID**. Years are always 4-digit to avoid century ambiguity; whitespace is tolerated; an unparseable token is flagged inline rather than silently dropped. Closed ranges only in MVP.
- **Multi-select from a vocabulary** — for **Major**, **Country**, and **State/Province** (the last a US/Canada controlled vocabulary, free text elsewhere — D37), with options populated from the values actually present in the dataset, so US/Canada surface their clean codes and the international tail surfaces its free text.
- **Substring** — for **City**. (Employer and Job Title are intentionally excluded from the MVP filter set, since only a single *current* employer is stored.)
- **Boolean / presence** *(manager/administrator only)* — contact completeness (has / missing **email**, **phone**), the consent flags (**allowNewsletterEmail**, **allowCommentReplyEmail**, **allowShareWithMITAA** — the comment-reply flag included for consistency with the column set, finding C3), and **verification status** (verified / never-verified, or last-verified before a given date) — the data-hygiene filters that let a manager find missing or stale records.

Filters compose **OR within a field** (a comma list or multi-select means "any of these") and **AND across fields** (Class Year *and* State), with the deceased default (§5.6.5) AND-ed in. A **Reset** control clears Name Search, all filters, and the sort, but deliberately leaves the column lens intact. All active filters are part of the shareable view and live in the URL query string (D31).

#### 5.6.5 Deceased brothers

The grid filters to **living brothers by default** (`deceased.isDeceased = false`). An **"Include deceased"** checkbox beside the Name Search box overrides this; its state is part of the shareable view (URL query, D31), since one sometimes looks specifically for a deceased brother. When deceased brothers appear, their thumbnail (and profile headshot) carries a **dark diagonal bar across one corner** — a convention borrowed from genealogy tools — paired, per the accessibility policy (D32), with a textual **"In Memoriam"** marker on the row/card and in the image's accessible name (e.g. alt text "James Smyth '84 — In Memoriam"), so the status is never conveyed by the visual alone.

**De-brothered brothers (decision D115).** A de-brothered record is **absent entirely** from a brother's grid, search, filters, and export — to an ordinary brother it does not exist. For **managers and administrators** it remains visible (so staff can still find and manage it), but its **Canonical Name renders with a strikethrough**, paired with a textual marker in the row/card and the accessible name so the status is not carried by the visual alone (per the accessibility policy, D32). The de-brother control itself is on the Profile page (§5.7.10).

**Unlisted brothers (decision D124).** A brother may make his own record **Unlisted** — a self-service privacy choice that removes it from the directory **for peers**. To an ordinary brother an unlisted record is **absent entirely** (grid, search, filters, export), exactly like a de-brothered one, and a direct link returns "not found." For **managers and administrators** it remains visible, badged **"UNLISTED"** (a marker distinct from the de-brothered strikethrough — present-but-private, not expelled) in the row/card and the accessible name. Unlike de-brothering, the brother stays a full member in every other respect; the control is on his Profile page (§5.7.3), and an administrator may also set it on his behalf.

#### 5.6.6 Stars

A brother's stars are a private per-user list of brother IDs (`users.stars`, DATABASE-SCHEMA §6.1) — a personal "favorites" set for quickly returning to people one looks up often. The **Star** column is one of the always-present pinned columns and is shown to *every* role (unlike Select). A starred brother shows a filled gold star and an unstarred one a hollow outline — a difference of shape and fill, not color alone, per the accessibility policy (D32) — each with an accessible label ("Starred" / "Not starred").

Clicking the star cell toggles membership **optimistically**: the icon flips immediately while a `PUT` or `DELETE /api/me/stars/{id}` is sent (API-SPEC; D24), reverting with a brief error notice if the write fails. Because clicking a row opens that brother's profile (§5.6.7), the star cell stops propagation so starring never also navigates.

A **"Starred only"** toggle sits beside the Name Search box; when on, the grid shows only the user's starred brothers. It is held in **History-API state, not the URL** (D31) — deliberately, since stars are per-user, so a shared link must not impose the sender's stars on the recipient; the back button still restores it. Unlike the field filters, "Starred only" **shows all of the user's starred brothers regardless of living/deceased status** (the deceased ones still carrying their In Memoriam marker), because a hand-picked list should not silently drop someone who has since died. When the user has starred no one, the result state shows a friendly prompt ("You haven't starred anyone yet — click a star to add them").

#### 5.6.7 Opening a profile

Each row is a real anchor (`<a href="/brother/:id">`, React Router `<Link>` — D31): a plain click navigates to the full **Profile** page (§5.7), and a shift/ctrl/cmd-click opens it in a new browser tab that reconstructs itself from the URL. Returning via the back button restores the Directory's scroll position and active view (D31), which is what lets a brother page through several profiles without losing their place — and is the main reason a **modal quick-view is *not* part of MVP**: the "don't lose my place" need it would serve is already met by back-button restoration, while a modal would add a second, edit-incapable view and undercut the anchor/URL model (it is logged as a possible post-MVP addition, needing no data or architecture change). Clicking anywhere on a row opens the profile **except** the interactive Star and Select cells, which perform their own action; the row is keyboard-focusable and opens on Enter, with Star and Select separately tabbable. A per-row **"Search News"** link is not in MVP, as it would depend on Ghost content search, which is deferred (D11).

#### 5.6.8 Manager and administrator actions

A slim **action bar** above the grid is visible only to managers and administrators — gated by the same capability predicate that reveals the Select column (D33), so ordinary brothers see neither. It carries **Export** (manager and admin) and, for administrators only, **Add Brother**. Both are always enabled (Export falls back to the whole current view; Add Brother navigates to `/brother/new`).

> **The Directory's bulk Delete and Regenerate-Thumbnails actions were removed.** Book has no online bulk-write path: the Directory **bulk-delete is dropped** (D100) — single-record delete on the Profile page (§5.7.10) covers the admin error-correction case — and the **thumbnail-regenerate action is dropped entirely** (D114; headshot and thumbnail never drift, so there is nothing to regenerate at runtime). The **Select column and row-selection mechanics are retained** because *Export* still operates over a selection (and the deferred "Share this selection," §3.2, will reuse them), but no destructive bulk action remains.

Row selection uses the per-row **Select** checkbox plus a header **select-all** that selects every row in the *current* search/filter view — not the whole database, and over the entire match set rather than only the virtualized window. Selection is transient in-memory state and **clears whenever the search or filters change**, so an Export stays scoped to what the user can currently see.

**Export** needs no backend endpoint: the SPA already holds the dataset in memory, projected to what the role may see (D4/D5), so generating the CSV is a pure client-side operation that automatically respects the projection — a manager's export omits values hidden behind off-toggles, an administrator's includes them, with no special server path. It emits the selected rows, or, if none are selected, every row in the current search/filter/sort view, in the canonical CSV format (DATABASE-SCHEMA §10); images are never included (they move only via the backup function).

The administrator action on this bar is **Add Brother**, which navigates to `/brother/new` (D31). (The earlier bulk **Delete** and **Regenerate Thumbnails** actions were removed — see the note above; single-record delete lives on the Profile page, §5.7.10, and the regenerate feature is dropped, D114.) On phone-width layouts the bar's actions remain reachable (collapsed into an overflow menu).

#### 5.6.9 Virtualization and thumbnail loading

The grid is a **single virtualized continuous list of the entire current result set** (TanStack Virtual, D29), not a paginated one: because the whole dataset is already in memory (D4), every matching brother is reachable by scrolling and there is nothing to "load more" of, so pagination or a Load-More control would be needless friction. Only the rows near the viewport are rendered, but the full match set scrolls as one list. A **result-count readout** ("248 brothers") reports the set's size in place of a page count. The **column-header row stays sticky** on vertical scroll (keeping the sort controls reachable), complementing the frozen left columns on horizontal scroll (§5.6.1); scroll-offset restoration on Back is implemented over the virtualizer (D31). The card layout below `md` (§5.5) follows the same model — a virtualized vertical list of cards.

Thumbnails follow the lazy-plus-prefetch architecture of D9 (precomputed, kept out of the bulk JSON payload, and served by the backend from a private bucket — decision D126, which replaced the Cloud CDN of D9/D23), with this session fixing the aggressiveness D9 left open:

- **Lookahead loading** tied to the virtualizer's overscan: a row's thumbnail loads as it comes within roughly one-to-two viewport-heights of the visible window, so it is ready just before scrolling into view. The exact margin is feel-tuned during the build.
- **Idle-time prefetch** of the remaining thumbnails in the current result set, in scroll order (`requestIdleCallback`, low priority), **cancelled on any view change** and bounded to the current filtered set. This is affordable — a 96×96 WEBP is a few KB, so even an entire ~2,000-row set is single-digit megabytes, served easily by the backend over HTTP/2 — and because the objects are immutable and indefinitely **browser-cacheable** (D17/D23/D126), re-views and new tabs are instant.
- **No layout shift:** the 96×96 box is reserved whether or not the image has loaded; a brother without a headshot shows the generic avatar immediately, one with a headshot a light placeholder until the thumbnail arrives.

This keeps the initial bulk load text-only and fast — the reason D9 kept thumbnails out of the payload — then streams the images in so scrolling stays smooth.

### 5.7 Profile page

The Profile page is where a brother's complete record is read and edited. One layout serves four audiences — the brother themselves (the *owner*), a peer brother, a manager, and an administrator — and what each may see and change is governed by the field-visibility projection (§4.2; DATABASE-SCHEMA §9), so the same page is materially different in each pair of hands. The page has a **view mode** and an **edit mode**; edit mode is its own URL (`/brother/:id/edit`), guarded by an unsaved-changes prompt (§5.4, D31). An **Edit** button — shown to the owner, managers, and administrators — enters edit mode, where **Save** and **Cancel** appear; Cancel discards every pending change (including a staged photo, §5.7.5) and returns to view mode.

#### 5.7.1 Layout

The profile is a **responsive multi-section grid**. At the `md` breakpoint and above it places sections two-up to use the available width; below `md` it collapses to a single column, which is also the phone form (§5.5). A firm accessibility constraint governs the two-up arrangement: the **visual placement never diverges from the source / reading / tab order** (WCAG 1.3.2 Meaningful Sequence, 2.4.3 Focus Order; D32), so only sections already adjacent in the reading order are ever paired side by side — the grid lays them out two-up while the DOM stays in sequence, and the column collapse reproduces that sequence exactly. The concrete composition (spans, spacing, exact pairing) is finalized in the Claude Design pass; the structural default is a full-width identity header over three paired rows:

- **Identity header** (full width) — the headshot (with crop-on-upload in edit mode, §5.7.5), the large constructed **Canonical Name** (DATABASE-SCHEMA §5.1), class year, the Constitution ID in house form (`#5247`), and the underlying name fields (first, middle, last, full legal, mug). The **In Memoriam** treatment overlays this band when the brother is deceased (§5.7.7). The "Names" content folds in here because the names *are* identity.
- **Contact ‖ Emergency contacts**
- **Professional & personal ‖ Relationships**
- **Preferences & consent ‖ Record status / verification**

Each left-right pair is adjacent in the reading sequence, so a screen reader and the keyboard tab order pass through the page in its intended order regardless of the two-column visual layout.

#### 5.7.2 What each role sees and may edit

The page renders from the same per-role projection that governs the bulk download (§4.2; DATABASE-SCHEMA §9), so a field a caller may not see is simply absent, never merely hidden in the DOM. Concretely:

- **Owner** — sees their entire record and, in edit mode, may change every field except the system-managed ones (`lastModified`, `lastVerifiedDate` / `verifiedBy`, the headshot metadata) and the immutable Constitution ID. The owner alone (with admins) controls their own privacy and consent toggles (§5.7.3).
- **Peer brother** — sees public fields plus the owner's contact fields only where the relevant share-toggle is on; everything restricted (the toggles themselves, the consent flags, the verification and modification dates) is **entirely absent** — a peer sees neither the value nor the fact that it exists. Peers have no Edit button.
- **Manager** — sees everything a peer sees, **plus** the restricted block — the share-toggles, `allowNewsletterEmail`, `allowCommentReplyEmail`, `allowShareWithMITAA`, `lastVerifiedDate` / `verifiedBy`, and `lastModified` — rendered **read-only and visibly locked** (a lock affordance; non-interactive). Where a brother's toggle is *off*, the manager sees a **"this field is private" marker in place of the value** — enough to see *that* a field is hidden (so they can coach the brother and chase stale records) without seeing the protected value, which stays visible only to the owner and admins. A manager may edit the visible directory fields and set deceased status, but may not change another brother's consent or privacy settings, nor see through an off-toggle.
- **Administrator** — sees everything, including the contact values behind off-toggles, is the only role that may change another brother's privacy and consent settings, and carries the on-profile administrator controls (§5.7.10).

Two things fall outside this owner/peer/manager/admin value-visibility model. Managers and administrators additionally see and may edit a **staff-internal Admin Note** — free text for staff coordination and the history of manual changes — which the owner and peers never see (DATABASE-SCHEMA §9); it renders in the Record-status area alongside the verification information. And the owner's own consent switches now include a third PBE-News email preference (§5.7.3).

This is the visual counterpart of the server-side projection: the projection decides what data arrives, and this section decides how the arriving data is presented (editable, locked, or marked private) for each role.

#### 5.7.3 Privacy and consent toggles

Eight boolean fields let a brother control sharing and consent: the five peer-visibility toggles (`shareEmail`, `sharePhone`, `shareAddress`, `shareEmergency`, `shareSpousePartner`) and three consent flags (`allowNewsletterEmail`, `allowCommentReplyEmail`, `allowShareWithMITAA`). Each renders as a **two-position switch that states the *currently-true* consequence inline in plain language**. Most default to the open side — for example, when on, *"Brothers can reach you by email"* and *"You'll receive PBE News by email"* — but the three switches that share *beyond a brother's own reachability inside the brotherhood* default **off / opt-in**: the two third-party-data toggles `shareEmergency` and `shareSpousePartner` (a non-member's data the brother shares on their behalf), and the external-sharing `allowShareWithMITAA` (decisions D89, D93). The **opposite-state consequence moves into the switch's `?` toggle-tip** (the §5.9 help control), rather than being stacked inline as a second sentence (decision D113). This is the middle path: with eight switches, rendering *both* sentences inline on a phone is real scrolling-fatigue against the design's "calm resting interface" goal — but collapsing all the way to a terse "Share email" label would resurrect the hidden-consequence pattern the nudge exists to prevent. Showing the active-side consequence inline, with the counterfactual one tap away, roughly halves the at-rest text while **each control is still named in plain language**, so flipping it stays an informed, considered act. Each share-toggle sits **inline with the field group it governs** (the email rows, the phone, the address block, the emergency-contacts section, the spouse/partner field), so its meaning is legible right where the data is; the three consent switches sit in the Preferences & consent section, with light grouping/disclosure so the eight don't read as an undifferentiated wall. Of those, the two PBE-News email preferences — `allowNewsletterEmail` (the newsletter itself) and `allowCommentReplyEmail` (emails about replies to the brother's article comments) — are grouped under an *"Emails from PBE News"* heading, since both govern mail that **Ghost** actually sends on Book's behalf and are pushed to Ghost (§6.1); `allowShareWithMITAA` is separate (§6.3).

The design is a deliberate, transparent *nudge* (after Thaler & Sunstein's *Nudge*): the pro-sharing default does most of the work, and naming the consequence in plain words — the active side inline, the counterfactual in the `?` tip (D113) — makes any opt-out a considered, informed choice rather than an inattentive click. It is defensible precisely because this is a closed, consensual, members-only directory whose purpose is for brothers to reach one another, the open-default data never leaves the brotherhood, and the consequence text keeps the nudge honest — and the switches whose data *does* reach beyond the brotherhood (the emergency-contact and spouse/partner third-party data, and MITAA sharing) deliberately default **off**, so the open-default nudge is never applied to data leaving the brotherhood (D89/D93). It directly answers the failure mode that crippled the MITAA data: a *single ambiguous global opt-out* ("no emails") that silently swept PBE in without the brother realizing its scope — every Book toggle instead names exactly what it controls. The treatment is **presentation only** — the stored fields remain plain booleans (the reachability toggles default open; the third-party-data toggles and the external-sharing MITAA switch default off / opt-in, D89/D93; DATABASE-SCHEMA §3) — and rendering the consequence as text (not color) satisfies the meaning-not-by-color rule (D32). The owner and admins may change these; a manager sees them locked (§5.7.2).

**Unlisted — hiding the whole record from peers (decision D124).** Separate from the per-field switches above, a brother may set his record **Unlisted**: a single control that removes his *entire* profile from the directory for other brothers, while he stays a full member (he still signs in, still receives PBE News, and is still shared with MITAA per his own switch). It is named in plain language with the same inline-consequence treatment — *"You won't appear in the directory for other brothers; managers and administrators can still see your record"* — with the counterfactual in its `?` tip (D113). It is **self-service** (the brother sets it on his own profile) and an **administrator** may also set it on his behalf, but a **manager may not** (changing another brother's privacy is admin-only). To managers and admins an unlisted record stays visible with an **"UNLISTED"** badge (§5.6.5). This is deliberately *not* de-brothering (§5.7.10), which is punitive and removes the member entirely.

#### 5.7.4 Special field controls

Four fields need richer controls than a plain input:

- **Majors** — edited as a row of **draggable colored "chips,"** one per course code, styled after Material 3 input chips but **built in Book's shadcn/Tailwind stack** (the M3 design is a visual reference only; pulling in MUI would reintroduce the second component system and weight rejected in D29). Chips are **colored by base course number**, so every Course 6 variant (6-1, 6-3, 6-4) shares a hue while the chip text carries the specific code; the color is a grouping aid only — the code text is the meaning, and each chip's accessible name carries the full course name (D32) — and the course-area palette is part of the design pass, gated on AA contrast. New majors are added from a **searchable combobox over the `majors` vocabulary** (by code or name; retired courses still selectable but de-emphasized; duplicates prevented because the box offers only valid unused codes). The **chip listed first is the one shown in the Directory listing**, and the UI says exactly that — *"The major listed first is what appears in the directory listing"* — and never uses the word "primary." Reordering is by **dragging the chips**, with keyboard **Move / Make-first controls** as the accessible equivalent, since drag alone is not keyboard-operable.
- **State / Province** — a control that **watches `country`**: a dropdown of USPS state/territory or Canadian province codes when country is `US` or `CA` (display name shown, code stored), and a free-text box for any other country or when country is unset (D37). New or blank addresses **default `country` to `US`** (≈96% of the membership is US/CA), with any other country one click away; changing country to one whose vocabulary does not contain the current subdivision **clears it with a brief note** rather than keeping an invalid value.
- **Big Brother** — set via a **typeahead combobox searching the brotherhood by canonical name**; the chosen brother shows as a chip linking to his profile. The picker excludes self, and a save that would create a loop is rejected with a plain message (the cycle check is server-authoritative, DATABASE-SCHEMA §8). The Relationships section also shows, **read-only, the derived Little Brothers** — the brothers who name this one as their Big Brother (DATABASE-SCHEMA §5.2) — each linking to their profile; it is free from the in-memory dataset and never stored.
- **Repeatable groups** — external **links** (up to 5: label + URL) and **emergency contacts** (up to 2: name / phone / email) use **progressive disclosure**: edit mode shows the rows that hold data plus a single **"Add a link" / "Add an emergency contact"** affordance that reveals the next empty row on demand, up to the cap, with per-row remove and the Add control disabling at the cap. A profile with none starts with just the Add button — no wall of blank fields. Each link **URL is validated on save against a strict `http`/`https` scheme allowlist** (rejecting `javascript:`, `data:`, and all others) and rendered as an anchor with **`rel="noopener noreferrer"`** — closing stored-XSS-on-click (decision D107; DATABASE-SCHEMA §8).

#### 5.7.5 Headshot: crop-on-upload

In edit mode the header photo area offers **Change photo** and, when one exists, **Remove photo**. Selecting a file (JPEG or PNG) opens a **crop modal** — react-easy-crop inside a Radix Dialog — with a fixed **1:1 frame**, zoom, and drag-to-pan, which serves the common case of cropping oneself out of a wider or group photo. The upload is validated server-side by **magic-byte inspection** (not the declared type) and bounded to **≈40 MP decoded** — a ceiling set deliberately high so a genuine high-resolution headshot is never bounced back to the brother, while a decompression-bomb image is rejected (decision D107; DATABASE-SCHEMA §7). On **Apply** the client sends the cropped region and the **backend normalizes it to the canonical 512×512 WEBP headshot and derives the 96×96 WEBP thumbnail** (DATABASE-SCHEMA §7). This division keeps human judgment (the crop) on the client and canonical encoding (format, exact dimensions, thumbnail derivation) on the server — and because the 512² headshot is retained, a thumbnail can always be re-derived from it by an offline operator script should the thumbnail spec ever change (there is no online "regenerate" feature, D114).

The cropped image is **staged with the form, not uploaded on crop-confirm**: it is previewed immediately but committed only on **Save**, and **Cancel discards it** — so the photo honors the same abandonable edit-session contract as every other field (D31). Because the headshot is its own sub-resource (`PUT /api/profiles/{id}/headshot`, D24), a Save that changes both text and photo is two calls, sequenced text-then-photo (§5.7.9); "Remove photo" stages the same way (a `DELETE` on Save). The crop modal must be **keyboard-operable** — explicit zoom buttons, arrow-key panning, and the Dialog's focus trap — to meet the AA policy (D32), since crop widgets are a known accessibility weak spot.

#### 5.7.6 Verification

The verification state lives in the restricted Record-status section (visible only to the owner, managers, and admins) and shows **"Last verified YYYY-MM-DD by [name]"** — the linked verifier name making provenance legible (self- vs. staff-attested) — or **"Not verified."** A **Verify** button carries role-specific copy (§4.3): for the owner, *"Confirm my information is correct"*; for a manager or admin, the cautioned *"Verify only after confirming directly with this brother."*

The edit↔verification coupling (D28) surfaces as follows. When the **owner** edits and Saves, the server auto-(re)verifies and a quiet toast confirms *"Saved — verified as of today"*; the owner's Verify button is the no-edit confirmation path. When a **manager or admin** Saves content changes to a **currently-verified** profile, a **courtesy dialog** first appears — *"This profile was verified by [name] on [date]. Saving will mark it unverified until re-confirmed."* (Cancel / Save anyway) — the unverify itself being the server's deterministic consequence, not a client flag; editing an already-unverified profile shows no dialog. To keep the directory fresh (the chief motivation for verification), when the **owner views their own profile and it was last verified more than two years ago, or never**, a gentle banner invites them to confirm it. On a **deceased** profile the Verify button is **hidden and verification is frozen** (D28): the obituary data is staff-attested when set, but nothing else can be owner-attested any longer, so the page-level verified status rightly stops moving.

#### 5.7.7 In Memoriam

When `deceased.isDeceased` is true, the profile opens with a **large, respectful "In Memoriam" banner** across the top — a dignified memorial, not a status flag — and the headshot carries the same **diagonal corner bar** as the Directory (D36). Where the years are known, a **lifespan line** appears in the banner in memorial `b.`/`d.` notation (decision D122): **"1940–2024"** when both birth and death year are known (a non-breaking en-dash); **"d. 2024"** when only the death year is known; and **nothing** when only the birth year is known (an open "1940–" would misread as still-living) or when neither is. The public deceased fields display below: the **date of death** (shown in full when known, independent of the lifespan line), a **public-obituary link**, and a **PBE News In Memoriam article link** where present. The two URLs are subject to the same strict `http`/`https` scheme allowlist and `rel="noopener noreferrer"` rendering as profile links (decision D107). These are public-class, so every brother sees the memorial.

Only managers and admins may set deceased status, in edit mode, via a control that — given the gravity — sits behind a **gentle, deliberate confirmation**: *"Mark [name] as deceased? This applies the In Memoriam treatment and removes them from the default directory view."* — distinct in tone from the destructive Delete modal (§5.7.10): significant, but reversible. Turning it on **reveals the now-five optional deceased fields** — date of death, year of death, year of birth, public-obituary link, and In Memoriam article link — inline in the edit form **and moves focus to them**, so the manager records what's known immediately and Saves with the normal page Save (decision D122); it also triggers the verification freeze (§5.7.6). The confirmation dialog stays a *pure confirmation* — a separate data-entry modal was rejected (it would duplicate the inline fields, which must exist for later edits anyway, since an obituary or birth year often surfaces weeks after a death), as was an always-visible disabled block (greyed-out death fields on every living brother's edit page).

#### 5.7.8 Field validation

Input is validated **inline and per field** — errors render adjacent to the field, programmatically associated (`aria-invalid` / `aria-describedby`) and announced (D32) — firing **on blur** for immediate feedback and re-validated in full **on Save**, which blocks submission and moves focus to the first error. The rules are exactly those in DATABASE-SCHEMA §8, surfaced where they apply: required first and last name; class year within range or "unknown"; the **alternate-email field disabled until a primary email is present**; email and phone formats; the controlled dropdowns (country, US/CA state, majors) preventing invalid entry by construction; up to five links with valid `http(s)` URLs; up to two emergency contacts; and dates entered as `YYYY-MM-DD` via a picker that emits ISO. The **Constitution ID is read-only** on this page — it is set only on the Add-Brother flow (`/brother/new`), where it is a required, uniqueness-checked entry (409 on duplicate, D24). Because Book is TypeScript end-to-end (D3), the client and server **share one validation module**, so the fast client check and the authoritative server check cannot drift — the same single-source principle as the shared canonical-name function (DATABASE-SCHEMA §5.1).

#### 5.7.9 Saving and conflict handling

Saving is one user action over a possibly two-call sequence, ordered so the meaningful concurrency guard runs first. For the hardest case — a manager editing a currently-verified profile that also has a newly staged photo — the order is:

1. the courtesy **unverify dialog** (§5.7.6), if applicable;
2. the text **`PATCH`** carrying `If-Match` (Firestore `updateTime` as the token, D25);
3. if the server returns **412**, the **conflict-reconcile flow** (below) — the photo is never touched, so the staged crop simply remains staged;
4. otherwise the staged photo **`PUT` / `DELETE`** (§5.7.5);
5. completion (a toast; return to view mode).

A **412** means the profile changed since it was loaded. The client then **preserves all of the user's unsaved edits** (never discards), re-fetches the current version, and shows an announced notice — **"This profile was changed while you were editing"** (deliberately *not* "by someone else": with independent per-tab SPA instances, the other editor may be the same user in another tab, D31) — **highlighting the fields that changed underneath**, shown read-only beside the pending edits for **manual reconciliation** (no auto-merge, D25). The user adjusts and re-saves with the refreshed ETag. This is rare at Book's scale, but it guarantees no silent clobber.

#### 5.7.10 Administrator controls on the profile

Two administrator-only controls act on the brother whose profile is shown — resolving where these live: on the individual's profile, not a central screen.

- **Delete Brother** — behind a scary confirmation in the same family as the Directory's bulk delete (D41): *"Delete [name]? This cannot be undone."* (Cancel / Proceed to Delete). It calls `DELETE /api/profiles/{id}` (D24) and then returns to the Directory, since the record no longer exists. The bulk equivalent for several brothers at once lives on the Directory (§5.6.8); both are admin-only, and both are hard deletes recoverable only from backups.
- **Change role** — changes the brother's role via `PUT /api/users/{id}/role` (renamed from "Toggle Privileges," finding C13). Rather than a button that cycles through the roles, it is an **explicit role selector** (Brother / Manager / Administrator) behind a confirmation — *"Change [name]'s role to Manager?"* — clearer and less error-prone for a consequential change. The safeguard that **prevents demoting the last remaining administrator** (and warns on self-demotion) is **enforced server-side** — a direct API call cannot remove the final admin either — with every role change audited before/after (decision D106; ENGINEERING-DESIGN §1.4).
- **De-brother / Reinstate** — the rare, admin-only action of de-brothering a member, or reinstating one (decision D115). Given its gravity it sits behind a **deliberate confirmation**, distinct in tone from both the soft mark-deceased dialog and the destructive Delete modal — significant and consequential, but reversible: *"De-brother [name]? This hides them from all brothers, removes their PBE website account and access, and stops all PBE News email. It can be reversed."* (Cancel / De-brother). It calls `PUT /api/profiles/{id}/debrothered` (API-SPEC §3): raising it snapshots the brother's consent/verification state and deletes the Ghost member; reinstating restores the snapshot and re-creates the member. When a brother is **already de-brothered**, the profile (visible only to managers/admins, since it is projected away for brothers) shows a clear **"De-brothered" status with the struck-through Canonical Name** in the identity header (§5.7.1), and the control reads **Reinstate**.

### 5.8 Admin page

The Admin page (`/admin`, an admin-only route enforced server-side, D31) is Book's control panel for the **whole-database operations that remain online** — the counterpart to the per-brother controls on the Profile page (§5.7.10) and the Export action on the Directory (§5.6.8). After the resolution-pass decision that **Book has no online bulk-write path** (D100), the page's online surface narrows to **Download backup** (§5.8.1), the **Ghost reconciliation audit** (§5.8.3), and the **system-message banner** control (decision D117); online **restore** and **bulk-CSV import** are descoped to an offline operator procedure and the post-MVP backlog respectively, and the **thumbnail-regenerate feature is dropped entirely** (D114) — see below. Each remaining action carries a strong, proportionate confirmation and clear result reporting. The page may still grow as new admin needs arise (majors-vocabulary curation is a noted future tenant).

**System message banner (decision D117).** The Admin page carries the one *new* online admin operation added since the resolution pass: a control to **compose, set, and clear the optional site-wide system banner** (§5.2). The admin enters the message text and an info/warning severity and toggles it on; it then shows across the top of every page for all roles and **persists until the admin clears it** (it is not per-user dismissible). Its appearance mirrors Ghost's announcement bar, but Book and Ghost banners are independent — neither propagates to the other. This is distinct from the maintenance page (D118), which is operator/script-triggered, not an Admin-page control.

#### 5.8.1 Backup (and offline restore)

A **Download backup** button produces a complete backup of the database, including images (the format is text plus a zipped folder of image files). A daily automated backup also runs server-side (ENGINEERING-DESIGN §6.3); the admin is the **custodian** of any downloaded archive (D101; USER-MANUAL). **Restore is *not* an online button (D100/D101).** It is the single most destructive operation in Book, and is performed as an **offline maintenance event**: Book goes hard-down, the database is replaced from a **structurally-validated** backup, and the single instance cold-hydrates on restart (ENGINEERING-DESIGN §6.3). This deletes an entire class of online-coherence problems and is the honest shape of a take-everything-down-and-replace operation. GCS object versioning (D8) is a second safety net beneath the backups.

#### 5.8.2 Bulk-CSV import — deferred (post-MVP)

Online bulk-CSV import is **not in MVP** (D100): Book has no online bulk-write path. The realistic driver — applying externally-vetted MITAA-sourced corrections — is, by measured cadence, a handful of **single** manual edits per year (§3.2), so a MITAA-specific bulk import is **backlogged** and any rare bulk update is an **offline** operator task. Whenever such an import runs, it retains the safeguards designed here — a **two-step dry-run preview** ("this will update X, add Y, and **unverify Z**," no change until Apply) and the **per-row non-owner-edit** verification coupling (a changed row unverifies; unchanged and deceased rows keep status; the system-managed `lastVerifiedDate`/`verifiedBy` are ignored on import) — per D28/D68; the mechanics live in ENGINEERING-DESIGN §6.5, and the canonical CSV format (with formula-injection neutralization) in DATABASE-SCHEMA §10.

#### 5.8.3 Ghost sync

A **Sync with Ghost** button triggers the reconciliation audit — a comparison between Book's database and the Ghost membership — with the results presented in a **discrepancy report** on the page (the JSON shape is in API-SPEC §7). It is **read-only into Book for every field except** the scoped most-recent-change-wins newsletter-subscription reconcile, which it applies and logs (D103); the same audit also runs on a schedule (D99). Discrepancy categories and the resolution policy are in ENGINEERING-DESIGN §5.1.

#### 5.8.4 Thumbnail regeneration — no feature (dropped)

There is **no thumbnail-regenerate feature** in Book — not online, and not even as a named offline tool (decision D114). The original motivation (thumbnails drifting out of sync with their headshots) is foreclosed by the **pointer-last write ordering** (D98): a version's headshot and thumbnail are written as a matched pair, so they never diverge at runtime. The only situations that would call for re-derivation — a change to the thumbnail spec, or a generation-bug fix — are rare developer events, handled by a one-off script over the retained 512² headshots written at the time, under the general offline-migration discipline (ENGINEERING-DESIGN §6.5). A brother who wants a different thumbnail simply re-uploads/re-crops.

#### 5.8.5 Not on this page

The per-brother **Change role** and **Delete Brother** actions are on the individual's Profile page (§5.7.10), and **Add Brother** and **Export** are on the Directory (§5.6.8) — each placed where its subject is chosen. The Admin page is reserved for the whole-database operations that remain online (backup download and the Ghost audit); restore and bulk-CSV import are offline/operator or backlog items (D100), and thumbnail regeneration is dropped entirely (D114) — none are buttons here.

### 5.9 In-context help and guidance

*(A cross-cutting standard added as a Session-4c addendum; it applies across the Directory, Profile, and Admin pages.)*

Book embeds usage guidance directly in its pages rather than depending on a separate manual, on the principle that a well-designed interface should not require one — almost no one reads manuals, and needing one is itself a design smell. The `USER-MANUAL` deliverable accordingly becomes a *reference* whose content is the **single source** for the in-page help, so the two never drift. This matters doubly for Book's heaviest users — alumni aged 60 and up, some with vision or fine-motor limitations.

Guidance is provided **where a control's purpose or usage is not self-evident — not on every element.** A help icon beside literally every field and button would clutter the page, and visual busyness is itself an accessibility burden for older users; obvious controls ("Save," "Cancel") get nothing, while genuinely unguessable affordances get help. The canonical example is the **class-year field, which silently accepts lists and ranges** (`1980-1989, 1992`) — something most users would never think to try.

Help is layered:

- **A persistent visible label** on every field, programmatically associated — never replaced by a placeholder.
- **Persistent helper text** beneath a control for anything the user may need *during* input, always visible and `aria-describedby`-linked so a screen reader announces it on focus (this is where the class-year range/list hint lives).
- **A supplementary placeholder** as a light example that auto-clears the instant the user types (native HTML `placeholder` — nothing to delete). It never carries essential instructions and is never the label; its color is verified to meet AA contrast in both themes; and its text is accurate to the field's real function — e.g., the Directory's name-only search (§5.6.3, D35) reads *"Search by name — e.g., Smyth, or Lissajous,"* not a misleading "names, years, majors."
- **A toggle-tip help control** for deeper "what is this and how do I use it" explanation (below).

The toggle-tip control is a **question-mark-in-a-circle icon** rendered as an **SVG** (Lucide `CircleHelp`) — not a Unicode glyph or overlaid characters, since only an SVG's color and size are fully controlled by the theme CSS: it inherits **`currentColor`** (so it follows the light/dark token automatically) and is sized in **`rem`** (so it grows with the font-size preference, §5.3). The icon sits inside a **real `<button>`** with a descriptive accessible name ("Help: Class year") and a ≥44px touch target (§5.5), the SVG itself `aria-hidden`. It is built on **Radix's Popover** — a click/toggle disclosure, explicitly *not* a hover tooltip: it opens on click and closes on a second click, on Escape, or on an outside click, returning focus to the trigger. Hover tooltips are rejected because they fail exactly this audience — they demand a precise, sustained hover, vanish on pointer drift or timeout, and do not exist on touch devices (tablets and phones, common in this cohort) — whereas a click-toggle disclosure is operable by fine-motor, touch, and keyboard users alike. The revealed text is **announced to assistive technology on open** (a live region or focus placement), so the help is never silent to screen-reader users, and the "?" is placed **consistently** — immediately after the control's label — so users learn where to look. A global **"Show help" toggle** that reveals every help affordance at once is noted as a possible future enhancement: a calm resting interface, with comprehensive guidance on demand.

## 6. Integration Requirements

Book is **one half of a composite system with Ghost** (§1): Ghost authenticates Book's users, and Book manages Ghost's members. This section states the integration requirements; the technical design is in ENGINEERING-DESIGN §5.

### 6.1 Ghost membership sync

- **Book is the single source of truth** for member data. Member-data editing in **Ghost is disabled and its account UI redirected to Book**, so brothers maintain their information in exactly one place.
- Book **pushes to Ghost synchronously on save** (diff-based — only changed fields, D96), the four things Ghost needs: the brother's **primary email**, their **name** (the Canonical Name, for bylines), and their two PBE-News email preferences — **`allowNewsletterEmail`** and **`allowCommentReplyEmail`**. No other field is pushed. An **email change commits in Book only if Ghost accepts it**, so there is no routine divergence; the diff-based push also avoids re-subscribing a brother who used Ghost's own unsubscribe link.
- The member **lifecycle** maps Book actions to Ghost: creating a brother creates the Ghost member **first** (capturing `ghostMemberId`, suppressing Ghost's signup email); editing email/name/preferences updates the member; deleting a brother deletes the Ghost member **first** (abort-clean on failure); and **marking a brother deceased forces both email preferences off** so no further PBE News email is sent (the prior state snapshotted for reversal, D80).
- Sign-in matches on the **primary email only** (normalized, enforced unique, D97); `alternateEmail` is a fallback contact, never a login identity and never pushed to Ghost.
- The admin **"Sync with Ghost"** action (§5.8.3), which **also runs on a schedule** (D99), is a **reconciliation audit**: it reports discrepancies (unmatched Ghost members, field drift, missing members, Book-internal orphans) for the admin to act on, and is **read-only into Book for every field except** the scoped most-recent-change-wins newsletter-subscription reconcile, which it applies automatically (D103) — so a brother's Ghost-side unsubscribe is honored without ever auto-undoing a Book-side opt-out.
- The **initial pre-launch data load is performed by standalone migration tools, not by Book** (ENGINEERING-DESIGN §5.1). That one-time process seeds the newsletter and comment-reply preferences *from* Ghost so existing opt-outs are honored.

### 6.2 PBE News Linter

Book provides the Linter a **read-only name-and-year roster** (`GET /api/roster`: IDs, name fields, class year, deceased flag, and the constructed canonical name — no contact or restricted data). The Linter pulls the whole roster once per run and matches locally; it never writes to Book. The endpoint is authenticated by a **Google service-account identity token**, separate from the Ghost session — so a non-browser, first-party tool can read the roster without a public API and without a leakable API key. Book serves the finished canonical name, so there is a single source of truth even though the Linter is a separate program in a possibly different language.

### 6.3 MITAA exchange

The MITAA exchange is **manual, occasional, and treats MITAA as a low-trust suggestion source**; automated exchange is deferred (§3.2). Book remains the system of record.

- **Import** uses external matching and curation followed by the ordinary admin **dry-run bulk import** (§5.8.2); its chief value is **learning of brothers' deaths** that PBE would otherwise miss.
- **Export** is gated by **`allowShareWithMITAA`**, which means *"may my contact information be shared with MITAA"* — a **master switch** over the brother's own email/phone/postal address that overrides the brother-facing share-toggles (a different audience). Emergency-contact data is never shared with MITAA; **name, class year, and public deceased information are shared regardless** of the flag (identity/join-key, and the reciprocal death correction — PBE sometimes knows of a death before MITAA does). Because honoring this consent can require contact values hidden from other brothers, **MITAA export is administrator-only** and is a full snapshot of consenting brothers (changes-only deferred). The exact format is kept flexible.

### 6.4 Mailman (deferred)

Integration with PBE's two **MIT-hosted Mailman 2.1.34** lists — **pbe-brothers-official** (moderated) and **pbe-connect** (unmoderated broadcast) — is **deferred** (§3.2); Mailman 2 offers no usable API and the instance cannot be modified. These lists are independent of PBE News (which is sent via Ghost). Future *replacement* options — migrating the moderated list into Ghost, or self-hosting Mailman 3 for a direct integration — are recorded as deferred work; the first would introduce multiple Ghost newsletters, itself a deferred enhancement (§3.2).
